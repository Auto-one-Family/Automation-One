## Empfehlung für T10-R4

### Root Causes jetzt klar:
1. **BUG-8 Root Cause bestätigt:** `sensor_handler.py:220` macht `scalar_one_or_none()` Query nach (esp_id, gpio, sensor_type) — findet 2 Rows (verschiedene onewire_addresses). Fix: WHERE-Clause um `onewire_address` erweitern.
2. **BUG-11 Root Cause klar:** `notification_repo.py:621` — falscher JSONB-Zugriff. Fix: `cast(Notification.extra_data["esp_id"], String)` oder `Notification.extra_data.op("->>")('"esp_id"')`.
3. **BUG-2 Root Cause klar:** Subzone-ACK-Handler übergibt offset-naive datetime an asyncpg. Fix: `datetime.now(timezone.utc)`.
4. **BUG-15 Root Cause klar:** simulation_scheduler hat Hard-coded oder residuale Device-IDs. Fix: Scheduler-Config gegen aktuelle esp_devices validieren.

### Brauchen weiteren Code-Zugriff:
- BUG-12/13: Frontend-Komponenten (`SensorInventoryView.vue`, Monitor Widget-Selector) müssen auf korrekte Sensor-Disambiguierung und unit-Mapping geprüft werden.
- BUG-14: Prüfen ob ESP_00000001 Firmware oder Mock-Simulator den Epoch-Timestamp sendet.
- BUG-16: Frontend Notification-Polling-Logik (mehrere Stores rufen gleichzeitig `alerts/stats` ab).

### Prioritätsreihenfolge für Fixes:
1. **BUG-8** (KRITISCH) — Blockiert gesamte ds18b20 Datenpipeline, ~120 Errors/Stunde
2. **BUG-11** (KRITISCH) — Device-Löschung komplett blockiert
3. **BUG-2** (HOCH) — Subzone-Zuweisungen scheitern
4. **BUG-15** (MITTEL) — Ghost-Device Cleanup

6. **BUG-14/16** (NIEDRIG) — Data-Quality und Performance
