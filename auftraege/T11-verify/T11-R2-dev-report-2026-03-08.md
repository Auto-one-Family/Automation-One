Der Notification-Service schreibt "aufgelöst" in den `body` (nicht `title`), und nur in einem Edge-Case: wenn `auto_resolve` keinen passenden Alert findet. In der Praxis tritt dies nicht auf.

### 4B: Alert-Storm-Analyse

| Metrik | Wert |
|--------|------|
| Alerts letzte Stunde | **14** |
| Alerts letzte 24h | **62** |
| Rate | ~1 neuer Alert alle **4,3 Minuten** |
| Hauptverursacher | "Sensordaten veraltet" (6x) + "Heartbeat-Lücke" (6x) |

**Root Cause Alert-Storm:**

Der Alert-Generator erstellt bei jedem Check-Zyklus einen **neuen DB-Row** statt den bestehenden aktiven Alert zu updaten. Obwohl 3 Ebenen Dedup-Prevention implementiert sind (fingerprint + 60s-Fenster + correlation_id), werden die Alerts trotzdem dupliziert. Mögliche Ursachen:

1. **Fingerprint stimmt nicht überein** — leichte Variationen in den Alert-Daten (z.B. Timestamp im Payload) erzeugen unterschiedliche Fingerprints
2. **60s-Fenster zu kurz** — der Check-Zyklus läuft alle ~15 Sekunden, aber das Dedup-Fenster ist 60s. Wenn ein Alert resolved und sofort re-created wird, entsteht eine Schleife
3. **BUG-08 Feedback-Loop** — `MultipleResultsFound` Exceptions (110x in 30 Min) triggern Error-Alerts, die wiederum neue Notifications erzeugen

### 4C: ISA-18.2 Lifecycle-Bewertung

**Bewertung: TEILWEISE FUNKTIONAL** — Resolve funktioniert, Acknowledge ist toter Code.

## Root-Cause-Analyse — Alle 3 Bugs

### BUG-06: Status-Desync (KRITISCH)

**Datei:** `El Servador/god_kaiser_server/src/mqtt/handlers/heartbeat_handler.py`

```
Zeile ~164-174: _handle_heartbeat()
  ├── if device.status == 'pending_approval':
  │     → _update_pending_heartbeat()  ← aktualisiert NUR last_seen via metadata
  │     → return                        ← EXIT HIER — update_status() wird ÜBERSPRUNGEN
  │
  └── else (approved devices):
        → update_status()              ← setzt status='online' UND last_seen
```

**Root Cause:** Der `pending_approval`-Zweig im Heartbeat-Handler ruft `_update_pending_heartbeat()` auf und returnt sofort. Die Methode `update_status()` (Zeile ~214), die sowohl `status` als auch `last_seen` korrekt setzt, wird **nie erreicht**.

**Auswirkung:**
- `last_seen` wird über einen Metadata-Flush aktualisiert → Frontend berechnet "online" (da `last_seen < 60s`)
- `status` bleibt `pending_approval` oder `offline` → Server-API prüft `status`-Feld → 403

**Frontend-Seite:** `El Frontend/src/composables/useESPStatus.ts:101` — berechnet Online-Status als Fallback aus dem Alter von `last_seen` (Priority 3). Dies widerspricht dem Server-Guard in `El Servador/.../api/v1/sensors.py:583`.

**Fix:** Im `pending_approval`-Pfad nach der Heartbeat-Verarbeitung auch `update_status()` aufrufen, oder `status` direkt in `_update_pending_heartbeat()` auf `'online'` setzen (wenn `approved_at IS NOT NULL`).

**Screenshot-Evidenz S33:** Device-Settings zeigt "Verbindung: Online" (Frontend-Berechnung aus last_seen) aber der Server sieht `status='offline'` in der DB.

### BUG-07: Device-Delete → 500 (KRITISCH)

**Datei:** `El Servador/god_kaiser_server/src/db/repositories/notification_repo.py:621`

```python
# Zeile 621 (ungefähr):
Notification.extra_data["esp_id"].astext == device_id
```

**Root Cause:** `.astext` ist eine **PostgreSQL JSONB**-API. Die Spalte `Notification.extra_data` ist aber als `JSON` deklariert (nicht `JSONB`):

```python
# notification.py:139
extra_data = Column(JSON, nullable=True)  # ← JSON, nicht JSONB!
```

Bei `JSON`-Spalten gibt es kein `.astext`-Attribut → `AttributeError` bei jedem DELETE-Versuch.

**Auswirkung:** Devices können nicht über das Frontend gelöscht werden (S48 zeigt 500-Error). Manuelles DB-Cleanup nötig.

**Fix-Optionen:**
1. `Column(JSON)` → `Column(JSONB)` migrieren (Alembic Migration) — **empfohlen**
2. `.astext` → `.as_string()` oder `cast(Notification.extra_data["esp_id"], Text)` — Quick-Fix

### BUG-08: MultipleResultsFound (HOCH)

**Datei:** `El Servador/god_kaiser_server/src/db/repositories/sensor_repo.py:109`

```python
# Zeile 109 (ungefähr):
def get_by_esp_gpio_and_type(self, esp_id, gpio, sensor_type):
    return session.execute(
        select(SensorConfig).where(
            SensorConfig.esp_id == esp_id,
            SensorConfig.gpio == gpio,
            SensorConfig.sensor_type == sensor_type
        )
    ).scalar_one_or_none()  # ← CRASHT bei 2+ Ergebnissen
```

**Root Cause:** Query filtert nur nach `(esp_id, gpio, sensor_type)`. Bei OneWire-Bus (mehrere DS18B20 auf demselben GPIO) gibt es mehrere Rows mit identischem `(esp_id, gpio, 'temperature')` aber unterschiedlichen `onewire_address`-Werten. `scalar_one_or_none()` wirft `MultipleResultsFound`.

**Korrekte Methode existiert bereits:** `get_by_esp_gpio_type_and_onewire()` auf Zeile ~864 — sie filtert zusätzlich nach `onewire_address`. Diese wird aber an den meisten Aufrufstellen **nicht verwendet**.

**Auswirkung:** 110 Exceptions in 30 Minuten (alle ~15s). Log-Noise + potenzielle Dateninkonsistenz + triggert weitere Error-Alerts (Feedback-Loop).

**Fix:** Alle Aufrufer von `get_by_esp_gpio_and_type()` durch `get_by_esp_gpio_type_and_onewire()` ersetzen, oder die alte Methode um optionalen `onewire_address`-Parameter erweitern.

## Zusätzliche Befunde

### BUG-09 (NEU): Subzone-Namen NULL

**Quelle:** DB-Query Q13

5 von 7 `subzone_configs` haben `subzone_name = NULL`. Die Subzone-Erstellung (implizit über Config-Panel) setzt den Namen nicht korrekt.

**Severity:** MITTEL — Subzonen existieren in der DB, aber ohne lesbaren Namen. Monitor-View und L2 können diese nicht sinnvoll anzeigen.

### BUG-10 (NEU): Heartbeat Epoch-Null bei Wokwi

**Quelle:** Screenshot S33

Device-Settings für ESP_00000001 zeigt "Heartbeat: 01.01.1970, 01:00" — das ist Unix Epoch 0. Bedeutet: `last_seen` wurde entweder nie gesetzt oder auf 0 initialisiert.

**Severity:** NIEDRIG — Wokwi war zum Testzeitpunkt nicht gestartet. Aber die Darstellung "01.01.1970" ist irreführend — sollte "Nie" oder "—" zeigen.

### BUG-11 (NEU): Acknowledged-Status toter Code

**Quelle:** DB-Query Q1

ISA-18.2 Acknowledge-Transition ist implementiert aber wird **nie genutzt** (0 Rows mit Status "acknowledged"). Entweder:
- Der Frontend-Button ist nicht erreichbar/sichtbar
- Oder der Acknowledge-Flow wird nie getriggert

**Severity:** NIEDRIG — Funktionalität existiert, wird aber nicht genutzt. Sollte im Alert-UX-Audit geprüft werden.
