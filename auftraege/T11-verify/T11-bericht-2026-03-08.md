### BUG-06 Detail: Status-Desync (KRITISCH)

**Urspruengliche Hypothese:** "Heartbeat aktualisiert `last_seen` aber NICHT `status`"
**Code-Review-Ergebnis:** HYPOTHESE WIDERLEGT — Root Cause ist komplexer.

**Exakte Code-Pfade:**

1. **Heartbeat-Handler SETZT status korrekt** — `heartbeat_handler.py:204`:
   ```python
   await esp_repo.update_status(esp_id_str, "online", last_seen)
   ```
   Dieser Call setzt SOWOHL `status="online"` ALS AUCH `last_seen`.

2. **ABER: `last_seen` wird aus `payload["ts"]` berechnet** — `heartbeat_handler.py:202`:
   ```python
   ts_value = payload["ts"] / 1000 if payload["ts"] > 1e10 else payload["ts"]
   last_seen = datetime.fromtimestamp(ts_value, tz=timezone.utc)
   ```
   Wenn Wokwi `ts=0` sendet → `last_seen = 1970-01-01T00:00:00+00:00`

3. **Maintenance-Service sieht `last_seen` aus 1970** → setzt `status="offline"` zurueck

4. **Flicker-Zyklus:**
   ```
   Heartbeat (ts=0) → status="online", last_seen=1970
       → Maintenance-Check → last_seen 56 Jahre alt → status="offline"
           → Naechster Heartbeat → status="online" (kurzzeitig)
               → Maintenance → offline → ...
   ```

5. **Fuer `pending_approval` Devices: Heartbeat-Handler kehrt FRUEH zurueck** — `heartbeat_handler.py:164-174`:
   ```python
   if status == "pending_approval":
       await self._update_pending_heartbeat(esp_device, payload)
       await session.commit()
       return True  # ← KEHRT ZURUECK OHNE status="online" zu setzen
   ```

**Root Cause (2 Probleme):**
- **Problem A:** `ts=0` von Wokwi fuehrt zu `last_seen=1970` → Maintenance setzt offline (= BUG-05 Folge)
- **Problem B:** `pending_approval`-Devices werden NIE auf "online" gesetzt, Frontend zeigt aber "Online" weil `useESPStatus.ts:97-103` den Status aus `last_seen`-Alter berechnet statt aus dem DB-`status`-Feld

**Betroffene Dateien:**
- `heartbeat_handler.py:164-174` (pending_approval Early Return)
- `heartbeat_handler.py:202` (ts=0 nicht abgefangen)
- `El Frontend/src/composables/useESPStatus.ts:97-103` (Frontend-Fallback)

**Blast Radius:** ALLE Wokwi-ESPs + ALLE pending_approval-Devices. Config-Saves auf ALLEN betroffenen Devices blockiert.

**Fix-Komplexitaet:** Mittel (2 Stellen)
- Server: `if ts_value <= 0: last_seen = datetime.now(timezone.utc)` in `heartbeat_handler.py:202`
- Server: Bei `pending_approval` nach Approval → Status-Transition zu "online" (Zeile 164-174)
- Frontend: `useESPStatus` muss `pending_approval` als eigenen Status behandeln, nicht als "online" Fallback

### BUG-07 Detail: Device-Delete 500 (KRITISCH)

**Code-Review-Ergebnis:** HYPOTHESE BESTAETIGT — `.astext` auf JSON-Spalte (nicht JSONB)

**Exakte Code-Stelle:** `notification_repo.py:621`
```python
Notification.extra_data["esp_id"].astext == esp_id,
```

**Root Cause:** `Notification.extra_data` ist als `JSON` Column-Typ definiert (nicht `JSONB`). SQLAlchemy's `.astext` Property existiert NUR auf `JSONB`-Columns. Bei `JSON` wirft es:
```
AttributeError: Neither 'BinaryExpression' object nor 'Comparator' object has an attribute 'astext'
```

**Aufrufkette:**
```
DELETE /api/v1/esp/devices/{id}
  → esp_service.delete_device()
    → notification_repo.resolve_all_for_device(esp_id)  ← CRASH
      → Notification.extra_data["esp_id"].astext == esp_id  ← Zeile 621
```

**Blast Radius:** JEDER Device-Delete-Versuch schlaegt fehl. Devices koennen NUR per manuellem DB-Cleanup entfernt werden.

**Fix-Komplexitaet:** Einzeiler
```python
# ALT (notification_repo.py:621):
Notification.extra_data["esp_id"].astext == esp_id,

# NEU:
cast(Notification.extra_data["esp_id"], Text) == esp_id,
# ODER: Column-Typ von JSON auf JSONB aendern (Alembic-Migration)
```

### BUG-08 Detail: MultipleResultsFound (HOCH)

**Code-Review-Ergebnis:** HYPOTHESE BESTAETIGT — fehlender `onewire_address`-Filter

**Exakte Code-Stelle:** `sensor_repo.py:103-109`
```python
stmt = select(SensorConfig).where(
    SensorConfig.esp_id == esp_id,
    SensorConfig.gpio == gpio,
    func.lower(SensorConfig.sensor_type) == sensor_type.lower(),
)
result = await self.session.execute(stmt)
return result.scalar_one_or_none()  # ← CRASH bei 2+ DS18B20 auf gleichem GPIO
```

**Root Cause:** Die Methode `get_by_esp_gpio_and_type()` filtert nur nach `esp_id`, `gpio`, `sensor_type` — aber NICHT nach `onewire_address`. Bei 2+ DS18B20 auf GPIO 4 (OneWire-Bus) liefert die Query 2 Rows → `scalar_one_or_none()` wirft `MultipleResultsFound`.

**Korrekte 4-Way-Methode existiert bereits:** `sensor_repo.py:864`
```python
async def get_by_esp_gpio_type_and_onewire(
    self, esp_id, gpio, sensor_type, onewire_address
) -> Optional[SensorConfig]:
```
Diese Methode wird aber vom Aufrufer (sensor_handler) NICHT verwendet.

**Blast Radius:** JEDER ESP mit 2+ DS18B20 auf gleichem GPIO. 110 Exceptions/30min. DS18B20-Daten werden NICHT gespeichert (Datenverlust).

**Fix-Komplexitaet:** Mittel
- Aufrufer in `sensor_handler.py` muss `get_by_esp_gpio_type_and_onewire()` verwenden wenn `onewire_address` im Payload vorhanden
- Fallback auf `get_by_esp_gpio_and_type()` nur wenn kein `onewire_address`

### BUG-01 Detail: Actuator-Geister (HOCH)

**Code-Review-Ergebnis:** Root Cause eingegrenzt auf `_has_pending_config` + `_auto_push_config`

**Exakte Code-Stellen:**
1. `heartbeat_handler.py:1194-1207` — Config-Mismatch-Detection:
   ```python
   needs_actuator_push = esp_actuator_count == 0 and db_actuator_count > 0
   if needs_sensor_push or needs_actuator_push:
       asyncio.create_task(self._auto_push_config(esp_device.device_id))
   ```

2. `heartbeat_handler.py:1207` — `asyncio.create_task()` ohne eigene Session:
   ```python
   asyncio.create_task(self._auto_push_config(esp_device.device_id))
   ```
   Dieser Task teilt die Heartbeat-Session. Nach `session.commit()` im Heartbeat-Handler wird die Session geschlossen, der Task versucht danach noch DB-Operationen → `InvalidRequestError`.

**Root Cause:** Heartbeat meldet `actuator_count > 0` (Firmware hat GPIO 27 hardcoded) → Server sieht Mismatch mit DB → pushst Config → Config-Push scheitert wegen Session-Race. Beim naechsten Heartbeat: gleicher Zyklus.

Der "Command-Handler returned False" kommt daher, dass die Firmware den Actuator zwar im Heartbeat meldet, aber keinen Command-Handler fuer GPIO 27 implementiert hat (Firmware-seitig nicht konfiguriert).

**sensor_count Mismatch (4 vs. 2):** SHT31 sendet 2 Sensor-Typen (`sht31_temp` + `sht31_humidity`). Firmware zaehlt ALLE eingehenden Werte → `sensor_count=4` (raw+processed). Server hat nur 2 `sensor_configs`.

**Blast Radius:** Nur ESP_472204 (einziger ESP mit Actuator).

**Fix-Komplexitaet:** Mittel
- `_auto_push_config` muss eigene `resilient_session()` verwenden statt die Heartbeat-Session
- sensor_count-Vergleich toleranter machen (Faktor 2 fuer raw+processed)

### BUG-02 Detail: Subzone-ACK Datetime-Crash (KRITISCH)

**Code-Review-Ergebnis:** HYPOTHESE BESTAETIGT — `DateTime` ohne `timezone=True`

**Exakte Code-Stellen:**
1. `subzone.py:127-131` — Model-Definition:
   ```python
   last_ack_at: Mapped[Optional[datetime]] = mapped_column(
       DateTime,  # ← FEHLT: timezone=True
       nullable=True,
   )
   ```

2. `subzone_service.py:674` — Update mit aware Datetime:
   ```python
   config.last_ack_at = datetime.now(timezone.utc)  # ← aware (timezone-aware)
   ```

3. `subzone_repo.py:221`:
   ```python
   return await self.update_subzone(esp_id, subzone_id, last_ack_at=datetime.now(timezone.utc))
   ```

**Root Cause:** `DateTime` ohne `timezone=True` → PostgreSQL speichert als `TIMESTAMP WITHOUT TIME ZONE` → DB liefert naive datetime zurueck. Python-Code vergleicht dann `naive_from_db - datetime.now(timezone.utc)` → `TypeError: can't subtract offset-naive and offset-aware datetimes`.

**Blast Radius:** ALLE Subzone-ACKs auf ALLEN ESPs. Jeder ACK-Versuch crasht.

**Fix-Komplexitaet:** Mittel (Alembic-Migration erforderlich)
```python
# Model (subzone.py:128):
DateTime(timezone=True),  # ← timezone=True hinzufuegen

# Alembic-Migration:
op.alter_column('subzone_configs', 'last_ack_at',
    type_=sa.DateTime(timezone=True))
```

**HINWEIS:** Die `.claude/rules/api-rules.md` Regel verbietet explizit `DateTime` ohne `timezone=True`:
> `DateTime` ohne `timezone=True` in Models → DB liefert naive Timestamps. Immer `DateTime(timezone=True)`

### BUG-03 Detail: Unit-Encoding Double-UTF8 (MITTEL)

**Code-Review-Ergebnis:** Root Cause im sensor_handler Payload-Verarbeitung

**Exakte Code-Stelle:** `sensor_handler.py:329` ff. — Unit wird direkt aus Payload uebernommen:
```python
unit = payload.get("unit", "")
```

**Root Cause:** Die ESP32-Firmware (Latin-1/CP1252 Encoding) sendet `°C` als Bytes `0xB0 0x43`. Der MQTT-Payload wird als UTF-8 interpretiert. Wenn die Firmware das `°`-Zeichen als Latin-1 Byte `0xB0` sendet (statt UTF-8 `0xC2 0xB0`), und der Server dies als UTF-8 re-kodiert, entsteht die Doppel-Kodierung:
```
Firmware: 0xB0 0x43 (Latin-1: "°C")
  → MQTT als UTF-8 gelesen: 0xC3 0xB0 0x43 (UTF-8: "ðC" oder Mojibake)
  → Nochmal serialisiert: "\u00c2\u00b0C" = "Â°C"
```

Es gibt KEINEN Encoding-Sanitizer im sensor_handler oder in den Pydantic-Schemas.

**Blast Radius:** Alle Nicht-ASCII-Einheiten (°C, µS/cm, %RH) von ESP32-Firmware.

**Fix-Komplexitaet:** Einzeiler
```python
# In sensor_handler.py nach payload.get("unit", ""):
unit = payload.get("unit", "").encode('latin-1', errors='ignore').decode('utf-8', errors='replace')
# ODER: Unit-Lookup in sensor_type_registry statt aus Payload
```

### BUG-05 Detail: Timestamp Epoch 0 (KRITISCH)

**Code-Review-Ergebnis:** HYPOTHESE BESTAETIGT — kein ts=0 Guard

**Exakte Code-Stellen:**

1. `sensor_handler.py:329-337` — Timestamp-Konvertierung ohne Guard:
   ```python
   esp32_timestamp_raw = payload.get("ts", payload.get("timestamp"))
   esp32_timestamp = datetime.fromtimestamp(
       esp32_timestamp_raw / 1000 if esp32_timestamp_raw > 1e10 else esp32_timestamp_raw,
       tz=timezone.utc,
   ).replace(tzinfo=None)
   ```
   Bei `ts=0` → `datetime.fromtimestamp(0)` → `1970-01-01 00:00:00`

2. `heartbeat_handler.py:202` — Gleiches Problem:
   ```python
   ts_value = payload["ts"] / 1000 if payload["ts"] > 1e10 else payload["ts"]
   last_seen = datetime.fromtimestamp(ts_value, tz=timezone.utc)
   ```

3. Validation in `sensor_handler.py:680-727` — Prueft nur ob `ts` vorhanden und `int` ist, NICHT ob `ts > 0`:
   ```python
   if "ts" not in payload and "timestamp" not in payload:
       return {"valid": False, ...}
   ts_value = payload.get("ts", payload.get("timestamp"))
   if not isinstance(ts_value, int):
       return {"valid": False, ...}
   # ← KEIN CHECK: if ts_value <= 0
   ```

**Root Cause:** Wokwi-Simulation hat keinen NTP-Sync → sendet `ts=0` bis zum Boot. Server akzeptiert `ts=0` als gueltig und speichert `created_at = 1970-01-01`. API-Zeitfilter (`WHERE created_at > NOW() - interval '24 hours'`) schliesst diese Daten aus.

**Kausale Verkettung:** BUG-05 → BUG-06 (ts=0 → last_seen=1970 → Maintenance setzt offline → Flicker)

**Blast Radius:** ALLE Wokwi-ESPs und ALLE ESPs ohne NTP-Sync. Datenverlust: Readings sind in DB aber unerreichbar via API.

**Fix-Komplexitaet:** Einzeiler (2 Stellen)
```python
# sensor_handler.py (nach Zeile 329):
if esp32_timestamp_raw is None or esp32_timestamp_raw <= 0:
    esp32_timestamp = datetime.now(timezone.utc).replace(tzinfo=None)

# heartbeat_handler.py (nach Zeile 202):
if ts_value <= 0:
    last_seen = datetime.now(timezone.utc)
```

### BUG-09 Detail: DS18B20 processed_value null (MITTEL)

**Code-Review-Ergebnis:** Invertierter raw_mode Default + fehlender Fallback

**Exakte Code-Stellen:**

1. `sensor_handler.py:717` — raw_mode Default ist `True`:
   ```python
   if "raw_mode" not in payload:
       payload["raw_mode"] = True  # ← Default: raw_mode=True
   ```

2. Processing-Pipeline — Wenn `raw_mode=True`, wird Processing uebersprungen:
   ```python
   if raw_mode:
       processed_value = None  # ← Kein Processing
       unit = ""               # ← Leere Unit
   ```

3. `DS18B20Processor` in `sensor_type_registry.py` — Hat `pi_enhanced=False` als Default:
   ```python
   # Ohne pi_enhanced → processed_value bleibt None
   # Kein Fallback: processed_value = raw_value
   ```

**Root Cause:** Mock-DS18B20 sendet Payload OHNE `raw_mode`-Feld → Default `raw_mode=True` → Processing-Pipeline wird komplett uebersprungen → `processed_value=None`, `unit=""`.

**Beziehung zu BUG-08:** Auch wenn BUG-08 gefixt waere, wuerden DS18B20-Daten trotzdem `processed_value=null` haben wegen des raw_mode-Defaults.

**Blast Radius:** ALLE DS18B20-Sensoren die kein `raw_mode=false` explizit senden.

**Fix-Komplexitaet:** Einzeiler
```python

if "raw_mode" not in payload:
    payload["raw_mode"] = False  # ← Default auf False (Processing aktiv)
# ODER: Fallback in Processing-Pipeline:
if processed_value is None:
    processed_value = raw_value  # Mindestens raw_value durchreichen
```

### BUG-10 Detail: MiniCard Count falsch (NIEDRIG)

**Code-Review-Ergebnis:** Computed Property zaehlt nur geladene Sensoren, nicht DB-Count

**Exakte Code-Stelle:** `DeviceMiniCard.vue:154-159`:
```typescript
const sensorCount = computed(() => {
  const sensors = props.device.sensors ?? []
  if (sensors.length === 0) return props.device.sensor_count ?? 0
  return groupSensorsByBaseType(sensors).length
})
```

**Root Cause:** `groupSensorsByBaseType(sensors)` zaehlt nur Sensoren die im `device.sensors[]` Array geladen sind. Wenn ein Sensor `config_status: 'pending'` hat, wird er vom Server NICHT im `sensors[]`-Array mitgeliefert (nur confirmed Configs). Aber `device.sensor_count` (DB-Wert) zaehlt ALLE Configs inkl. pending.

- `sensors.length > 0` → Fallback auf `sensor_count` greift NICHT
- `groupSensorsByBaseType()` liefert 3 (nur confirmed)
- `sensor_count` waere 4 (inkl. pending)

**Blast Radius:** Jeder ESP mit pending sensor_configs zeigt falschen Count.

**Fix-Komplexitaet:** Einzeiler
```typescript
// DeviceMiniCard.vue:154-159:
const sensorCount = computed(() => {
  const sensors = props.device.sensors ?? []
  if (sensors.length === 0) return props.device.sensor_count ?? 0
  return Math.max(groupSensorsByBaseType(sensors).length, props.device.sensor_count ?? 0)
})
```

### BUG-WeakWiFi Detail: Weak WiFi Spam (NIEDRIG)

**Code-Review-Ergebnis:** Statischer Schwellwert ohne Hardware-Typ-Filter

**Exakte Code-Stelle:** `heartbeat_handler.py:1091-1093`:
```python
# Check for weak WiFi signal
if wifi_rssi < -70:  # Weak signal
    logger.warning(f"Weak WiFi signal on {esp_id}: rssi={wifi_rssi} dBm")
```

**Root Cause:** Wokwi-Simulation sendet festen RSSI-Wert `-72 dBm` (oder aehnlich). Schwellwert ist `-70 dBm`. Kein Filter fuer `hardware_type` (Mock/Wokwi vs. Real). Jeder Heartbeat → Warning-Log.

**Inkonsistenz:** Es gibt 3 verschiedene RSSI-Schwellwerte im Codebase:
- `heartbeat_handler.py:1092` → `-70 dBm`
- `esp_heartbeat.py:216` → `-70 dBm`
- Test-Fixtures → `-75 dBm`

**Blast Radius:** Nur Log-Noise. Keine Funktionsstoerung. Aber 1 Warning pro Heartbeat (~alle 30-60s) pro Wokwi-ESP.

**Fix-Komplexitaet:** Einzeiler

## Fazit

### Gesamteindruck Frontend-Qualitaet
Das AutomationOne-Frontend bietet eine **solide Basis** fuer die Geraete- und Sensorverwaltung. Die 3-Level-Zoom-Architektur (L1 → L2 → Config-Panels) ist durchdacht und die Drag-and-Drop-Konfiguration von Sensoren und Aktoren funktioniert intuitiv. Das Dark-Theme ist konsistent, die Datenfluss-Pipeline (MQTT → DB → WebSocket → UI) liefert Echtzeit-Updates innerhalb ~2 Sekunden.

### Top-4 kritischste Bugs (nach Code-Review)

1. **BUG-05+06 (KRITISCH, KAUSALKETTE):** `ts=0` von Wokwi → `last_seen=1970` → Maintenance setzt offline → Flicker → 403. **Root Cause:** Fehlender `ts<=0` Guard in `sensor_handler.py:329` + `heartbeat_handler.py:202`. Fix: Server-Timestamp als Fallback.
2. **BUG-07 (KRITISCH):** Device-Delete → 500. **Root Cause:** `.astext` auf `JSON`-Spalte (nicht JSONB) in `notification_repo.py:621`. Fix: `cast(col, Text)` — Einzeiler.
3. **BUG-08 (KRITISCH, KAUSALKETTE):** `scalar_one_or_none()` in `sensor_repo.py:109` ohne `onewire_address`-Filter. Die korrekte 4-Way-Methode existiert bereits auf Zeile 864, wird aber nicht aufgerufen. → DS18B20-Datenverlust.
4. **BUG-02 (KRITISCH):** `DateTime` ohne `timezone=True` in `subzone.py:128`. Verstoss gegen eigene api-rules.md Regel. Alembic-Migration noetig.

### Top-3 wichtigste Frontend-Luecken

1. **FL-01 (HOCH):** Zone-Erstellen-Button auf L1 disabled — grundlegender Workflow blockiert
2. **FL-03 (MITTEL):** Aktoren nicht auf L1 MiniCards — Informationsluecke auf Uebersichtsebene
3. **FL-02 (MITTEL):** Kein Subzone-Management-Dialog — implizite Erstellung ist nicht intuitiv

### Top-3 UX-Verbesserungen

1. **Schwellwert-Visualisierung:** Werte ausserhalb min/max rot/orange markieren
2. **Error-Messages:** Spezifische statt generische Fehlertexte (403 → "Device muss online sein")
3. **Breadcrumbs:** Navigationspfad auf L2 anzeigen (Hardware > Zone > Device)
