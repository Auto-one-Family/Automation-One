## Executive Summary

Auf Monitor L1 zeigt dieselbe **ZoneTileCard** zwei getrennte Informationskanäle:

1. **KPI-Zeile** (Standard-Slot): Werte aus `useZoneKPIs` → `aggregateZoneSensors(group.devices)` — **Zonenmittel** pro abstrakter Kategorie (Ø explizit in `ZoneTileCard`).
2. **Mini-Dashboard** (`InlineDashboardPanel`, `scope='zone-tile'`): bis zu **zwei `gauge`-Widgets**, deren Konfiguration einen **konkreten Sensor** per `sensorId` (`espId:gpio:sensorType`) trägt. Auswahl erfolgt über **`TILE_SENSOR_PRIORITY`** und Sortierung — **kein** Aufruf von `aggregateZoneSensors`.

**Folge:** Beide Quellen sind **live** (reagieren auf `espStore.devices`), aber **semantisch verschieden**: Gauge = **Repräsentativ-/Spot-Sensor** nach Prioritätsheuristik; KPI = **aggregiertes Zonenmittel** mit Stale-/Null-Filtern aus `sensorDefaults.ts`.

Verwandter Bericht (Label Ø vs. Hardware-Header): `docs/analysen/BERICHT-monitor-zone-kpi-ssot-zoneplate-zonetilecard-2026-04-10.md`.

/** Sensor type priority for zone-tile mini-widgets (highest first) */
const TILE_SENSOR_PRIORITY: string[] = [
  'temp', 'humi', 'vpd', 'ph', 'ec', 'co2', 'soil', 'light', 'pressure', 'flow',
]

function getTileSensorPriority(sensorType: string): number {
  const idx = TILE_SENSOR_PRIORITY.findIndex(p => sensorType.includes(p))
  return idx >= 0 ? idx : TILE_SENSOR_PRIORITY.length
}

/**
 * Ensures a tile mini-dashboard exists for the given zone.
 * Creates gauge widgets for the top-2 sensor types (by priority).
 * ...
 */
function ensureZoneTileDashboard(zoneId: string, zoneName: string): void {
  // ...
  const sorted = [...sensors].sort((a, b) => getTileSensorPriority(a.sensorType) - getTileSensorPriority(b.sensorType))
  const picked: typeof sensors = []
  const seenBaseTypes = new Set<string>()
  for (const s of sorted) {
    const base = TILE_SENSOR_PRIORITY.find(p => s.sensorType.includes(p)) ?? s.sensorType
    if (seenBaseTypes.has(base)) continue
    seenBaseTypes.add(base)
    picked.push(s)
    if (picked.length >= 2) break
  }
  // ...
  for (let i = 0; i < picked.length; i++) {
    const s = picked[i]
    dashStore.addWidget(layout.id, {
      type: 'gauge',
      x: i * 6, y: 0, w: 6, h: 1,
      config: {
        sensorId: `${s.espId}:${s.gpio}:${s.sensorType}`,
        title: getSensorLabel(s.sensorType),
        zoneId,
      },
    })
  }
}

// Auto-generate tile mini-dashboards for L1 zone tiles (gauge widgets only).
// Per-zone tracking: retries zones whose sensors weren't loaded yet on first trigger.
watch([filteredZoneKPIs, () => espStore.devices.length], ([zones]) => {
  if (zones.length === 0) return
  for (const zone of zones) {
    if (ensuredTileZoneIds.has(zone.zoneId)) continue
    const hasSensors = espStore.devices.some(
      d => d.zone_id === zone.zoneId && (d.sensors?.length ?? 0) > 0,
    )
    if (!hasSensors) continue
    ensuredTileZoneIds.add(zone.zoneId)
    ensureZoneTileDashboard(zone.zoneId, zone.zoneName)
  }
}, { immediate: true })

## Mental-Model (Operator)

| Anzeige | Bedeutung (IST) |
|---------|------------------|
| KPI „Ø …“ unter dem Zonennamen | Mittelwert über **alle** in die Aggregation einbezogenen Messpunkte der Zone (nach Kategorie, Filter siehe `aggregateZoneSensors`). |
| Kreis-Gauge in der Kachel | Messwert **eines** ausgewählten Sensors (erster Treffer pro Prioritäts-Basis-Typ nach Geräte-/Sensor-Reihenfolge in `espStore.devices`). |

Wenn mehrere gleichartige Sensoren in der Zone existieren, können **Gauge und Ø voneinander abweichen**, ohne dass ein technischer Fehler vorliegt.

| Option | Beschreibung | Aufwand | Kurzrisiko |
|--------|----------------|---------|------------|
| **A — Label / Legende** | Kleines Badge oder Untertitel am Gauge: z. B. „Repräsentativ“ oder „Sensor A“; optional Kurztext unter KPI: „Zonenmittel“. | Gering | Kein Datenrisiko; Platzbedarf in der Kachel (~70px Höhe). |
| **B — Tooltip / `title`** | `title`-Attribut oder Design-System-Tooltip auf Gauge und/oder KPI: erklärt Spot vs. Ø in einem Satz. | Gering | Entdeckbarkeit ohne Hover auf Touch schwächer — ggf. Icon-Hinweis. |
| **C — Heuristik angleichen** | Auswahl des Tile-Sensors so ändern, dass er z. B. dem Sensor am nächsten am Zonen-Ø entspricht (pro Kategorie). | Mittel | Mehr CPU bei vielen Sensoren; Definition „nächster“ muss spezifiziert werden (Einheit, Ausreißer). |
| **D — Gauge an Zonenmittel binden** | Neuer Anzeigemodus: Gauge erhält **aggregierten** Wert (Prop oder virtuelle Datenquelle), konsistent mit `aggregateZoneSensors`; optional Min/Max aus Aggregation. | Höher | `GaugeWidget`/Config-Schema erweitern; Konsistenz mit Stale-Regeln der Aggregation; mögliche Sync-/Persistenz von Widget-Configs. |
| **E — `useZoneKPIs`-Reuse im Tile** | KPI-Zahlen als einzige Kennzahl, Gauge optional abschaltbar oder nur bei einem Sensor in der Zone. | Mittel | Produktentscheid: Informationsdichte vs. Schnellüberblick. |

Empfehlung für schnelle UX-Klarheit ohne Architekturbruch: **A + B** kombinieren; **D** nur nach explizitem Gate und UX-Spezifikation (Skala, Stale, Mobile/Guest-Zonen).

## Abhängigkeiten und Hinweise für Tickets

- **Performance:** Zonenmittel ist bereits in `useZoneKPIs` debounced; zusätzliche Abgleich-Heuristik (Option C) sollte nicht pro Frame laufen, sondern an bestehende Recompute-Zyklen anbinden.
- **Reuse:** `ZoneKPI.aggregation` könnte für Option D als Quelle dienen, müsste aber in `InlineDashboardPanel` / Gauge-Mount verfügbar gemacht werden (Prop-Drill oder kleiner Provide), ohne `aggregateZoneSensors` doppelt logisch zu verzweigen.
- **Konsistenz:** `scope='zone-tile'` trennt Layouts von `generateZoneDashboard()` — bei Änderungen an Widget-Config die Kollisionen mit Server-Sync und `getZoneMiniPanelId` prüfen.
