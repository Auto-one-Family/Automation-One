# Editor Anzeigeort - Kanonisches Datenmodell (Vorschlag, 2026-04-15)

**Ziel:** Eine einzige, eindeutige Source of Truth fuer Platzierung und Rendering von Dashboard- und Widget-Inhalten in Monitor L1/L2.

---

## 1) Problem mit aktuellem Modell

Aktuell wird Platzierung ueber gemischte Felder entschieden:

- Dashboard-Ebene: `scope`, `zoneId`, `target.view`, `target.placement`, `target.order`, `autoGenerated`
- Widget-Ebene: `config.zoneId`, Sensorauswahl mit Zone/Subzone-Labels

Das fuehrt zu:

1. doppelte Semantik (`zone-tile` ueber `scope` und implizit ueber `target`)  
2. fehlende Subzone-Platzierung (`subzoneId` fehlt als Zielsignal)  
3. konkurrierende Regeln fuer denselben Render-Slot

---

## 2) Kanonisches Zielmodell

## 2.1 Pflichtfelder (neu)

### `display_context`

Definiert den **konkreten Renderkontext**:

- `monitor_inline`
- `monitor_bottom`
- `monitor_side`
- `hardware_side`
- `zone_inline`
- `subzone_inline`

### `entity_scope`

Definiert die **fachliche Bindung**:

- `global`
- `zone`
- `subzone`

### `render_role`

Definiert die **Darstellungsrolle**:

- `dashboard_panel` (klassisches Panel)
- `widget_card` (Card in bestehendem Kartenkontext)
- `zone_tile` (L1 ZoneTile-Erweiterung)

### `placement_priority`

Numerischer Wert fuer Reihenfolge/Kollision:

- `0..999` (kleiner = frueher)
- Default: `100`

---

## 2.2 Referenzobjekt (pro Platzierungseintrag)

```json
{
  "placement": {
    "display_context": "subzone_inline",
    "entity_scope": "subzone",
    "render_role": "widget_card",
    "placement_priority": 120,
    "zone_id": "zone-a",
    "subzone_id": "subzone-3"
  }
}
```

---

## 3) Kollisionen und Invarianten

## 3.1 Harte Invarianten

1. `entity_scope=global` -> `zone_id` und `subzone_id` muessen `null` sein.  
2. `entity_scope=zone` -> `zone_id` Pflicht, `subzone_id=null`.  
3. `entity_scope=subzone` -> `zone_id` und `subzone_id` Pflicht.  
4. `render_role=zone_tile` erlaubt nur `display_context=zone_inline` und `entity_scope=zone`.  
5. `display_context=subzone_inline` erlaubt nur `entity_scope=subzone`.

## 3.2 Kollisionsregel

Schluessel:

`(display_context, render_role, zone_id, subzone_id, placement_priority, owner_scope)`

Regel:

- harte Kollision bei identischem Schluessel + konkurrierender exklusiver Rolle  
- bei nicht-exklusiven Rollen nach `placement_priority` stabil sortieren

---

## 4) Render-Entscheidung (einheitlicher Algorithmus)

1. Eingabe: aktive Seite + aktiver Kontext (`zone_id`, `subzone_id`, view-level).  
2. Filter auf `display_context`.  
3. Filter auf Scope-Konsistenz (`global/zone/subzone`).  
4. Filter auf Rolle (`zone_tile`, `widget_card`, `dashboard_panel`) passend zum Slot.  
5. Sortierung nach `placement_priority`, dann `updated_at`, dann `id`.  
6. Ausgabe: eindeutige Liste fuer Renderer.

**Wichtig:** Kein Renderer darf eigene Zusatzregeln auf alten Feldern fahren.

---

## 5) Mapping ALT -> NEU

| Alt | Neu (Regel) |
|---|---|
| `target.view='monitor'`, `placement='inline'`, `scope='cross-zone'` | `display_context=monitor_inline`, `entity_scope=global`, `render_role=dashboard_panel` |
| `target.view='monitor'`, `placement='inline'`, `scope='zone'`, `zoneId=x` | `display_context=zone_inline`, `entity_scope=zone`, `render_role=dashboard_panel`, `zone_id=x` |
| `scope='zone-tile'`, `zoneId=x` | `display_context=zone_inline`, `entity_scope=zone`, `render_role=zone_tile`, `zone_id=x` |
| `target.view='monitor'`, `placement='bottom-panel'` | `display_context=monitor_bottom`, `render_role=dashboard_panel` |
| `target.view='monitor'`, `placement='side-panel'` | `display_context=monitor_side`, `render_role=dashboard_panel` |
| `target.view='hardware'`, `placement='side-panel'` | `display_context=hardware_side`, `render_role=dashboard_panel` |

---

## 6) ZoneTile-Sonderfall (vereinheitlichungspflichtig)

## Ist

- ZoneTile wird ueber `scope='zone-tile'` gesucht.
- Gleichzeitig wird oft `target=monitor/inline` gesetzt.
- Renderer fuer L1 und L2 nutzen unterschiedliche Filter.

## Soll

- ZoneTile nur ueber `render_role=zone_tile` plus `display_context=zone_inline`.
- Kein eigener Sonderfilter auf Legacy-Feld `scope='zone-tile'`.

## Migrationsregel

1. Wenn `scope='zone-tile'`: mappe auf neues ZoneTile-Placement.  
2. Ignoriere widerspruechliches altes `target` fuer ZoneTile im Zielmodell.  
3. Schreibe Ergebnis explizit in `placement` und markiere `migration_version`.

---

## 7) API- und Store-Auswirkung

## Store

- Neue zentrale Funktion: `resolvePlacements(context)`  
- Alte Filter (`inlineMonitorPanelsCrossZone`, `inlineMonitorPanelsForZone`) werden durch kanonische Filter ersetzt.

## API DTO

- Neues Feld `placement` als Objekt (siehe 2.2).
- Legacy-Felder waehrend Uebergang read-only weiter akzeptieren.

## UI Editor

- "Anzeigeort" schreibt nur noch `placement`.
- "Scope"-Dropdown wird kontextsensitiv aus `entity_scope` abgeleitet.
- Subzone-Auswahl wird als Platzierungsziel ergaenzt.

---

## 8) Validierung

Pflichtvalidierung bei Save:

1. semantische Konsistenz (`display_context` vs `entity_scope`)  
2. Pflichtfelder je Scope  
3. Kollisionserkennung vor Persist  
4. klare Fehlermeldung pro Konflikt

---

## 9) Ergebnis

Mit `display_context + entity_scope + render_role + placement_priority` wird Platzierung erstmals eindeutig und konfliktarm.  
Dieses Modell ist die Grundlage fuer:

- Paket 1 (Anzeigeort-SSOT)
- Paket 2 (Zone/Subzone-Widget-Einbindung)
- Paket 5 (Migration und Rueckwaertskompatibilitaet)
