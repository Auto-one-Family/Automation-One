# Editor Anzeigeort - IST/SOLL Analyse (2026-04-15)

**Datum:** 2026-04-15  
**Prioritaet:** P0  
**Scope:** `El Frontend` Dashboard-Editor + Monitor L1/L2 + Dashboard-Store  
**Methode:** Codebasierte Ist-Analyse mit reproduzierbaren Pfaden

---

## 1) Kurzfazit

Die aktuelle Anzeigeort-Logik ist funktional, aber nicht kanonisch. Sie verteilt die Platzierungsentscheidung auf mehrere Felder (`scope`, `zoneId`, `target.view`, `target.placement`, `autoGenerated`) und auf mehrere Renderpfade (L1 ZoneTile, L2 Inline/Side/Bottom, L3 Viewer).  

Dadurch entstehen drei Hauptprobleme:

1. **Ueberflutung:** auto-generierte Zonen-Dashboards wachsen ungegrenzt mit Sensor/Aktor-Bestand und werden bei Strukturwechseln erneut beschrieben.  
2. **Doppelsteuerung:** `zone-tile` wird ueber `scope` gerendert, traegt aber zusaetzlich `target=monitor/inline`; beide Signale koennen widerspruechlich sein.  
3. **Persistenz-Desyncs:** Teile der ZoneTile-Migration mutieren Layout-Objekte direkt, ohne den Store-Patchpfad zu nutzen (kein garantierter Persist/Sync-Trigger).

## 3) Verbindliche Analysefragen (1-10)

## 3.1 Welche Quellen bestimmen aktuell den Anzeigeort?

**Belegt**

1. `target.view` + `target.placement` (Store-Filter):  
   - inline: `inlineMonitorPanelsCrossZone`, `inlineMonitorPanelsForZone`  
   - side: `sideMonitorPanels`  
   - bottom: `bottomMonitorPanels`  
2. `scope` + `zoneId`: bestimmt Zone-spezifische Zuordnung fuer L2 (`scope === 'zone' && zoneId match`).  
3. Sonderpfad `scope === 'zone-tile'` in Monitor L1 via `getZoneMiniPanelId()` (unabhaengig vom `target`-Filter).

**Kernaussage:** Anzeigeort wird heute **nicht** von einem Feld bestimmt, sondern von einer Kombination aus Store-Filtern + Sonderlogik im View.

---

## 3.2 Welche Felder/Flags wirken gleichzeitig auf denselben Anzeigeplatz?

**Belegt**

- `scope`, `zoneId`, `target.view`, `target.placement`, `target.order`, `autoGenerated`.
- Besonders kritisch: `zone-tile` + `target=monitor/inline`.

**Konfliktbild**

- `scope='zone-tile'` steuert L1-Renderung ueber `getZoneMiniPanelId()`.
- Dasselbe Layout kann durch `target=monitor/inline` in Cross-Zone-Filter laufen (`scope !== 'zone'`), was fuer L2 Inline-Panels relevant ist.

---

## 3.3 Wann entstehen mehrere Dashboards fuer denselben fachlichen Zweck?

**Belegt**

1. **Pro Zone** wird bei Erstbesuch in L2 ein auto-Zonen-Dashboard erzeugt (`generateZoneDashboard()` in MonitorView-Watcher).  
2. **Pro Zone** wird in L1 ein separates `zone-tile`-Layout sichergestellt (`ensureZoneTileDashboard()`).
3. FAB-Add (`AddWidgetDialog`) verwendet fuer Zone-Widgets `scope='zone'` Dashboards; ZoneTile nutzt eigenen `scope='zone-tile'` Pfad.

**Folge:** Eine Zone kann mindestens zwei verschiedene Dashboard-Artefakte haben (L2-Zone + L1-ZoneTile), plus weitere manuelle Layouts.

---

## 3.4 Wann wird "ZoneTile" doppelt oder widerspruechlich konfiguriert?

**Belegt**

- ZoneTile wird fachlich ueber `scope='zone-tile'` identifiziert.
- Gleichzeitig wird bei Erstellung `target={ view:'monitor', placement:'inline' }` gesetzt.
- L1-Rendering prueft **nicht** den `target`, sondern nur `scope/zoneId`.

**Widerspruchspunkte**

1. `target` kann geaendert/entfernt werden, L1 kann trotzdem rendern (weil `scope` reicht).  
2. `target=inline` kann zusaetzlich L2-Effekte ausloesen, sobald Widgets im ZoneTile-Layout vorhanden sind.

---

## 3.5 Wie wird Zone/Subzone aktuell im Editor modelliert?

**Belegt**

- Dashboard-Level: `scope` kennt nur `zone`, `zone-tile`, `cross-zone`, `sensor-detail`.  
- Dashboard-Level: `zoneId` vorhanden, **kein `subzoneId`** im Layoutmodell/DTO.
- Widget-Level: Sensorauswahl nutzt gruppierte Optionen Zone/Subzone (`useSensorOptions`), aber Zielort bleibt Dashboard-basiert.

**Ist-Zustand**

- Subzone existiert heute als **Filter/Label im Auswahlprozess**, nicht als kanonisches Platzierungsziel fuer Dashboard-Instanzen.

---

## 3.6 Koennen Widgets heute direkt in Zone/Subzone-Karten gerendert werden?

**Belegt**

- **Zone (L2):** ja, indirekt ueber InlineDashboardPanel im Zonendetail, wenn Layout als monitor/inline geroutet ist.
- **Subzone-Karte:** nein, es gibt keinen Dashboard-Container pro Subzone-ID und kein `subzoneId` im Layoutmodell.

**Stabilitaet**

- Zone-inline ist grundsaetzlich stabil.  
- Subzone-direkt ist strukturell nicht implementiert (kein eindeutiger Render-Slot pro Subzone).

---

## 3.7 Welche Auto-Generierungsregeln existieren und wann triggern sie?

**Belegt**

1. `generateZoneDashboard(zoneId, devices, zoneName)`  
   - Trigger: MonitorView Watcher bei Zonenaufruf/Device-Aenderung.  
   - Verhalten: erstellt/aktualisiert auto Zone-Dashboard mit Widgets aus Sensoren/Aktoren.
2. `ensureZoneTileDashboard(zoneId, zoneName)`  
   - Trigger: L1-Zonenwatcher mit Sensor-Check.  
   - Verhalten: stellt `zone-tile`-Shell sicher.
3. `AddWidgetDialog`  
   - Trigger: FAB-Flow beim Hinzufuegen.  
   - Verhalten: generiert Zone-Dashboard, falls keines existiert.

---

## 3.8 Welche Persistenzkette gilt pro Aktion?

**Kanonischer Pfad (belegt)**

Editor/FAB -> Store Mutation (`applyLayoutPatch`/`saveLayout`/`setLayout*`) -> `persistLayouts()` (localStorage) -> debounce `syncLayoutToServer()` -> API create/update -> Merge bei `fetchLayouts()`.

**Desync-gefaehrdeter Pfad (belegt)**

`ensureZoneTileDashboard()` mutiert bestehende Layoutobjekte direkt (`existing.widgets = []`, Dim-Updates, direkte Feldzuweisungen bei neuem Layout), ohne garantierten Patch-Sync-Pfad.

---

## 3.9 Wo entstehen stille Desyncs zwischen Editorzustand und Monitoranzeige?

**Belegt**

1. Direkte Objektmutation in `MonitorView` (ZoneTile-Migration) ohne expliziten Persist/Sync-Call.  
2. Globales Slot-Override in `setLayoutTarget()` (ein Layout uebernimmt Slot, anderes verliert `target`) ohne klare Konfliktfuehrung im Monitor-Kontext.  
3. Kommentar/Intention vs. echtes Rendering: `inlineMonitorPanelsCrossZone` ist fuer L1+L2 kommentiert, wird real aber nur in L2 gerendert.

---

## 3.10 Welche Konfliktfaelle gibt es?

**Belegt**

- Gleicher `target`-Slot (view+placement): globaler Konflikt, alter Slot-Holder wird ent-targetet.
- ZoneTile mit Widgets + `target=monitor/inline`: potentiell gleichzeitig L1 und L2 sichtbar.
- Kein `subzoneId` im Layout: mehrere Inhalte konkurrieren im gleichen zonenweiten Bereich.
- Auto-generated vs. manuell: automatische Regeneration kann manuelle Erwartung ueberschreiben (wenn weiter `autoGenerated=true`).

## 4) Lebenszyklusanalyse (Create -> Configure -> Save -> Reload -> Monitor)

## 4.1 Flow A - Editor (CustomDashboardView)

1. **Create:** `createLayout()` (oder Template)  
2. **Configure:** Grid/WC-Config aktualisiert `widgetConfigs`, `autoSave()`  
3. **Save:** `saveLayout()` -> patch + `persistLayouts` + debounced API Sync  
4. **Reload:** `loadLayouts()` zuerst lokal, danach `fetchLayouts()` merge server/local  
5. **Monitor Render:** je nach `target` + `scope` in L2/Side/Bottom (L1 nicht direkt)

**Kanonisch:** ja (Store-Patchpfad)  
**Risiko:** Slot-Konflikt durch `setLayoutTarget()`.

## 4.2 Flow B - Monitor Auto Zone Dashboard

1. Trigger durch Monitor-Watcher bei Zonezugriff/DeviceCount-Aenderung  
2. `generateZoneDashboard()` erstellt/aktualisiert Widgets  
3. Persist + Sync ueber Storepfad  
4. Anzeige in L2 ueber `inlineMonitorPanelsL2`

**Kanonisch:** ja (Store-Patchpfad)  
**Risiko:** Mengenausweitung bei grossen Zonen.

## 4.3 Flow C - Monitor ZoneTile Dashboard

1. Trigger L1-Watcher `ensureZoneTileDashboard()`  
2. Existiert Layout: direkte Migration/Mutation  
3. Neues Layout: `createLayout()` + direkte Feldmutation (`scope`, `zoneId`, `target`, `autoGenerated`)  
4. L1-Anzeige ueber `getZoneMiniPanelId()`

**Kanonisch:** teilweise nein  
**Risiko:** Persistenz-/Sync-Luecken durch direkte Mutation.

## 4.4 Flow D - FAB Add Widget

1. User waehlt Widget + Zone (+ Sensor optional)  
2. `AddWidgetDialog` sucht `zoneDashboards(zoneId)` oder generiert eines  
3. `addWidget()` im Store + Save/Sync  
4. Sichtbar in L2 Inline-Kontext

**Kanonisch:** ja  
**Risiko:** fuegt weitere auto-generated Dashboards hinzu, wenn keines existiert.

## 5) IST gegen SOLL (kompakt)

| Nutzerfall | IST | SOLL | Delta/Ursache | Fix-Typ |
|---|---|---|---|---|
| Widget in bestehende Zone | moeglich, zonenweit (L2) | explizit, kollisionsfrei, reproduzierbar | kein kanonisches Placement-Modell | Datenmodell + Store |
| Widget in Subzone | nur indirekte Sensorfilter, kein Subzone-Slot | explizites Subzone-Ziel + Card im Subzone-Block | `subzoneId` fehlt im Layoutmodell | Datenmodell + Monitor UI |
| ZoneTile konfigurieren | Sonderpfad mit `scope=zone-tile`, plus `target` | eine Quelle der Wahrheit | doppelte Semantik scope vs target | Store + Migration |
| Custom Dashboard unten | moeglich via `target=bottom-panel` | klar priorisiert, mehrere erlaubt | globales Slot-Override in setLayoutTarget | Store-Regeln |
| Auto Dashboard | pro Zone auto + updates | begrenzt, nur bei Mehrwert | fehlende Guardrails, triggerbasiert statt nutzenbasiert | Guardrail-Logik |

## 6) Belegt / Wahrscheinlich / Offen

## Belegt

- Keine Subzone-Platzierung im Dashboardmodell (`subzoneId` fehlt).
- ZoneTile-Doppelung (`scope=zone-tile` + `target=monitor/inline`).
- Direkte ZoneTile-Mutationen ausserhalb Store-Patchpfad.
- Auto-Generation in Monitor-Watchern + FAB.

## Wahrscheinlich

- L2-Ueberfrachtung steigt mit Zonenanzahl linear bis ueberlinear (Widgets pro Zone * Zonen).
- Nutzerwahrnehmung "zufaellige Dashboards" entsteht durch automatische Erstellung im Kontextwechsel.

## Offen

- Gewuenschte Maximalanzahl sichtbarer Panels pro L2 fuer konkrete Betriebsprofile.
- Fachliche Prioritaet: Subzone als eigenstaendiges Dashboard-Target oder nur Widget-Target.

---

## 7) Reproduzierbare Problemfaelle

1. **ZoneTile-Semantikbruch**
   - Oeffne L1 Monitor, nutze ZoneTile-Editorlink.
   - Fuelle ZoneTile mit erlaubten Widgets.
   - Wechsle in L2 einer beliebigen Zone.
   - Beobachtung: ZoneTile-Inhalt kann ausserhalb des eigentlichen L1-Kontexts auftauchen (cross-zone inline Pfad).

2. **Persistenzluecke bei ZoneTile-Migration**
   - Trigger Legacy-Migrationpfad (bestehendes auto zone-tile Layout mit alten Widgetdimensionen).
   - Kein expliziter Store-Patch/Save-Aufruf in `ensureZoneTileDashboard`.
   - Reload/Testlauf zeigt potentiell inkonsistente Rehydration.

3. **Target-Konflikt**
   - Setze im Editor Dashboard A auf `monitor/inline`.
   - Setze Dashboard B auf denselben Slot.
   - Dashboard A verliert `target` implizit.
