# V19-F05 — Auto-generierte Dashboards fuer geloeschte Zonen bereinigen

> **Typ:** Feature + Cleanup (Frontend + Backend)
> **Erstellt:** 2026-03-26
> **Prioritaet:** HIGH
> **Geschaetzter Aufwand:** ~2-3h
> **Abhaengigkeit:** Keine

---

## Kontext

### Dashboard-Erstellung in AutomationOne

Dashboards werden auf zwei Wegen erstellt:

1. **Manuell:** Der Benutzer erstellt ein Dashboard ueber den Editor (Dashboard Builder) und waehlt einen Namen, Widgets und Sensoren. Diese Dashboards haben `autoGenerated: false`.
2. **Automatisch:** `generateZoneDashboard()` in `dashboard.store.ts` erstellt ein Dashboard wenn der Benutzer erstmals eine Zone auf Monitor L2 oeffnet oder ein Widget ueber den FAB hinzufuegt. Diese Dashboards haben `autoGenerated: true` und sind an eine `zoneId` gebunden.

### Dashboard-Verwaltung (Phase 7, D1)

In D1 wurde die Dashboard-Verwaltung implementiert:
- **Einzelloeschen:** Trash2-Icon pro Dashboard-Card → Confirm-Dialog → `deleteLayout()`.
- **Bulk-Cleanup:** Button "Auto-generierte aufraeumen (N)" → BaseModal mit Checkbox-Liste → `bulkDeleteLayouts(layoutIds)`.
- **computed `autoGeneratedLayouts`** im Store filtert Dashboards mit `autoGenerated: true`.

### Das Problem

Aktuell existieren **24 Dashboards**, davon **15 auto-generiert**. Die meisten dieser 15 gehoeren zu Zonen die laengst geloescht wurden (durch Tests, Mock-ESPs, Entwicklung):
- "Zelt 1 Dashboard", "Testneu Dashboard", "FINALERTEST Dashboard", "Gewaechshaus-Alpha Dashboard", "Gewaeckshaus Dashboard" (Typo!), "Naehrloesung Dashboard", "Zone 1 Dashboard", "Wokwi-Testzone Dashboard", "Echter ESP Dashboard", "Testzone-Alpha Dashboard", "Test Dashboard"

Der Bulk-Cleanup-Button existiert und funktioniert — **aber er muss manuell gedrueckt werden**. Es gibt keinen automatischen Mechanismus der verwaiste Dashboards erkennt und entfernt.

Ein neuer Benutzer sieht im Dashboard-Dropdown 24 Eintraege wovon >60% Muell sind. Das ist schaedlich fuer die UX — die 5-Sekunden-Regel (IoT Dashboard Best Practice: "Das Dashboard beantwortet in 5 Sekunden ob alles in Ordnung ist") gilt auch fuer die Dashboard-Verwaltung selbst. Ein Benutzer sollte nicht 24 Eintraege durchsuchen muessen um sein relevantes Dashboard zu finden.

## SOLL-Zustand

### Schritt 1: Verwaiste Auto-Dashboards erkennen (Pflicht)

Ein Auto-Dashboard ist "verwaist" wenn:
- `autoGenerated === true` UND
- `zoneId` ist gesetzt UND
- Die Zone mit dieser `zoneId` existiert NICHT mehr im Zone-Store (`zoneEntities`)

**Wichtig — Archivierte Zonen sind NICHT verwaist:**
Der Zone-Store unterscheidet zwischen `activeZones` (status=active) und `archivedZones` (status=archived). Archivierte Zonen bleiben in `zoneEntities` enthalten — nur geloeschte Zonen werden aus `zoneEntities` entfernt (via `filter()` nach erfolgreichem API-Delete). Ein Dashboard einer archivierten Zone darf NICHT geloescht werden.

**Zone-Identifikation:** Das relevante Feld ist `zone_id` (Slug-Format wie `"zone-abc123"`), NICHT `id` (UUID). Dashboard-`zoneId` speichert den Slug.

### Schritt 2: Automatisches Cleanup (eine der folgenden Optionen)

**Option A — Cleanup nach Store-Initialisierung (empfohlen):**

Nachdem BEIDE Stores geladen sind (Zone-Store mit `zoneEntities` UND Dashboard-Store mit `layouts`), wird geprueft ob Auto-Dashboards auf nicht-existente Zonen verweisen. Falls ja, werden sie automatisch geloescht.

**ACHTUNG — Store-Initialisierungsreihenfolge:**
`dashboard.store.ts` ruft `fetchLayouts()` automatisch bei Store-Erstellung auf (Zeile 995). Der Zone-Store wird als separate Pinia-Store geladen. Es gibt **keine Garantie**, dass `zoneEntities` bereits gefuellt ist wenn `fetchLayouts()` abschliesst. Deshalb `cleanupOrphanedDashboards()` **NICHT** direkt in `fetchLayouts()` aufrufen, sondern einen der folgenden Ansaetze waehlen:

- **Bevorzugt: `watch()` im Dashboard-Store** auf `zoneStore.zoneEntities` — reagiert sobald Zonen geladen sind und loest dann das Cleanup aus. Einmalig mit `{ once: true }`.
- **Alternativ: Aufruf aus uebergeordneter Komponente** (z.B. `App.vue` oder `MonitorView.vue`) nachdem beide Stores initialisiert sind.

**Vorbedingung `fetchZoneEntities()`:** Die Funktion akzeptiert einen optionalen `status`-Parameter (Default laedt nur aktive Zonen). Fuer korrektes Orphan-Detection muss sichergestellt werden, dass auch archivierte Zonen geladen sind (`zoneEntities` enthaelt active + archived), damit deren Dashboards NICHT faelschlicherweise als verwaist geloescht werden. Pruefen ob der initiale Aufruf ohne Parameter alle nicht-geloeschten Zonen laedt.

Pseudocode:
```typescript
async function cleanupOrphanedDashboards() {
  const existingZoneIds = new Set(
    zoneStore.zoneEntities.map(z => z.zone_id)
  )
  const orphaned = layouts.value.filter(l =>
    l.autoGenerated && l.zoneId && !existingZoneIds.has(l.zoneId)
  )
  if (orphaned.length > 0) {
    await bulkDeleteLayouts(orphaned.map(l => l.id))
  }
}
```

**Hinweis Bulk-Delete:** Es gibt keinen Bulk-DELETE-Endpoint im Backend — nur `DELETE /api/v1/dashboards/{dashboard_id}` (einzeln). `bulkDeleteLayouts()` im Frontend iteriert und ruft `deleteLayoutFromServer()` pro Dashboard auf. Bei 15 verwaisten Dashboards = 15 sequentielle DELETE-Requests. Funktioniert, aber ggf. langsam. Fuer den aktuellen Anwendungsfall akzeptabel.

**Option B — Cleanup beim Zone-Loeschen (Backend-seitig):**

Wenn eine Zone geloescht wird (Soft-Delete: `status='deleted'`, `deleted_at` gesetzt), loescht das Backend auch die zugehoerigen Auto-Dashboards. Das setzt voraus dass die Dashboard-API ein `zoneId`-Feld hat und das Backend beim Zone-Delete die zugehoerigen Dashboards identifizieren kann.

**Option C — Cleanup beim Zone-Loeschen (Frontend-seitig, komplementaer zu A):**

Im Zone-Store wird nach dem erfolgreichen Zone-Delete ein Cleanup im Dashboard-Store ausgeloest. Konkret: `deleteZoneEntity()` im Zone-Store (Zeile 131-133) entfernt die Zone aus `zoneEntities`. Direkt danach `dashboardStore.cleanupOrphanedDashboards()` aufrufen.

**Option C ist komplementaer zu Option A** — nicht alternativ. Option A bereinigt historischen Muell beim App-Start, Option C verhindert neuen Muell an der Quelle. Beide zusammen implementieren ergibt den robustesten Ansatz.

### Schritt 3: Sofortige Bereinigung der bestehenden 15 verwaisten Dashboards

Unabhaengig von der automatischen Loesung muessen die aktuell 15 verwaisten Dashboards entfernt werden. Das kann entweder manuell ueber den bestehenden Bulk-Button passieren, oder der automatische Mechanismus macht es beim naechsten Laden.

---

## Vorgehen

1. **Bestandsaufnahme:** `GET /api/v1/dashboards` aufrufen und pruefen welche Dashboards `autoGenerated: true` und welche `zoneId` sie haben.
2. **Zone-Existenz pruefen:** Fuer jede `zoneId` pruefen ob die Zone noch in `zoneEntities` existiert.
3. **Option A implementieren** (Frontend-Cleanup nach Store-Init):
   - In `dashboard.store.ts` eine `cleanupOrphanedDashboards()`-Funktion hinzufuegen.
   - Ausloesung via `watch()` auf `zoneStore.zoneEntities` (mit `{ once: true }`) — NICHT direkt in `fetchLayouts()`.
   - Sicherstellen dass `zoneEntities` aktive UND archivierte Zonen enthaelt (nur geloeschte Zonen fehlen).
4. **Option C implementieren** (komplementaer):
   - In `zone.store.ts` nach erfolgreichem `deleteZoneEntity()` ein `dashboardStore.cleanupOrphanedDashboards()` aufrufen.
5. **Testen:** Zone loeschen → Dashboard-Liste pruefen → verwaistes Dashboard muss weg sein.
6. **Verifikation:** Dashboard-Dropdown zeigt nur noch relevante Dashboards.
