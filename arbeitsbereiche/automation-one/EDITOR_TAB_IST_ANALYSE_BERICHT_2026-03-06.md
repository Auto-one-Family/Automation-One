## 2. Datenfluss & Stores

### B1: Genutzte Stores

| Store | Verwendung |
|-------|------------|
| **dashboard.store** | layouts, activeLayoutId, activeLayout, createLayout, saveLayout, deleteLayout, setLayoutTarget, inlineMonitorPanels, sideMonitorPanels, generateZoneDashboard, DASHBOARD_TEMPLATES, exportLayout, importLayout |

| **dragState.store** | isDraggingDashboardWidget, dashboardWidgetPayload, endDrag() |
| **esp.store** | devices, fetchAll() (für Widget-Daten) |

### B2: Layout-Persistierung

| Ebene | Mechanismus |
|-------|-------------|
| **localStorage** | Key `automation-one-dashboard-layouts`, JSON aller Layouts |
| **Server-API** | `dashboardsApi.list()`, `create()`, `update()`, `delete()` |
| **Sync-Flow** | `saveLayout()` → `persistLayouts()` → `syncLayoutToServer()` (debounced 2s) |
| **Merge** | Server-Layouts haben Priorität (matched by serverId), lokale ohne serverId bleiben |

### B3: DashboardLayout Interface (vollständig)

```typescript
interface DashboardLayout {
  id: string
  serverId?: string           // Server UUID nach erstem Sync
  name: string
  description?: string
  createdAt: string
  updatedAt: string
  widgets: DashboardWidget[]
  scope?: DashboardScope      // 'zone' | 'cross-zone' | 'sensor-detail'
  zoneId?: string
  autoGenerated?: boolean
  sensorId?: string
  target?: DashboardTarget
}
```

### B4: DashboardTarget Interface

```typescript
interface DashboardTarget {
  view: 'monitor' | 'hardware'
  placement: 'page' | 'inline' | 'side-panel'
  anchor?: string
  panelPosition?: 'left' | 'right'
  panelWidth?: number
  order?: number
}
```

**placement-Werte (IST):** `page`, `inline`, `side-panel` — **kein `bottom-panel`**

## 3. Target-System & Monitor-Integration

### C1: Wo wird target gesetzt?

- **CustomDashboardView.vue** Zeilen 91–104: `setTarget(view, placement)` ruft `dashStore.setLayoutTarget(layoutId, { view, placement })`
- **Target-Config-Dropdown** (MapPin-Button) Zeilen 479–519: 4 Optionen:
  - Monitor — Inline
  - Monitor — Seitenpanel
  - Übersicht — Seitenpanel (hardware + inline — Label inkonsistent)
  - Anzeigeort entfernen

### C2: Unterstützte placement-Werte

| placement | Verwendung |
|-----------|------------|
| `inline` | Unter Zone-Kacheln (L1) bzw. unter Zone-Content (L2) |
| `side-panel` | Fixiert rechts neben Monitor-Inhalt |
| `page` | Im Interface definiert, aber **nicht im Target-Dropdown** — nur `inline` und `side-panel` wählbar |

### C3: MonitorView — Inline-Panels

| Ebene | Zeilen | Bedingung |
|------|--------|-----------|
| **L1** | 1512–1522 | `dashStore.inlineMonitorPanels` — nach Zone-Tiles und Dashboards-Karte |
| **L2** | 1821–1826 | `dashStore.inlineMonitorPanels` — nach Zone-Content, vor Empty-State |

**Hinweis:** Inline-Panels werden auf L1 und L2 **identisch** gerendert (gleiche Liste, kein Zone-Scoping).

### C4: MonitorView — Side-Panels

| Aspekt | IST |
|--------|-----|
| **Container** | `<aside class="monitor-layout__side">` Zeilen 1838–1846 |
| **Bedingung** | `dashStore.sideMonitorPanels.length > 0` |
| **Layout** | `monitor-layout--has-side` auf Parent → CSS `grid-template-columns: 1fr 300px` |
| **panelPosition/panelWidth** | Im DashboardTarget definiert, aber **nicht im Monitor-Layout ausgewertet** (fixe 300px) |

### C5: "Unteres Panel" (bottom-panel)

| Aspekt | IST |
|--------|-----|
| **Existenz** | **Nicht vorhanden** |
| **placement** | `bottom-panel` nicht in `DashboardTarget.placement` |
| **Monitor-Rendering** | Kein Platzhalter, kein DOM für bottom-panel |

**Ergänzungsort:**  
- `DashboardTarget.placement` um `'bottom-panel'` erweitern  
- `dashboard.store`: neues Computed `bottomMonitorPanels`  
- `MonitorView.vue`: neuer Bereich unter `monitor-layout__main` für bottom-panels

## 6. Datenfluss-Diagramm (Text)

```
User klickt "Widget hinzufügen" (Katalog)
    → addWidget(type)
    → widgetDef = widgetTypes.find(w => w.type === type)
    → id = `widget-${Date.now()}-${random}`
    → config = { title, ...WIDGET_DEFAULT_CONFIGS[type] }
    → widgetConfigs.set(id, config)
    → grid.addWidget({ w, h, minW, minH, id })
    → nextTick: createWidgetElement + mountWidgetComponent
    → autoSave()

User ändert Target (MapPin-Dropdown)
    → setTarget(view, placement)
    → dashStore.setLayoutTarget(layoutId, { view, placement })
    → Enforce uniqueness: anderes Layout verliert gleichen Slot
    → persistLayouts() + syncLayoutToServer()

Monitor rendert Inline-Panels
    → dashStore.inlineMonitorPanels (computed)
    → filter: target.view === 'monitor' && target.placement === 'inline'
    → InlineDashboardPanel v-for panel
    → layout = dashStore.getLayoutById(panel.id)
    → CSS-Grid mit widgetStyle(w)
```

## 9. Offene Fragen

1. **Scope-Setzung im Editor:** Aktuell wird `scope`/`zoneId` nur bei `generateZoneDashboard` und manuell gesetzten Layouts verwendet. Soll der Editor eine explizite "Scope"-Sektion haben (Zone wählen, Cross-Zone, Sensor-Detail)?

2. **monitorRouteForLayout:** Der "Im Monitor anzeigen"-Link erscheint nur wenn `layout.scope` gesetzt ist. Sollen auch Layouts ohne Scope im Monitor angezeigt werden können (z.B. als Cross-Zone-Default)?

3. **Inline-Panels auf L2:** Aktuell werden dieselben `inlineMonitorPanels` auf L1 und L2 gerendert. Soll auf L2 ein Zone-Filter greifen (nur Panels mit `scope === 'zone' && zoneId === selectedZoneId`)?

4. **page placement:** Was ist die konkrete Semantik von `placement: 'page'`? Eigene Route wie `/monitor/dashboard/:id` oder Vollbild-Overlay?
