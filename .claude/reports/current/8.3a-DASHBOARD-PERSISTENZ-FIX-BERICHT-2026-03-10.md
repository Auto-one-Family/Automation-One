## 1. Root-Cause F-01: Warum liefert PUT 404?

### Backend: Vollstaendig implementiert (KEIN Backend-Bug)

Die gesamte Dashboard-Backend-Infrastruktur ist **funktionsfaehig**:

| Komponente | Datei | Status |
|------------|-------|--------|
| API Router | `El Servador/.../api/v1/dashboards.py:32` | `prefix="/v1/dashboards"`, 5 CRUD Endpoints |
| Service | `El Servador/.../services/dashboard_service.py` | CRUD + Ownership-Check |
| Repository | `El Servador/.../db/repositories/dashboard_repo.py` | BaseRepository + Custom Queries |
| ORM Model | `El Servador/.../db/models/dashboard.py:39` | Tabelle `dashboards`, UUID PK |
| Schemas | `El Servador/.../schemas/dashboard.py` | DashboardCreate, Update, Response |
| Migrationen | `alembic/versions/add_dashboards_table.py` + `add_dashboard_target_field.py` | 2 Migrationen |
| App-Mount | `main.py:1097-1100` | `app.include_router(api_v1_router, prefix="/api")` |

**Routing-Kette:** Frontend `/api/v1/dashboards/{id}` → Vite Proxy → FastAPI `/api` + `/v1/dashboards/{id}` ✓

### Frontend: Root Cause identifiziert

**Hypothese H4 bestaetigt:** Die 404 kommt aus der DashboardService-Schicht, weil das Dashboard mit der angegebenen `serverId` nicht in der DB existiert.

**Ursache:** Nach einem DB-Reset (z.B. Docker-Volume neu, Alembic re-init) bleiben in localStorage die alten `serverId`-UUIDs gespeichert. Der Frontend-Code versuchte mit diesen stale UUIDs ein PUT — der Server antwortet korrekt mit 404 ("Dashboard not found").

**Verstaerkender Bug:** `fetchLayouts()` fuehrte den Orphan-Sync (Re-Upload lokaler Dashboards) NUR aus wenn der Server >=1 Dashboard zurueckgibt (`response.data.length > 0` Bedingung in Zeile 356). Bei leerer DB (0 Dashboards) wurde der gesamte Merge/Sync-Block uebersprungen — stale serverIds wurden nie korrigiert.

## 2. Implementierte Fixes

### Fix F-01a: PUT 404 Recovery in `syncLayoutToServer()`

**Datei:** `El Frontend/src/shared/stores/dashboard.store.ts:467-512`

**Vorher:** PUT-Fehler wurde generisch in `lastSyncError` geschrieben, stale `serverId` blieb bestehen → endlose 404-Schleife.

**Nachher:** Bei PUT 404 wird die stale `serverId` geleert und das Dashboard automatisch per POST neu auf dem Server erstellt:

```typescript
// PUT returns 404 → serverId is stale → clear + re-create via POST
if (isHttpStatus(putError, 404)) {
  layouts.value[idx].serverId = undefined
  persistLayouts()
  const response = await dashboardsApi.create(payload)
  layouts.value[idx2].serverId = response.data.id
  persistLayouts()
}
```

Zusaetzlich: `buildSyncPayload()` als eigene Funktion extrahiert (DRY), `isHttpStatus()` Helper fuer HTTP-Status-Erkennung.

### Fix F-01b: fetchLayouts Orphan-Sync bei leerer Server-Response

**Datei:** `El Frontend/src/shared/stores/dashboard.store.ts:365`

**Vorher:** `if (response.success && response.data.length > 0)` — bei leerer DB passierte NICHTS.

**Nachher:** `if (response.success)` — Merge, Migration und Orphan-Sync laufen auch wenn der Server 0 Dashboards hat. Orphans (lokale Dashboards ohne serverId) werden immer zum Server hochgeladen.

### Fix F-02: DELETE 404 Silent-Handling

**Datei:** `El Frontend/src/shared/stores/dashboard.store.ts:525-537`

**Vorher:** Jeder DELETE-Fehler wurde als `logger.warn()` geloggt — bei 8 Duplikaten = 8x Warn-Log.

**Nachher:** 404 bei DELETE wird als `logger.debug()` behandelt (idempotent: "schon geloescht oder nie synchronisiert"). Nur echte Fehler (500, Netzwerk) bleiben als `warn`.

### Fix F-10: Error-Banner Suppression

**Implizit durch F-01a geloest:** PUT 404 wird jetzt intern recovered (clear serverId → POST). Der `lastSyncError` wird nur noch bei echten, nicht-recoverbaren Fehlern gesetzt. Das rote Banner in `CustomDashboardView.vue` erscheint nur noch bei tatsaechlichen Problemen (Netzwerkfehler, Server-Crash).

### Fix F-03: Template-Persistenz

**Kein Code-Fix noetig:** `createLayoutFromTemplate()` ruft bereits `persistLayouts()` (Zeile 583) und `syncLayoutToServer()` (Zeile 584) auf. Die Template-Widgets gehen nicht mehr verloren, weil:
1. localStorage-Persistierung funktioniert (war nie kaputt)
2. Server-Sync funktioniert jetzt (F-01a Fix)
3. Bei Tab-Wechsel (keep-alive) bleibt der In-Memory-State erhalten

## 4. Unerwartete Findings

### F-U1: `serverToLocal()` setzt `id` = `serverId` (Zeile 323)

Wenn Dashboards vom Server geladen werden, ist die lokale ID = Server-UUID. Das ist korrekt und gewollt (`getLayoutById` sucht nach beiden), aber bei Mix aus lokalen und Server-Dashboards kann es verwirrend sein.

### F-U2: Debounce-Timer pro Layout (gut)

`_saveDebounceTimers` nutzt eine Map pro Layout-ID — schnelles Wechseln zwischen Dashboards verliert keine Edits. Dieses Pattern ist korrekt implementiert.

### F-U3: `target`-Type-Mismatch zwischen Store und API

`DashboardTarget` (Interface) ist nicht direkt zu `Record<string, unknown>` (API-Type) assignable. Fix: expliziter Cast in `buildSyncPayload()`.
