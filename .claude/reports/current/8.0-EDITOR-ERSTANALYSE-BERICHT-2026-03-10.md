### A-02: Widget-Config-Schema

**IST-Zustand:**
Ein einziges, flaches Config-Interface fuer ALLE Widget-Typen in `dashboard.store.ts` Zeilen 38-58:

```typescript
config: {
  sensorId?: string        // "espId:gpio:sensorType"
  actuatorId?: string      // "espId:gpio"
  espId?: string
  gpio?: number
  sensorType?: string
  zoneId?: string
  timeRange?: '1h' | '6h' | '24h' | '7d' | 'custom'
  showThresholds?: boolean
  title?: string
  color?: string
  syncTimeAxis?: boolean   // TOTER CODE — wird nirgends ausgewertet
  dataSources?: string     // Komma-getrennte sensorIds fuer multi-sensor
  yMin?: number
  yMax?: number
  zoneFilter?: string | null
  warnLow?: number
  warnHigh?: number
  alarmLow?: number
  alarmHigh?: number
}
```

**sensorId-Parsing:**
Format: `"espId:gpio:sensorType"` (z.B. `"ESP_001:34:temperature"`)
Parsing ist in JEDEM Widget dupliziert:
- `LineChartWidget.vue` Zeilen 73-84
- `GaugeWidget.vue` Zeilen 44-54
- `SensorCardWidget.vue` Zeilen 45-56
- `HistoricalChartWidget.vue` Zeilen 57-70

Kein zentraler Parser.

**Config-Validierung:** KEINE. Props-Defaults sind `undefined`/`null`, kein Schema-Validator.

**Gap-Analyse:**
- sensorId-Parsing sollte in eine Utility-Funktion extrahiert werden
- Fuer neue Widget-Typen (VPD, Heatmap) braucht das Config-Interface Erweiterungen (z.B. `growthPhase`, `resolution`, `comparisonMode`)
- `syncTimeAxis` ist toter Code — kann fuer Korrelations-Widget reaktiviert werden

## Block B: Daten-Pipeline Bestandsaufnahme (Backend)

### B-01: Sensor-Data API

**Endpoint:** `GET /v1/sensors/data` (`sensors.py` Zeilen 1219-1329)

**Vollstaendige Parameter:**

| Parameter | Typ | Default | Status |
|---|---|---|---|
| `esp_id` | str? | None | VORHANDEN |
| `gpio` | int? (0-39) | None | VORHANDEN |
| `sensor_type` | str? | None | VORHANDEN |
| `start_time` | datetime? | NOW()-24h | VORHANDEN |
| `end_time` | datetime? | NOW() | VORHANDEN |
| `quality` | str? | None | VORHANDEN |
| `zone_id` | str? | None | VORHANDEN |
| `subzone_id` | str? | None | VORHANDEN |
| `sensor_config_id` | UUID? | None | VORHANDEN (T13-R2) |
| `limit` | int (1-1000) | 100 | VORHANDEN |
| `resolution` | — | — | **FEHLT** |

**KRITISCH:** `resolution`/Aggregation ist NICHT implementiert. Das Schema `SensorDataQuery` enthaelt ein `aggregation`-Feld (`schemas/sensor.py` Zeile 576-580, Pattern `none|minute|hour|day`), aber der Endpoint wertet es NICHT aus. Response liefert `aggregation=None` hartcodiert. **Totes Schema-Artefakt.**

**Repository-Implementierung** (`sensor_repo.py` Zeilen 487-558): Reines `SELECT ... ORDER BY timestamp DESC LIMIT limit` — kein GROUP BY, keine Window-Funktionen, kein Downsampling.

**Performance-Problem:** 7 Tage bei 30s Intervall = 20.160 Datenpunkte. Mit `limit=1000` bekommt man nur die neuesten 1.000. Die aelteren 19.160 Punkte fehlen. Fuer ein 7-Tage-Chart unbrauchbar.

**Indizes:** `idx_esp_gpio_timestamp`, `idx_sensor_type_timestamp`, `idx_timestamp_desc`, `idx_sensor_data_zone_timestamp` — vorhanden und gut.

**Endpoint 1:** `GET /v1/sensors/data/stats/by-source` (Zeilen 1418-1439)

- **KEIN Zone/Subzone-Filter**

**Gap-Analyse:** Stats-Endpoint existiert und ist maechtig, wird aber von keinem Widget genutzt. Zone/Subzone-Filter fehlen.

### B-03: Actuator States und History — KRITISCH

**`actuator_states` Tabelle** (`models/actuator.py` Zeilen 223-349):
Speichert NUR aktuellen Zustand. Kein History-Tracking. `runtime_seconds` ist ein monotoner Counter, kein Zeitstempel-Paar.

**`actuator_history` Tabelle** (`models/actuator.py` Zeilen 352-477):
EXISTIERT — speichert Commands (Befehle), NICHT State-Transitions mit Dauer.

| Spalte | Typ | Beschreibung |
|---|---|---|
| `id` | UUID PK | |
| `esp_id` | UUID FK nullable | |
| `gpio` | Integer | |
| `command_type` | String(50) | set, stop, emergency_stop |
| `value` | Float nullable | 1.0=ON, 0.0=OFF |
| `issued_by` | String(100) | user:123, logic:456, system |
| `success` | Boolean | |
| `timestamp` | DateTime(tz) indexed | |

**KEIN `duration`-, `started_at`- oder `ended_at`-Feld.** ON/OFF-Uebergaenge sind als separate Kommandos gespeichert.

**History-Endpoint:** `GET /v1/actuators/{esp_id}/{gpio}/history` (Zeilen 1207-1267)
- Parameter: `esp_id`, `gpio`, `limit` (1-100, default: 20)
- **KEIN Zeitfilter** (`start_time`/`end_time` FEHLEN)
- **KEINE Pagination** (nur `limit`)

**Gap-Analyse fuer Aktor-Laufzeitanalyse:**

| Feature | Status |
|---|---|
| Aktuelle Zustaende | VORHANDEN (`actuator_states`) |
| Kommando-History | VORHANDEN (`actuator_history`) |
| ON/OFF-Transitions mit Timestamp | BEDINGT (separate Commands mit value=1.0/0.0) |
| Laufzeit pro Aktivierung | **FEHLT** (muss aus Command-Paaren berechnet werden) |
| Zeitfilter im History-Endpoint | **FEHLT** |
| Pagination | **FEHLT** (max 100) |
| Aggregierte Laufzeit (pro Tag) | **FEHLT KOMPLETT** |

**Fazit:** Aktor-Laufzeitanalyse ist mit dem aktuellen Backend NICHT moeglich. Erweiterungen noetig:
1. `start_time`/`end_time` Filter auf History-Endpoint
2. `limit` erhoehen auf 1000+
3. App-seitige ON/OFF-Paar-Korrelation ODER neuer Aggregations-Endpoint

## Block C: Zone/Subzone-Integration im Editor

### C-01: Zone/Subzone-Bewusstsein

**IST-Zustand:** Der Editor kennt Zonen nur indirekt.
- `WidgetConfigPanel.vue` Zeilen 58-68: `availableZones` wird aus `espStore.devices` abgeleitet (kein `/zones` API-Call)
- `zoneFilter` existiert NUR fuer alarm-list, esp-health, actuator-runtime
- **Subzone-Awareness: NULL.** Weder WidgetConfigPanel noch Sensor-Auswahl kennt Subzonen.

**Dashboard-Level:** `zoneId` als Feld in DashboardLayout. `zoneDashboards(zoneId)` filtert korrekt. Aber einzelne Widgets wissen nichts vom Dashboard-Zonen-Kontext.

**Gap:** Kein Dashboard→Widget Zone-Kontext-Propagation. Jedes Widget muss individuell konfiguriert werden.

---

### C-02: Sensor-Auswahl

**Mechanismus:** Einfaches HTML-`<select>` Dropdown (`WidgetConfigPanel.vue` Zeilen 176-189).

**Struktur:** FLACH. Keine Gruppierung nach Zone/Subzone. Label-Format: `"{name} ({deviceId} GPIO {gpio} — {sensorType})"` (Zeile 77).

**Mehrfachauswahl:** Nicht moeglich im ConfigPanel. MultiSensorWidget hat eigene Chip-basierte Auswahl.

**Gap:** Sensor-Dropdown muss nach Zone → Subzone → Device gruppiert werden.

---

### C-03: Multi-Sensor-Vergleich

**Aktueller Stand:**
- Konfiguration: Comma-separated `dataSources` String
- Sensoren aus verschiedenen Subzonen/Zonen KOENNEN kombiniert werden (kein Filter)
- **Dual-Y-Achsen: NICHT implementiert.** Nur eine Y-Achse. `computedYRange` berechnet gemeinsamen Bereich.
- `syncTimeAxis` ist toter Code
- Threshold-Lines fehlen im MultiSensorWidget (kein annotationPlugin registriert)

### C-04: Dashboard-Scope und Zone-Binding

- `scope`: `'zone' | 'cross-zone' | 'sensor-detail'` (kein `subzone`)
- `zoneId`: Direkt im DashboardLayout
- `generateZoneDashboard()`: Automatische Erstellung basierend auf Zone-Devices mit CATEGORY_WIDGET_MAP

**Gap:** Kein Subzone-Scope. Kein Dashboard→Widget Zone-Kontext-Propagation.

### D-03: Performance

| Limit | Wert | Datei |
|---|---|---|
| LiveLineChart maxDataPoints | 50-60 | useDashboardWidgets.ts |

## Kritische Entscheidungen (brauchen Klaerung)

1. **VPD-Berechnung: Frontend oder Backend?**
   - Empfehlung: **Frontend** fuer Live-Werte (SHT31 liefert Temp+RH via WS)
   - Backend nur fuer historische VPD-Zeitreihen (aus sensor_data Temp+RH Paaren berechnen)

2. **Heatmap Chart-Bibliothek?**
   - Option A: `chartjs-chart-matrix` Plugin (bleibt im Chart.js Oekosystem)
   - Option B: ECharts nur fuer Heatmap (maechtiger aber neue Dependency)
   - Empfehlung: **Option A** — konsistent, weniger Bundle-Size

3. **Virtuelle Sensoren: Eigener Store oder in espStore integriert?**
   - Empfehlung: **Eigener Store** (`virtualSensor.store.ts`) — saubere Trennung

4. **Server-Aggregation: Neuer Endpoint oder Parameter auf existierendem?**
   - Empfehlung: **Parameter auf existierendem** `/sensors/data?resolution=1h` — Schema-Artefakt bereits vorhanden
