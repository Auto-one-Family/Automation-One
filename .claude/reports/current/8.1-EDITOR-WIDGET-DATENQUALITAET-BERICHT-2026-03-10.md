## Bug 1 (HIGH): Historical Y-Achse zeigt Raw-Werte (30000 %RH)

### Root Cause (H1 + H2 kombiniert)

**Datei:** `El Frontend/src/components/charts/HistoricalChart.vue`

1. **Zeile 196 (alt):** `d.raw_value` wurde explizit als Y-Wert verwendet statt `processed_value`. Fuer SHT31: `raw_value=24158` (Register-Wert) vs `processed_value=19.5` (Ā°C).
2. **Zeile 179-185:** `sensor_type` wurde NICHT an `sensorsApi.queryData()` uebergeben. Ohne sensor_type liefert die API alle SHT31-Sub-Types (temp + humidity) gemischt auf einer Y-Achse.
3. **Zeile 243-252:** Live-Append-Watch filterte nicht nach `sensor_type` ā€” neue WS-Events von einem anderen Sub-Type (z.B. sht31_temp) wurden in den humidity-Chart geschrieben.

### Implementierte Fixes

**HistoricalChart.vue Zeile 193-197** ā€” `raw_value` ā†’ `processed_value ?? raw_value`:
```typescript
// VORHER
const rawPoints = dataResponse.readings.map((d) => ({
  timestamp: new Date(d.timestamp),
  value: typeof d.raw_value === 'number' ? d.raw_value : parseFloat(String(d.raw_value)),
}))

// NACHHER
const rawPoints = dataResponse.readings.map((d) => {
  const val = d.processed_value != null ? d.processed_value : d.raw_value
  return {
    timestamp: new Date(d.timestamp),
    value: typeof val === 'number' ? val : parseFloat(String(val)),
  }
})
```

**HistoricalChart.vue Zeile 179-185** ā€” `sensor_type` Parameter hinzugefuegt:
```typescript
// VORHER
const dataResponse = await sensorsApi.queryData({
  esp_id: espId,
  gpio: Number(gpio),
  ...
})

// NACHHER
const dataResponse = await sensorsApi.queryData({
  esp_id: espId,
  gpio: Number(gpio),
  sensor_type: props.sensorType,
  ...
})
```

**HistoricalChart.vue Zeile 241-255** ā€” Live-Append sensor_type Filter:
```typescript
// VORHER
(s) => s.esp_id === espId && s.gpio === Number(gpio)

// NACHHER
(s) => s.esp_id === espId && s.gpio === Number(gpio)
  && (!props.sensorType || s.sensor_type === props.sensorType)
```

## Bug 2 (MEDIUM): Schwellwert-Linien bei Sensor-Hardware-Range

### Root Cause

**Datei:** `El Frontend/src/components/dashboard-widgets/WidgetConfigPanel.vue` Zeile 119-135

`handleSensorChange()` auto-populierte Thresholds aus `SENSOR_TYPE_CONFIG.min/max` ā€” das sind die **Hardware-Grenzen** des Sensors (z.B. SHT31 Temperatur: -40Ā°C bis 125Ā°C), NICHT sinnvolle Betriebs-Schwellwerte. Die berechneten Werte:
- `warnLow = -40 + (125-(-40)) * 0.1 = -23.5Ā°C`
- `warnHigh = 125 - (125-(-40)) * 0.1 = 108.5Ā°C`
- `alarmLow = -40Ā°C`, `alarmHigh = 125Ā°C`

Zusaetzlich: `generateZoneDashboard()` in `dashboard.store.ts` setzte `showThresholds: true` fuer auto-generierte Widgets.

### Implementierte Fixes

**WidgetConfigPanel.vue Zeile 119-135** ā€” Threshold-Auto-Population komplett entfernt:
```typescript
// VORHER
if (cfg) {
  if (localConfig.value.yMin == null && localConfig.value.yMax == null) { ... }
  const range = cfg.max - cfg.min
  if (!localConfig.value.warnLow) updates.warnLow = cfg.min + range * 0.1
  if (!localConfig.value.warnHigh) updates.warnHigh = cfg.max - range * 0.1
  if (!localConfig.value.alarmLow) updates.alarmLow = cfg.min
  if (!localConfig.value.alarmHigh) updates.alarmHigh = cfg.max
}

// NACHHER
if (cfg) {
  if (localConfig.value.yMin == null && localConfig.value.yMax == null) {
    updates.yMin = cfg.min
    updates.yMax = cfg.max
  }
  // Thresholds are NOT auto-populated ā€” sensor range (e.g. -40..125Ā°C) is not a
  // meaningful threshold. Users must configure thresholds explicitly via the panel.
}
```

**dashboard.store.ts Zeile 624 + 646** ā€” `showThresholds: false` fuer auto-generierte Widgets:
```typescript
// VORHER (line-chart + gauge in generateZoneDashboard)
showThresholds: true

// NACHHER
showThresholds: false
```

## Bug 4 (MEDIUM): Template "Sensor-Detail" erzeugt leeres Dashboard

### Root Cause

**Datei:** `El Frontend/src/shared/stores/dashboard.store.ts` Zeile 516-536

`createLayoutFromTemplate()` erzeugte Widgets OHNE `sensorId`. Da Sensor-Widgets ohne `sensorId` nichts rendern (kein Sensor ā†’ kein Daten ā†’ leerer Canvas), erschien das gesamte Dashboard leer.

### Implementierter Fix

`createLayoutFromTemplate()` ermittelt jetzt den **ersten verfuegbaren Sensor** aus `espStore.devices` und setzt `sensorId` als Default fuer alle Sensor-Widget-Typen (line-chart, gauge, sensor-card, historical):

```typescript
// NEU: Auto-populate sensorId from first available sensor
const espStore = useEspStore()
let defaultSensorId: string | undefined
for (const device of espStore.devices) {
  const deviceId = espStore.getDeviceId(device)
  const sensors = (device.sensors || []) as { gpio: number; sensor_type: string }[]
  if (sensors.length > 0) {
    const s = sensors[0]
    defaultSensorId = `${deviceId}:${s.gpio}:${s.sensor_type}`
    break
  }
}

const sensorWidgetTypes = new Set(['line-chart', 'gauge', 'sensor-card', 'historical'])

const widgets = template.widgets.map((tw, idx) => ({
  ...tw,
  id: `w-${Date.now()}-${idx}`,
  config: {
    ...(tw.config || {}),
    ...(sensorWidgetTypes.has(tw.type) && defaultSensorId ? { sensorId: defaultSensorId } : {}),
  },
}))
```

## Bug 5 (MEDIUM): Dashboard-Duplikate im Selector

### Root Cause

**Datei:** `El Frontend/src/shared/stores/dashboard.store.ts`

1. `generateZoneDashboard()` hatte bereits einen Update-Pfad (existierendes Dashboard updaten statt neu anlegen) ā€” dieses Problem betraf hauptsaechlich Templates.
2. `createLayoutFromTemplate()` prueft nicht auf existierende Dashboards mit gleichem Namen.
3. `fetchLayouts()` hatte keine Dedup-Logik ā€” Server-Daten und lokale Daten konnten Duplikate erzeugen.

### Implementierter Fix

**dashboard.store.ts nach Zeile 394** ā€” Dedup-Migration in `fetchLayouts()`:

```typescript
// Deduplicate dashboards by name (keep newest per name)
const nameMap = new Map<string, { idx: number; updatedAt: string }[]>()
for (let i = 0; i < layouts.value.length; i++) {
  const l = layouts.value[i]
  const key = (l.name || '').toLowerCase().trim()
  if (!key) continue
  const entries = nameMap.get(key) || []
  entries.push({ idx: i, updatedAt: l.updatedAt || l.createdAt || '' })
  nameMap.set(key, entries)
}

const indicesToRemove = new Set<number>()
for (const [, entries] of nameMap) {
  if (entries.length <= 1) continue
  entries.sort((a, b) => b.updatedAt.localeCompare(a.updatedAt))
  for (let i = 1; i < entries.length; i++) {
    indicesToRemove.add(entries[i].idx)
    const dup = layouts.value[entries[i].idx]
    if (dup.serverId) deleteLayoutFromServer(dup.serverId)
  }
}

if (indicesToRemove.size > 0) {
  layouts.value = layouts.value.filter((_, idx) => !indicesToRemove.has(idx))
  persistLayouts()
  logger.info(`[Dashboard] Removed ${indicesToRemove.size} duplicate dashboard(s)`)
}
```

## Unerwartete Findings

### Finding 1: Live-Append ohne sensor_type Filter
**HistoricalChart.vue** hatte einen Watch auf `espStore.latestReadings` der neue Datenpunkte live an den Chart anhaengte. Dieser Watch filterte NUR nach `esp_id` + `gpio`, nicht nach `sensor_type`. Bei Multi-Value-Sensoren (SHT31: temp + humidity auf GPIO 0) wurden dadurch Cross-Type-Updates in den falschen Chart geschrieben. Behoben als Teil von Bug 1.

### Finding 2: LineChartWidget Threshold-Guard bereits korrekt
`LineChartWidget.vue` hatte bereits eine korrekte Guard-Logik: Thresholds werden NUR an LiveLineChart weitergegeben wenn `config.showThresholds` explizit `true` ist UND die Werte `!= null` sind. Das Problem war upstream (WidgetConfigPanel auto-populate + generateZoneDashboard `showThresholds: true`).

### Finding 3: GaugeWidget bereits auf 3-Part sensorId
`GaugeWidget.vue` verwendet bereits das 3-Part sensorId-Format (`espId:gpio:sensorType`) korrekt und nutzt SENSOR_TYPE_CONFIG fuer Y-Range. Kein Fix noetig.
