## 3. Fehlerhafte Lookup-Logik: Warum der Crash auftritt

### Fehlerkette

Der `SensorDataHandler` in `sensor_handler.py` Zeile 219 fuhrt bei eingehenden Sensor-Daten einen 3-Wege-Lookup durch:

```python
# Zeile 216-221: Standard Sensor (kein onewire_address im Payload)
sensor_config = await sensor_repo.get_by_esp_gpio_and_type(
    esp_device.id, gpio, sensor_type
)
```

Die Methode `get_by_esp_gpio_and_type` (sensor_repo.py Zeile 101-107) verwendet `scalar_one_or_none()`:

```python
stmt = select(SensorConfig).where(
    SensorConfig.esp_id == esp_id,
    SensorConfig.gpio == gpio,
    SensorConfig.sensor_type == sensor_type,  # Case-sensitive!
)
result = await self.session.execute(stmt)
return result.scalar_one_or_none()  # CRASH wenn > 1 Zeile
```

### Warum das bei DS18B20 NICHT crasht (aber der HTTP-Endpoint schon)

Der 3-Wege-Lookup wird nur verwendet wenn **kein** `onewire_address` im MQTT-Payload. Wenn der Wokwi-ESP eine `onewire_address` sendet, wird stattdessen `get_by_esp_gpio_type_and_onewire` (4-Wege) aufgerufen. Das war korrekt.

**Aber:** Der HTTP-Endpoint `GET /api/v1/sensors/{esp_id}/{gpio}` (sensors.py Zeile 435) ruft direkt `get_by_esp_gpio_and_type` auf und crashed bei 2+ Zeilen auf demselben GPIO.

## 4. Zweites Problem: Case-Sensitivity sensor_type

Die DB zeigt:
- Eintrag 1: `sensor_type = 'DS18B20'` (Grossbuchstaben, alter Eintrag)
- Eintrag 2: `sensor_type = 'ds18b20'` (Kleinbuchstaben, neuer Eintrag)

Der SQL-Lookup in `get_by_esp_gpio_and_type` ist **case-sensitiv** (kein `func.lower()`). Je nach Case im MQTT-Payload matcht mal ein Eintrag, mal der andere, mal beide (wenn PostgreSQL case-insensitiv collation).

In `query_paginated` (sensor_repo.py Zeile 190) wird `func.lower()` verwendet. In `get_by_esp_gpio_and_type` nicht.

## 7. Diagnose: Wo genau schlaegt die DS18B20-Konfiguration fehl?

### Fehler #1 (KRITISCH): MQTT sensor_handler crash bei eingehenden Sensor-Daten

**Ursache:** `get_by_esp_gpio_and_type` wird mit `sensor_type='ds18b20'` aufgerufen (lowercase aus MQTT-Payload). Die Query matcht BEIDE Eintraege wenn PostgreSQL case-insensitive ist, oder keinen / einen wenn case-sensitive. Mit `scalar_one_or_none()` crasht es sobald > 1 Zeile gefunden wird.

**Betroffene Datei:** `El Servador/god_kaiser_server/src/db/repositories/sensor_repo.py`, Zeile 107

**Betroffener Pfad:** Nur wenn kein `onewire_address` im MQTT-Payload. Falls der ESP den ROM-Code sendet, wird der 4-Wege-Lookup verwendet (korrekt). Die Frage ist ob der Wokwi-ESP den `onewire_address`-Key im Sensor-Daten-Payload mit ubertragt.

### Fehler #2 (HOCH): HTTP 500 bei GET /api/v1/sensors/{esp_id}/{gpio}

**Ursache:** Gleicher `scalar_one_or_none()` Crash im HTTP-Endpoint wenn Frontend Sensor-Details abruft.

**Betroffene Datei:** `El Servador/god_kaiser_server/src/api/v1/sensors.py`, Zeile 435

### Fehler #3 (HOCH): sensor_type Case-Inkonsistenz

- Erster Eintrag wurde mit `sensor_type='DS18B20'` erstellt (Uppercase, wahrscheinlich vom MQTT-Payload des alten Wokwi-ESP)
- Zweiter Eintrag wurde mit `sensor_type='ds18b20'` erstellt (Lowercase, neuer Eintrag)
- `get_by_esp_gpio_and_type` normalisiert den `sensor_type` NICHT (kein `func.lower()`)

### Fehler #4 (MEDIUM): Wokwi DS18B20 sendet keinen onewire_address im Sensor-Daten-Payload (Vermutung)

Wenn der Wokwi-ESP DS18B20-Daten ohne `onewire_address` sendet, greift der 3-Wege-Fallback, der mit `scalar_one_or_none()` crasht. Der 4-Wege-Lookup (mit onewire_address) wuerde korrekt funktionieren.

**Beleg aus MQTT-Traffic:** `ESP_00000001` hat den Scan korrekt durchgefuhrt und ROM `280102030405069E` gefunden, aber im kurzen MQTT-Beobachtungsfenster wurden keine Sensor-Daten-Messages empfangen.

## 8. Warum das Wokwi-Setup (ESP_00000001) selbst korrekt ist

Das Serial-Log zeigt einen erfolgreichen Flow:
1. OneWire-Scan triggert → Server sendet `onewire/scan` Command
2. ESP antwortet mit `scan_result` (ROM `280102030405069E`, 1 Sensor)
3. Zone-Assignment: OK (`echter_esp`)
4. Subzone-Assignment: OK (`naehrloesung_zelt`, GPIO 4)
5. NVS-Fehler (`sz_idx_map NOT_FOUND`, `subzone_ids NOT_FOUND`): **Harmlos**, erste Schreiboperation

Der eigentliche Fehler liegt **nicht im Wokwi-ESP-Setup** sondern im Server: Die DB enthalt einen Duplikat-Eintrag fuer `MOCK_A3592B7E` auf GPIO 4.

**Die DS18B20-Konfiguration fuer ESP_00000001 hat noch nicht stattgefunden** (sensor_configs zeigen keine Eintraege fur ESP_00000001). Das Frontend-Konfigurationsformular wurde vermutlich versucht abzuschicken aber kam durch HTTP 500 nicht durch, oder der Konfigurations-Schritt stand noch aus.
