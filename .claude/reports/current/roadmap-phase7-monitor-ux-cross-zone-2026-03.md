## Kern-Problem: Visuelles Chaos durch 4 fragmentierte Ansichten

Der User sieht aktuell 4 verschiedene Stellen die alle "Dashboards/Widgets" zeigen, aber keine davon tut es richtig:

1. **Monitor L1** — Zone-Tiles oben, darunter ein losgeloestes Dashboard (z.B. "T18-Test Dashboard") das keiner Zone gehoert. Zwei verschiedene Informationen uebereinander gestapelt ohne Zusammenhang.
2. **Editor Ansichtsmodus** — ~25 Dashboards in endloser Liste. Grossteils Auto-Muell ("QA Basiszonen", "Sensordaten digital", "CSS widget", "Testone digital"). Niemand braucht sie, niemand loescht sie.
3. **Monitor Dashboard-Ansicht** — "Mock Zone Dashboard, 14 Widgets, automatisch erstellt". Drei Buttons (Uebernehmen/Anpassen/Im Editor bearbeiten) die keiner versteht. Unklar ob man im Monitor, Editor oder einer dritten View ist.
4. **FAB/Action Ball** — Zeigt Widget-Typen (Gauge, Line Chart etc.) aber kann sie nirgends sinnvoll platzieren. Ungenutztes Potenzial.

**Der Editor Bearbeitungsmodus** ist das einzige das funktioniert: Widget-Katalog links mit 9 Typen, GridStack-Layout rechts. Aber man braucht 3 Klicks um dorthin zu kommen.

## Leitprinzipien (aus IoT Dashboard Best Practices)

Diese Regeln gelten fuer ALLE Auftraege in Phase 7:

1. **5-Sekunden-Regel:** L1 beantwortet in 5 Sekunden "Ist alles in Ordnung?" — Farbe zeigt Status, Zone-Tiles zeigen Uebersicht. Kein Rauschen.
2. **Einstellungen gehoeren zum Objekt:** Sensor-Schwellwerte ins Sensor-Config-Panel (HardwareView), Widget-Konfiguration ins Widget-Panel (Editor). Nicht verteilen.
3. **Monitor = Read-Only.** Kein Konfigurieren im Monitor. Ausnahme: Widgets loeschen/hinzufuegen (D3/D4) — das ist Personalisierung, nicht Konfiguration.
4. **Progressive Offenlegung:** L1 = Zusammenfassung, L2 = Detail. Details erst auf Nachfrage (Klick).
5. **Layout-Stabilitaet:** Kein Element springt. Feste Zeilen-Reihenfolge innerhalb jeder Subzone (Geraete → Regeln → Charts). Feste Hoehen wo noetig.
6. **Hierarchisches Drill-Down:** Zone → Subzone → Sensor/Aktor. Dieser Pfad ist Standard in IoT-Topologie-Views (AWS IoT Fleet Hub, ThingsBoard, SCADA ISA-101).
7. **Bestehende Tokens nutzen:** `--color-*`, `--glass-*`, `--space-*`, `--elevation-*` (129 Tokens). Kein `--ao-*` Prefix. Sub-10px font-sizes fuer Badges sind bewusst.

## Store-Vorbedingungen (betrifft D3 + D4)

D3 und D4 setzen Store-Actions voraus die aktuell NICHT existieren. Vor oder als Teil von D3 muessen folgende Actions in `shared/stores/dashboard.store.ts` implementiert werden:

| Action | Signatur | Nutzt intern |
|--------|----------|--------------|
| `addWidget(layoutId, widgetConfig)` | `(layoutId: string, config: WidgetConfig) → void` | `saveLayout(layoutId, [...widgets, newWidget])` |
| `removeWidget(layoutId, widgetId)` | `(layoutId: string, widgetId: string) → void` | `saveLayout(layoutId, widgets.filter(w => w.id !== widgetId))` |
| `updateWidgetConfig(layoutId, widgetId, newConfig)` | `(layoutId: string, widgetId: string, config: Partial<WidgetConfig>) → void` | `saveLayout(layoutId, widgets.map(...))` |

Alternativ: `createLayout()` Signatur erweitern um `options?: { scope, zoneId, autoGenerated }` — oder D3 nutzt `generateZoneDashboard()` direkt.

## D1 — Dashboard-Liste entmuellen (~2-3h)

Der User kann nur ueber Browser-DevTools oder API manuell aufaeumen.

### IST-Zustand

- `dashboard.store.ts` hat `deleteLayout(layoutId)` — loescht aus lokalem State + synced zum Server (`DELETE /api/v1/dashboards/{id}`)

### SOLL-Zustand

Klick oeffnet Bestaetigungsdialog: "Dashboard '{name}' loeschen? Diese Aktion kann nicht rueckgaengig gemacht werden." → Bestaetigen ruft `dashboardStore.deleteLayout(layoutId)`.

2. **Bulk-Action "Auto-generierte entfernen":** Button oberhalb der Dashboard-Liste: "Auto-generierte aufraeumen" (nur sichtbar wenn mindestens 1 Dashboard mit `autoGenerated: true` existiert). Klick zeigt Liste der betroffenen Dashboards mit Checkboxen (alle vorausgewaehlt). User kann einzelne abwaehlen. Bestaetigen loescht alle ausgewaehlten via `deleteLayout()` in einer Schleife.

3. **Leerer State:** Wenn nach dem Loeschen 0 Dashboards uebrig: Freundlicher Hinweis "Kein Dashboard vorhanden. Erstelle ein neues Dashboard fuer deine Zonen." mit prominentem "Neues Dashboard" Button.

### Betroffene Dateien

- `CustomDashboardView.vue` — Dashboard-Liste Template erweitern (Trash-Icon, Bulk-Button)
- `dashboard.store.ts` — `deleteLayout()` existiert bereits, ggf. `bulkDeleteAutoGenerated()` Action hinzufuegen
- Ggf. neues Modal/Dialog fuer Bulk-Delete Bestaetigung (nutze bestehende Dialog-Komponente falls vorhanden)

### Einschraenkungen

- KEIN "Archivieren" — Loeschen ist endgueltig (Dashboard kann neu generiert werden)
- KEIN automatisches Loeschen beim App-Start — User entscheidet bewusst
- KEINE Aenderung am generateZoneDashboard()-Mechanismus — der bleibt wie er ist
- Bestehende Dashboards die der User manuell bearbeitet hat (`autoGenerated: false` nach `claimAutoLayout()`) werden NICHT in der Bulk-Action angeboten

### Akzeptanzkriterien

1. Jede Dashboard-Card hat einen funktionalen Loeschen-Button
2. Bestaetigungsdialog vor jedem Loeschen
3. "Auto-generierte aufraeumen" Button funktioniert, zeigt Vorschau, loescht nur ausgewaehlte
4. Nach Loeschen: Dashboard-Liste aktualisiert sich sofort (reactive)

## D3 — FAB nutzbar machen: Quick-Add Widget Dialog (~3-4h)

### Kontext

Der FAB (Floating Action Button / Action Ball) zeigt Widget-Typen als Icons (Gauge, Line Chart, Historical, Sensor Card etc.). Aktuell kann man sie aber nirgends sinnvoll platzieren — der FAB lebt im Editor-Kontext und hat keine Verbindung zum Monitor. Das Potenzial ist: User klickt auf "Gauge" → einfacher Dialog fragt Zone + Sensor → Widget erscheint in der Zone-Kachel auf L1 oder auf L2.

Die Zone-gruppierte Sensor-Auswahl (`useSensorOptions` mit `groupedSensorOptions`) existiert bereits seit Editor Phase A (PA-02b). Das Dashboard→Widget Zone-Kontext-System (`zoneId` Propagation) existiert seit PA-02c. Die Mini-Widget-Integration in ZoneTileCard existiert seit A1 Phase 3. Alle Bausteine sind da.

### IST-Zustand

- FAB besteht aus 3 Komponenten: `components/quick-action/QuickActionBall.vue`, `QuickActionMenu.vue`, `QuickWidgetPanel.vue` + Composable `composables/useWidgetDragFromFab.ts`
- FAB zeigt 9 Widget-Typen als klickbare Icons
- Klick auf Icon in CustomDashboardView erstellt Widget im GridStack-Layout
- FAB ist aktuell NUR in `CustomDashboardView.vue` eingebunden — NICHT im Monitor
- `useSensorOptions(filterZoneId)` liefert gruppierte Sensor-Listen pro Zone
- `dashboard.store.ts` hat **KEIN** `addWidget()` — Widgets werden via `saveLayout(layoutId, widgets)` als ganzes Array gespeichert. `createLayout(name, description?)` nimmt nur name + description, KEIN Objekt mit scope/zoneId
- `generateZoneDashboard()` existiert und erstellt Zone-Dashboards mit zoneId + autoGenerated: true
- Zone-Dashboards existieren mit `scope: 'zone'` und `zoneId`

### SOLL-Zustand

**Neuer Dialog: "Widget hinzufuegen"** — erreichbar ueber den FAB. Der Dialog fuehrt den User in 3 Schritten:

```
┌─── Widget hinzufuegen ─────────────────────────────┐
│                                                     │
│  1. Widget-Typ waehlen:                             │
│     [Gauge] [Line Chart] [Sensor Card] [...]        │
│                                                     │
│  2. Zone waehlen:                                   │
│     [Dropdown: Zelt Wohnzimmer ▼]                   │
│                                                     │
│  3. Sensor waehlen:                                 │
│     [Grouped Dropdown via useSensorOptions]          │
│       Pflanze 1                                     │
│         SHT31 Temperatur                            │
│         SHT31 Feuchte                               │
│       Pflanze 2                                     │
│         SHT31 Temperatur                            │
│                                                     │
│                    [Abbrechen]  [Hinzufuegen]       │
└─────────────────────────────────────────────────────┘
```

**Verhalten nach "Hinzufuegen":**
- Widget wird zum Zone-Dashboard hinzugefuegt (Dashboard mit `scope: 'zone'` + `zoneId`)
- Wenn fuer die Zone noch kein Dashboard existiert: automatisch eins erstellen via `generateZoneDashboard()` oder `createLayout({ scope: 'zone', zoneId, name: '{zoneName} Dashboard' })`
- Widget erscheint sofort im Mini-Widget-Bereich der ZoneTileCard (extra-Slot, falls `TILE_ALLOWED_WIDGET_TYPES`)
- Bei Widget-Typen die NICHT in TILE_ALLOWED_WIDGET_TYPES sind (z.B. historical, multi-sensor): Widget wird trotzdem zum Zone-Dashboard hinzugefuegt, ist aber nur auf L2 oder im Editor sichtbar. Hinweis im Dialog: "Dieses Widget ist auf L2 und im Editor sichtbar."

### Technischer Ansatz

1. **Neue Komponente: `AddWidgetDialog.vue`** (in `components/monitor/` oder `components/dashboard/`)
   - Props: `open: boolean`, `defaultZoneId?: string` (fuer Kontextaufruf von L2)
   - Emits: `close`, `added(widgetConfig)`
   - Intern: 3-Step-Form (Widget-Typ → Zone → Sensor)
   - Widget-Typ-Auswahl: Nutze `WIDGET_TYPE_META` Array aus Widget-Registrierung fuer Icon + Label
   - Zone-Auswahl: Dropdown aus `zoneStore.allZones` oder `useZoneKPIs().allZones`
   - Sensor-Auswahl: `useSensorOptions({ filterZoneId: selectedZoneId })` → `<select>` mit `<optgroup>` (wie WidgetConfigPanel)

2. **QuickActionBall im Monitor einbinden:** `QuickActionBall.vue` (+ QuickActionMenu, QuickWidgetPanel) auch in MonitorView.vue einbinden (unten rechts, floating). Aktuell NUR in CustomDashboardView — muss nach MonitorView dupliziert werden. Nur sichtbar wenn User eingeloggt ist (ActiveUser Auth-Tier).

3. **FAB-Verhalten im Monitor aendern:** Im Monitor-Kontext oeffnet Klick auf Widget-Typ den `AddWidgetDialog` (statt direkt GridStack-Insert). Im Editor-Kontext: bestehendes Verhalten beibehalten (direkter GridStack-Insert). Unterscheidung ueber Prop oder Context.

4. **Dashboard-Erstellung + Widget-Hinzufuegen:** Wenn Zone kein Dashboard hat:
   ```typescript
   // Option A: generateZoneDashboard() nutzen (hat zoneId + setzt autoGenerated: true)
   const layout = await dashboardStore.generateZoneDashboard(selectedZoneId, zoneName)

   // Option B: createLayout() erweitern (Signatur-Aenderung noetig!)
   // Aktuell: createLayout(name: string, description?: string): DashboardLayout
   // Gebraucht: createLayout(name, { scope: 'zone', zoneId, autoGenerated: true })
   // → Signatur muss im Auftrag als Aenderung definiert werden
   ```
   Widget zum Layout hinzufuegen:
   ```typescript
   // ACHTUNG: addWidget() existiert NICHT im Store!
   // Entweder: Neue Action addWidget() als Teil dieses Auftrags definieren
   // Oder: Bestehendes saveLayout() Pattern nutzen:
   const widgets = [...layout.widgets, {
     type: selectedWidgetType,
     config: { sensorId: selectedSensorId },
     x: 0, y: 0,
     w: WIDGET_DEFAULT_CONFIGS[selectedWidgetType].w,
     h: WIDGET_DEFAULT_CONFIGS[selectedWidgetType].h
   }]
   await dashboardStore.saveLayout(layout.id, widgets)
   ```

   **Vorbedingung fuer D3:** Entweder `addWidget()` Action im Store implementieren (empfohlen — kapselt Widget-Array-Mutation + Server-Sync), oder das saveLayout()-Pattern direkt nutzen.

### Einschraenkungen

- KEIN Drag-and-Drop — Dialog reicht fuer 90% der Faelle (DnD ist D5)
- KEIN Multi-Sensor-Dialog — nur ein Sensor pro Schritt. Fuer Multi-Sensor: im Editor
- KEIN Widget-Positioning — Widget wird automatisch an naechste freie Position gesetzt
- KEINE Threshold-Konfiguration im Dialog — das passiert im Widget-Config-Panel (Editor)
- FAB im Monitor ist NUR fuer "Widget hinzufuegen" — kein Dashboard-Erstellen, kein Layout-Bearbeiten

### Akzeptanzkriterien

1. FAB ist im Monitor sichtbar (unten rechts, floating)
2. Klick auf FAB Widget-Typ oeffnet AddWidgetDialog
3. Dialog zeigt Widget-Typ → Zone → Sensor Auswahl
4. Sensor-Dropdown nutzt `useSensorOptions` mit Zone-Gruppierung und `filterZoneId`
5. Nach "Hinzufuegen": Widget erscheint im Zone-Dashboard
6. Fuer gauge/sensor-card: Widget erscheint sofort in ZoneTileCard Mini-Widget-Bereich
7. Zone-Dashboard wird automatisch erstellt wenn keins existiert
8. `vue-tsc` und `vite build` fehlerfrei

## D4 — Widget-Inline-Verwaltung im Monitor (~2-3h)

### Kontext

### IST-Zustand

- Widgets werden via `InlineDashboardPanel` und `useDashboardWidgets` gerendert
- `InlineDashboardPanel` hat `mode` Prop (nicht readOnly Boolean). MonitorView nutzt `mode="inline"`. InlineDashboardPanel hat intern `readOnly: true` als Default.
- Im Monitor ist readOnly effektiv true via `mode="inline"` (6.3 implementiert)
- Widgets im Monitor haben keine Hover-Toolbar
- Loeschen/Konfigurieren nur ueber Editor moeglich

### SOLL-Zustand

**Hover-Toolbar auf Widgets im Monitor:**

```
┌─── Widget: Gauge pH ──────────────────────────────┐
│                                                     │
│            ◉ 6.24 pH                                │
│                                                     │
│  ┌─ Toolbar (bei Hover) ─────────────────────────┐ │
│  │  [⚙ Konfigurieren]  [🗑 Entfernen]            │ │
│  └────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────┘
```

**Verhalten:**

- **Hover:** Toolbar erscheint oben rechts im Widget (halbtransparent, `background: var(--glass-bg)`, `backdrop-filter: blur(8px)`)
- **[Konfigurieren]:** Oeffnet ein Panel/Modal mit dem bestehenden `WidgetConfigPanel` fuer dieses Widget. User kann sensorId, Zeitbereich, Schwellwerte aendern. Speichern synced zum Server.
- **[Entfernen]:** Bestaetigungsdialog: "Widget '{type}' entfernen?" → Bestaetigen ruft `dashboardStore.removeWidget(layoutId, widgetId)` + Server-Sync. Widget verschwindet sofort.
- **Toolbar NICHT sichtbar** wenn User nicht eingeloggt (kein Auth-Token) oder `readOnly: true` explizit gesetzt ist (z.B. Gaestezugang, spaeter).

### Technischer Ansatz

1. **Widget-Wrapper-Komponente erweitern:** Jede Widget-Instanz innerhalb von `useDashboardWidgets` wird in einem Container gerendert. Diesem Container eine Hover-Toolbar hinzufuegen.

   - Entweder in `InlineDashboardPanel.vue` Template (dort wo `v-for` ueber Widgets iteriert)
   - Oder in einer neuen `WidgetWrapper.vue` Komponente die jedes Widget umschliesst

2. **mode-System erweitern:** InlineDashboardPanel hat BEREITS einen `mode` Prop (aktuell: `'inline'`). Diesen zu einem 3-Werte-System erweitern:
   - `'view'` = Monitor ohne Toolbar (Gaeste, readonly) — wie bisheriges `'inline'`
   - `'manage'` = Monitor mit Hover-Toolbar (eingeloggte User) ← NEU
   - `'edit'` = Editor mit GridStack DnD (Editor-Modus)

   MonitorView setzt `mode="manage"` statt `mode="inline"`. Kein neuer Prop noetig — bestehenden `mode` Prop erweitern.

3. **WidgetConfigPanel als Modal:** Beim Klick auf "Konfigurieren" oeffnet sich ein Modal/Dialog mit dem bestehenden `WidgetConfigPanel` (die gleiche Komponente die im Editor rechts angezeigt wird). Props: `widgetConfig`, `zoneId`, `onSave(updatedConfig)`. Save: Widget im Array mutieren + `dashboardStore.saveLayout(layoutId, updatedWidgets)`.

4. **Entfernen:** Widget aus dem widgets-Array filtern + `dashboardStore.saveLayout(layoutId, filteredWidgets)` → Server-Sync.

   **Vorbedingung fuer D4:** Die Store-Methoden `addWidget()`, `removeWidget()`, `updateWidgetConfig()` existieren NICHT. Entweder als neue Actions im Store implementieren (empfohlen — kapselt Array-Mutation + saveLayout), oder direkt das widgets-Array manipulieren + saveLayout() aufrufen. Falls D3 bereits addWidget() implementiert hat, hier removeWidget() + updateWidgetConfig() analog ergaenzen.

### Einschraenkungen

- KEIN Drag-and-Drop im Monitor — Widgets bleiben an ihrer Position
- KEIN Resize im Monitor — nur im Editor
- KEIN "Widget verschieben" — nur im Editor
- WidgetConfigPanel wird 1:1 wiederverwendet — keine Monitor-spezifische Version
- Hover-Toolbar nur auf Desktop (Touch: Long-Press oder kleines Kebab-Menu ⋮)

### Akzeptanzkriterien

1. Hover auf Widget im Monitor zeigt Toolbar mit [Konfigurieren] und [Entfernen]
2. [Konfigurieren] oeffnet WidgetConfigPanel Modal mit aktueller Widget-Config
3. Aenderungen in WidgetConfigPanel werden gespeichert und sofort sichtbar
4. [Entfernen] mit Bestaetigungsdialog loescht Widget + Server-Sync
5. Keine Toolbar im Editor-Modus (dort hat GridStack eigene Controls)
6. Keine Toolbar ohne Auth-Token
7. ActuatorCardWidget Toggle bleibt ausgeblendet im Monitor (6.3 Regression-Check)
8. `vue-tsc` und `vite build` fehlerfrei

## D5 — Echtes Drag-and-Drop vom FAB (optional, ~4-6h)

### Kontext

D3 loest 90% der Faelle — der Dialog reicht. D5 ist fuer Power-User die Widget-Platzierung direkt steuern wollen. Der User klickt im FAB auf einen Widget-Typ, ein Drag-Ghost heftet sich an den Cursor, der User droppt auf eine Zone-Kachel (L1) oder eine Subzone (L2).

### Voraussetzungen

- D3 muss KOMPLETT sein (AddWidgetDialog als Fallback)
- D4 muss KOMPLETT sein (Widget-Verwaltung existiert)

### Technischer Ansatz

1. **FAB-Item Drag-Start:** `useWidgetDragFromFab.ts` existiert bereits als DnD-Composable — darauf aufbauen statt neu implementieren. Klick auf Widget-Icon im FAB startet Drag (HTML5 DnD oder Pointer Events). Ein Ghost-Element (Widget-Typ-Icon + Name, halbtransparent) folgt dem Cursor.

2. **Drop-Targets:**
   - **ZoneTileCard (L1):** Visueller Indicator bei Hover (Border-Highlight `var(--color-primary)`, `opacity: 0.1` Overlay). Drop erstellt Widget im Zone-Dashboard (wie D3, ohne Dialog — Widget-Typ + Zone sind durch Drop-Target bekannt, Sensor = erster verfuegbarer oder User waehlt danach).
   - **Subzone-Accordion (L2):** Drop erstellt Widget im Zone-Dashboard mit Subzone-Kontext.

3. **useZoneDragDrop Pattern:** AutomationOne hat bereits ein DnD-Pattern fuer Zone/Subzone-Zuweisungen (`useZoneDragDrop` oder aehnlich). Dieses Pattern wiederverwenden fuer Drop-Target-Registrierung und visuelles Feedback.

4. **Post-Drop:** Nach dem Drop oeffnet sich der AddWidgetDialog mit vorausgefuellter Zone + Widget-Typ. User waehlt nur noch den Sensor. Das verbindet D5 (DnD) mit D3 (Dialog) — DnD ist eine ABKUERZUNG, nicht ein komplett anderer Flow.

### Einschraenkungen

- KEIN Drag innerhalb des Monitors (Widget-Repositionierung = Editor only)
- KEIN Drag auf beliebige Stellen — nur auf registrierte Drop-Targets (Zone-Tiles, Subzones)
- KEIN mobiles DnD — Dialog (D3) ist der Mobile-Pfad
- Falls DnD-Implementierung zu komplex: D3 reicht. D5 ist Nice-to-have.

### Akzeptanzkriterien

1. FAB-Items sind draggable
2. Ghost-Element folgt dem Cursor mit Widget-Typ-Icon
3. ZoneTileCard und Subzone-Accordions sind Drop-Targets mit visuellem Feedback
4. Drop oeffnet AddWidgetDialog mit vorausgefuelltem Widget-Typ + Zone
5. Kein DnD auf Mobile (Fallback: D3 Dialog)
6. `vue-tsc` und `vite build` fehlerfrei

## D6 — Cross-Zone + Mobile Sensoren (spaeter, braucht Backend)

### Kontext

`device_scope` ('zone_local' | 'multi_zone' | 'mobile') existiert im Backend. `device_active_context` Tabelle existiert. 3 API-Endpoints existieren (`GET/PUT/DELETE /api/v1/device-context/{config_type}/{config_id}`). WS-Event `device_context_changed` wird bereits empfangen. Das Frontend ignoriert all das.

### Abhaengigkeit

- **Backend-Verifikation (B1 Analyse):** Auftrag `analyse-B1-backend-api-response-audit-2026-03-25.md` muss zuerst ausgefuehrt werden. Klaert: Liefern die Sensor/Zone-Endpoints `device_scope`, `assigned_zones`, `active_zone_id`? Ohne diese Antwort kann D6 nicht praezise formuliert werden.
- **Entspricht weitgehend Auftrag 6.7** (`auftrag-6-7-cross-zone-mobile-2026-03-25.md`), aber mit Analyse-Unterbau.

### Was D6 bringt

1. **device_scope Badges:** SensorCard und ActuatorCard zeigen Badge "Mobile" oder "Shared" wenn `device_scope !== 'zone_local'`
2. **Active Zone Counting:** `computeZoneKPIs()` in `useZoneKPIs.ts` beruecksichtigt `device_active_context.active_zone_id` bei mobilen Sensoren
3. **Shared Device Reference Cards:** In bedienten Zonen erscheint eine kompakte Referenz-Card fuer Shared Sensors: "EC: 1.8 mS/cm (via Technikzone)" — kein volles Device, nur KPI-Wert + Link zur Heimzone
4. **Config-Panel Erweiterung:** SensorConfigPanel und ActuatorConfigPanel in HardwareView erhalten `device_scope` Dropdown + `assigned_zones` Multi-Select

### Nicht jetzt formulierbar

D6 wird NACH der B1-Analyse als eigener detaillierter Auftrag geschrieben. Die Analyse klaert welche Backend-Luecken existieren und ob neue Endpoints noetig sind.
