## Polishing-Anforderungen Ebene 1

### P1.1: Zone-Cards Design und Informationsdichte

**Problem:** Zone-Cards zeigen aktuell nur Basis-KPIs (Temp, Humidity, Sensor/Aktor-Counts). Fuer die 5-Sekunden-Regel ("Ist alles OK?") fehlt kontextueller Status.

**Anforderung — autoops:debug Analyse:**
1. Zone-Cards via Playwright screenshotten und visuell analysieren
2. IST-Zustand dokumentieren: Welche Felder werden tatsaechlich angezeigt? Welche Werte kommen durch?
3. Pruefen ob `avgTemperature`/`avgHumidity` korrekt berechnet werden (nur von aktiven Sensoren?)
4. Pruefen ob `alarmCount` korrekt zaehlt (basierend auf welchem Schwellwert?)

**Anforderung — Polishing:**
- **Status-Ampel pro Zone:** Gruen (OK) / Gelb (Warnung) / Rot (Alarm) — basierend auf `alarmCount` und Device-Connectivity
  - Gruen: `alarmCount === 0 && activeSensors === sensorCount`
  - Gelb: `alarmCount > 0 || activeSensors < sensorCount` (mindestens 1 Warnung ODER offline Sensor)
  - Rot: `activeSensors === 0` (Zone komplett offline)
  - **Doppelte Kodierung (Pflicht):** Farbe + Text + Icon — nie Farbe allein. Beispiel: "✅ Alles OK", "⚠ 1 Alarm", "❌ Offline"
- **Kern-Sensorwerte:** Werden via `zone.aggregation.sensorTypes`-Array gerendert (NICHT als direkte avgTemperature/avgHumidity Felder). Aktuell: `formatAggregatedValue(st, zone.aggregation.deviceCount)` pro sensorType.
  - Statt "0.0°C" → Pruefen ob Sensor tatsaechlich 0.0 meldet oder ob der Wert nicht empfangen wird (Stale-Check)
  - Format: "23.5°C" mit kleinem Trend-Indikator (↑ steigend, → stabil, ↓ fallend) wenn Sparkline-Daten vorhanden
- **Sensor/Aktor-Status:** Aktive vs. Gesamt als Ratio mit visueller Indikation
  - "3/3 Sensoren" (gruen) vs. "2/3 Sensoren" (gelb) vs. "0/3 Sensoren" (rot)
- **Letzte Aktivitaet:** Timestamp des letzten empfangenen Sensor-Events pro Zone
  - Stale-Indikator wenn >60s kein Event (konfigurierbar ueber Design-Token oder Konstante)

**UX-Prinzipien angewandt:**
| Prinzip | Anwendung auf Zone-Cards |
|---------|--------------------------|
| 5-Sekunden-Regel | Status-Ampel beantwortet sofort "Muss ich handeln?" |
| Doppelte Kodierung | Farbe + Text + Icon fuer jeden Status |
| Kontext statt Zahlen | "23.5°C (optimal)" statt nackte Zahl, "2/3 online" statt nur "2" |
| SPOG-Formel | Nur die wichtigsten KPIs auf L1, Details erst auf L2 |

### P1.2: Cross-Zone-Dashboard-Links

**IST-Zustand (aus Konsolidierung):** `crossZoneDashboards` Computed-Property und LinkCards existieren als Implementierung.

**Anforderung — autoops:debug Analyse:**
1. Pruefen ob Cross-Zone-Dashboard-Sektion tatsaechlich gerendert wird
2. Pruefen ob sie korrekt NUR angezeigt wird wenn User Cross-Zone-Dashboards erstellt hat (leerer State)
3. LinkCard-Design und Klick-Verhalten testen

**Anforderung — Polishing:**
- **Empty State:** ~~Wenn keine Cross-Zone-Dashboards existieren → Sektion komplett ausblenden~~ **[BEREITS IMPLEMENTIERT]** — `v-if="dashStore.crossZoneDashboards.length > 0"` blendet die Sektion bereits komplett aus
- **LinkCard-Design:** Dashboard-Name, Widget-Count, letzte Bearbeitung (aus `updatedAt`) — **[Hinweis]** Aktuelles Template zeigt nur Name + Widget-Count, `updatedAt` wird noch NICHT angezeigt
- **Klick-Verhalten:** LinkCard nutzt BEREITS `{ path: '/editor', query: { layout: dash.id } }` — Query-Parameter heißt `layout` (nicht `dashboard`). Route `/editor/:dashboardId` (name: `editor-dashboard`) existiert ebenfalls als Alternative.
  - **Abhaengigkeit:** Edit/View-Mode-Trennung aus `auftrag-dashboard-editor-polishing.md` Punkt B
  - Aktuelle Implementierung: `router-link :to="{ path: '/editor', query: { layout: dash.id } }"`
- **Reihenfolge:** Cross-Zone-Dashboards NACH den Zone-Cards anzeigen
- **Max-Anzeige:** Bei >4 Cross-Zone-Dashboards → "Alle anzeigen" Link (navigiert zum Editor mit Filter)

**Anforderung:**

- **Referenz:** Die Logic-Rules-Integration wird in `auftrag-logic-rules-live-monitoring-integration.md` folgende Elemente auf L1 platzieren:

### P1.4: Seitentitel und Header

**Problem:** Aktueller Header "Sensor & Aktor Monitoring" mit Subtitle "Echtzeit-Daten aller Zonen" ist generisch.

**Anforderung — autoops:debug Analyse:**
1. Aktuellen Header-Text und Styling dokumentieren
2. Pruefen ob der Header responsive ist (Mobile, Tablet, Desktop)
3. Pruefen ob Breadcrumbs auf L1 korrekt als "Monitor" angezeigt werden

**Anforderung — Polishing:**
- **Header:** "Monitor" (kurz, praegnant) — der Tab-Name IST der Seitenname
- **Subtitle:** System-Status-Zusammenfassung: "{X} Zonen · {Y}/{Z} Sensoren online · {N} Alarme"
  - Dynamisch berechnet aus allen Zone-KPIs
  - Bei Alarmen: Subtitle-Text in Warn-Farbe
- **Kein separater Page-Header wenn ViewTabBar bereits "Monitor" als aktiven Tab zeigt** — Redundanz vermeiden
- **Design-Token Konformitaet:** Pruefe ob `El Frontend/src/styles/tokens.css` Glassmorphism-Variablen korrekt angewandt werden

### P1.5: Responsive Layout

**Anforderung — autoops:debug Analyse:**
1. Playwright-Tests bei 3 Breakpoints: Mobile (375px), Tablet (768px), Desktop (1280px)
2. Zone-Cards: Grid-Verhalten bei verschiedenen Bildschirmgroessen
3. Cross-Zone-Dashboard-Links: Verhalten bei wenig Platz
4. ViewTabBar: Labels vs. Icons-Only unter 480px (bereits implementiert)

**Anforderung — Polishing:**
- **Zone-Cards Grid:** `grid-cols-1 sm:grid-cols-2 lg:grid-cols-3` (oder entsprechend dem bestehenden Grid-System)
- **Mobile:** Zone-Cards als volle Breite, Status-Ampel prominent oben
- **Overflow:** Bei >6 Zonen → Scrollbar oder "Alle Zonen" Erweiterung (NICHT unendlich lange Seite)

### P1.6: Performance-Check

**Anforderung — autoops:debug Analyse:**
1. WebSocket-Event-Frequenz messen: Wie oft kommen `sensor_data` Events?
2. Zone-KPI-Neuberechnung: Wird bei JEDEM sensor_data Event die gesamte L1 neu gerendert?
3. `groupDevicesByZone()` Performance: Wird bei jedem Event neu gruppiert oder ist es gecached?
4. Sparkline-Cache via `useSparklineCache.ts`: Funktioniert 5s-Deduplizierung korrekt?

**Anforderung — Polishing:**
- Zone-KPIs sollten via `computed` automatisch reaktiv sein, aber NICHT bei jedem WS-Event ein volles Re-Render ausloesen
- Wenn Performance-Problem identifiziert → `shallowRef` oder `triggerRef` fuer gezielte Updates evaluieren

## UX-Prinzipien (eingebettet — nicht extern referenziert)

| Prinzip | Quelle | Anwendung auf Ebene 1 |
|---------|--------|----------------------|
| **5-Sekunden-Regel** | Dashboard UX Best Practices | Zone-Card-Status-Ampel beantwortet sofort: "Muss ich handeln?" |
| **7-Sekunden-Regel** | Sarikaya et al. 2019 | Gesamt-Monitor L1 liefert in <7s die Kernaussage des Systems |
| **Doppelte Kodierung** | Bancilhon et al. 2023, CHI | Nie Farbe allein — Status als Farbe + Text + Icon |
| **Kontext statt Zahlen** | Dashboard Chart-Editor UX | "23.5°C (optimal)" statt "23.5", "2/3 online" statt "2" |
| **SPOG-Formel** | Konsolidierungs-Grundregel | Right Info (KPIs), Right Time (live), Right Person (Operator), Minimal Friction (1 Blick) |
| **Progressive Disclosure** | Nielsen Norman Group | L1 zeigt nur Zonen-Status — Details auf L2/L3 |
| **Shneiderman-Mantra** | Baudisch et al. 2002 | "Overview first" — L1 IST der Overview |
