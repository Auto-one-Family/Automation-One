# Plan: Monitor Ebene 3 — Editor-Integration & Nutzerkonfigurierbare Dashboards

## Kontext

Der Monitor hat 3 Navigationsebenen (L1 Zone-Uebersicht, L2 Zonenansicht, L3 Detail). Aktuell fuehren alle Dashboard-Links auf L1/L2 zum Editor-Tab (`/editor?layout=<id>`), was den Monitor-Kontext bricht. Dieser Plan verbindet Editor und Monitor zu einem durchgaengigen Erlebnis: Dashboards werden im Monitor angezeigt (View-Mode), der Editor wird zum Bearbeitungswerkzeug.

## Kontext: Was ist Ebene 3?

Der Monitor hat drei Navigationsebenen:

```
Ebene 1: Zonen-Uebersicht (alle Zonen, Cross-Zone-Dashboard-Links)
    ↓ Klick auf Zone
Ebene 2: Zonenansicht (Subzone-Accordions, Sensor/Aktor-Cards, Zone-Dashboard-Links)
    ↓ Klick auf Sensor ODER Klick auf Dashboard-Link
Ebene 3: Detail-Ebene (Sensor-Detail ODER User-konfiguriertes Dashboard)
```

Ebene 3 hat ZWEI Einstiegspunkte:
1. **Sensor-Detail-SlideOver:** Klick auf SensorCard in L2 → Zeitreihe + Detail-Infos (bereits als SlideOver durch Konsolidierung implementiert)
2. **User-Dashboard-Ansicht:** Klick auf Dashboard-Link (L1 oder L2) → Dashboard im View-Mode anzeigen

Dieser Auftrag verbindet die im **Editor-Tab** erstellten Dashboards mit der **Monitor-Ansicht** und schafft den Kreislauf:
- Monitor zeigt Links → User klickt → Dashboard wird angezeigt → User will anpassen → Editor oeffnet sich → User speichert → Monitor zeigt aktualisiertes Dashboard

## Erweiterte Vision: Editor als universeller Flaechenkonfigurator

Der Editor ist NICHT nur ein isolierter Dashboard-Builder — er ist das zentrale Werkzeug,
mit dem der Benutzer ALLE konfigurierbaren Flaechen der Anwendung personalisieren kann:

### Was der Benutzer koennen soll:
1. **Widgets/Panels im Editor erstellen** → und dann entscheiden, WO sie dauerhaft angezeigt werden:
   - Im Monitor-Tab (als L3-Dashboard-Seite ODER als Erweiterung der bestehenden Zone-Cards auf L1/L2)
   - Im Uebersichtstab (Hardware-View) als benutzerdefiniertes Side-Panel oder Karten-Erweiterung
   - Als eigenstaendige Dashboard-Seite (bisheriges Verhalten)
2. **An der Endposition bewegen** — einmal platzierte Elemente bleiben interaktiv:
   - Drag-Rearrange innerhalb der Zielansicht (Monitor L1/L2, Hardware)
   - Groesse anpassen (wenn das Ziel es erlaubt)
3. **Side-Panels erstellen** — schmale Widget-Leisten am Rand einer View:
   - Z.B. ein "Quick-Gauges"-Panel rechts neben dem Monitor L2
   - Z.B. ein "System-Health"-Panel in der Uebersicht
4. **Vollstaendig editierbar bleiben** — jedes platzierte Element kann jederzeit:
   - Im Editor geoeffnet werden → Konfiguration aendern
   - Verschoben werden (andere Position, andere View)
   - Geloescht werden (aus der Zielansicht entfernen)

### Architektur-Implikation: DashboardLayout.target

```typescript
/** Erweitert: Ziel-Platzierung eines Dashboards/Panels */
interface DashboardLayout {
  // ... bestehende Felder ...
  target?: DashboardTarget  // NEU: Wo wird dieses Layout angezeigt?
}

interface DashboardTarget {
  view: 'editor' | 'monitor' | 'hardware'  // Ziel-Tab
  placement: 'page' | 'inline' | 'side-panel'  // Darstellungsart
  /** Bei 'inline': An welcher Stelle in der Ziel-View (z.B. 'zone-header', 'after-sensors') */
  anchor?: string
  /** Bei 'side-panel': Position links/rechts, Breite */
  panelPosition?: 'left' | 'right'
  panelWidth?: number  // in Spalten (1-4 von 12)
  /** Reihenfolge innerhalb der gleichen anchor-Position */
  order?: number
}
```

### Verknuepfung mit bestehendem System:
- `scope` bleibt fuer die INHALTLICHE Zuordnung (zone, cross-zone, sensor-detail)
- `target` ist NEU fuer die VISUELLE Platzierung (wo wird es gerendert)
- Dashboard OHNE target = Standard-Verhalten (Editor-Tab eigene Seite)
- Dashboard MIT target.view='monitor' + target.placement='page' = Block 2+3 (DashboardViewer)
- Dashboard MIT target.view='monitor' + target.placement='inline' = Eingebettetes Widget-Panel auf L1/L2
- Dashboard MIT target.view='hardware' + target.placement='side-panel' = Seitliches Panel in Uebersicht

### Persistierung:
- `target` wird als Teil des DashboardLayout gespeichert (localStorage + Server-API)
- MonitorView und HardwareView lesen beim Mount alle Layouts mit passendem `target.view`
- Aenderungen am Target via Editor → sofort in Zielansicht sichtbar (reactiv via Pinia Store)

## Kernfrage: Wo werden User-Dashboards angezeigt?

### Option A: Im Editor-Tab (aktueller Zustand — Navigation weg vom Monitor)

```
Monitor L2 → Klick auf Dashboard-Link → /editor/:dashboardId (Editor-Tab oeffnet sich)
```

**Pro:** Einfach, kein neues Rendering noetig
**Contra:** User verliert Monitor-Kontext, muss zuruecknavigieren

### Option B: Inline im Monitor (Dashboard rendered innerhalb MonitorView)

```
Monitor L2 → Klick auf Dashboard-Link → MonitorView rendert Dashboard inline (SlideOver oder Fullscreen-Overlay)
```

**Pro:** User bleibt im Monitor-Kontext
**Contra:** Komplexer, GridStack muss in SlideOver/Overlay funktionieren

### Option C: Hybrid (EMPFOHLEN)

```
Monitor L2 → Klick auf Dashboard-Link → Dashboard oeffnet sich im VIEW-MODE
  - Kleiner "Im Editor oeffnen" Button oben rechts
  - Zurueck-Button → zurueck zu L2
  - Dashboard wird INNERHALB des Monitor-Tabs als eigene Sub-View gerendert
```

**Pro:** Bester Kompromiss — User bleibt im Monitor, kann aber zum Editor wechseln
**Contra:** Erfordert Dashboard-Rendering-Komponente die OHNE GridStack-Edit-Features funktioniert

**Empfohlene Architektur fuer Option C:**
```
/monitor/:zoneId/dashboard/:dashboardId → MonitorView rendert DashboardViewer
/monitor/dashboard/:dashboardId          → Cross-Zone-Dashboard im Monitor
```

**DashboardViewer** = Neue Komponente (oder Modus von CustomDashboardView) die:
- GridStack im Static-Mode rendert (kein Drag, kein Resize)
- Alle Widget-Typen anzeigen kann
- "Im Editor bearbeiten" Button hat
- Breadcrumb: Monitor > Zone > Dashboard-Name

### Block 2: Dashboard-Viewer Komponente (Option C)

**Neue Komponente:** `DashboardViewer.vue`

**Aufgabe:** Rendert ein Dashboard-Layout im REINEN VIEW-MODE (kein Edit, kein DnD).

#### GridStack View-Only: Warum `staticGrid: true`?

> **Plattform-Vergleich (GridStack.js Docs + GitHub Issues #743, #517):**
> Es gibt DREI Ansaetze fuer View-Only-Rendering mit GridStack. Die Unterschiede
> sind erheblich — falsche Wahl kostet Performance oder Flexibilitaet.

| Ansatz | Code | Bundle-Impact | Performance | Wann verwenden? |
|--------|------|---------------|-------------|-----------------|
| **`staticGrid: true`** | `GridStack.init({ staticGrid: true })` | Voll-Library geladen, aber DnD-Events werden NICHT registriert | **Sehr gut** — keine Event-Listener fuer Drag/Resize | **DashboardViewer** (Block 2) |
| **`setStatic(true)`** | `grid.setStatic(true)` nach `init()` | Voll-Library + DnD bereits initialisiert, dann deaktiviert | Gut — Events registriert aber blockiert | Editor View/Edit-Toggle (bereits implementiert in CustomDashboardView) |
| **CSS-Grid-Only** | `grid-column: x/span w; grid-row: y/span h` | **0KB extra** — kein GridStack noetig | **Exzellent** — reines CSS | **InlineDashboardPanel** (Block 7) |

**Entscheidung fuer DashboardViewer:** `staticGrid: true` mit der Voll-Library.
- Die CSS-Klasse `.grid-stack-static` wird automatisch gesetzt → `cursor: default` statt `cursor: move`
- Float-Berechnung und Animate bleiben verfuegbar (visuelle Konsistenz mit Editor)
- Kein DnD-Overhead, da Event-Listener gar nicht erst registriert werden
- Zukunftssicher: Falls spaeter ein "Quick-Edit" Button im Viewer gewuenscht wird,
  kann per `grid.setStatic(false)` umgeschaltet werden ohne Re-Init

**Architektur:**
```
DashboardViewer.vue
  Props:
    - layoutId: string (Dashboard-Layout-ID)
    - showHeader: boolean (Dashboard-Name + Actions)

  Rendering:
    - GridStack mit staticGrid: true (NICHT setStatic() — wichtiger Unterschied!)
    - Alle Widget-Typen aus widgetComponentMap rendern
    - Widgets empfangen Live-Daten via WebSocket (gleich wie im Editor)
    - KEIN Drag, KEIN Resize, KEIN Zahnrad-Icon, KEIN Loeschen
    - CSS-Klasse .grid-stack-static → cursor: default auf allen Zellen

  Actions (als <router-link>, NICHT als @click!):
    - "Im Editor bearbeiten" → <router-link :to="{ name: 'editor-dashboard', params: { dashboardId: layoutId } }">
    - "Zurueck" → router.back() (einzige Ausnahme — Back-Button hat kein sinnvolles href)

  Header (wenn showHeader=true):
    - Dashboard-Name (aus DashboardLayout.name)
    - Breadcrumb: Monitor > Zone > Dashboard-Name
    - "Im Editor bearbeiten" Button (rechts)
    - Zeitbereich-Info (wenn Dashboard zeitbasierte Widgets hat)
```

**Wiederverwendung via `useDashboardWidgets.ts` Composable (Block 8):**
- `widgetComponentMap` (alle 9 Widget-Typen: line-chart, gauge, sensor-card, historical, multi-sensor, actuator-card, actuator-runtime, esp-health, alarm-list)
- `mountWidgetToElement()` / `unmountWidgetFromElement()` — Vue `h()`/`render()` in GridStack-Cells
- Widget-Konfiguration laden + an Komponente Props uebergeben
- `cleanupAllWidgets()` — `render(null, mountEl)` fuer alle gemounteten Widgets

### Block 3: Monitor-Routen fuer Dashboard-Ansicht

#### SPA-Navigation-Grundregeln (KRITISCH fuer gesamten Block 3+4+5)

> **Anti-Pattern das KONSEQUENT vermieden werden muss:**
> ```html
> <!-- FALSCH — niemals so implementieren: -->
> <button @click="router.push('/monitor/dashboard/' + id)">Dashboard anzeigen</button>
> ```
> **Problem:** Ein `<button>` ohne `href` verhindert:
> - Middle-Click (neuer Tab) — User muss immer im gleichen Tab bleiben
> - Rechtsklick → "Link kopieren" — URL ist nicht inspizierbar
> - Rechtsklick → "In neuem Tab oeffnen" — nicht verfuegbar
> - Browser "Zurueck"-Vorhersage (Prefetch bei Hover) — funktioniert nicht
>
> **Korrekt — IMMER `<router-link>` verwenden:**
> ```html
> <router-link :to="{ name: 'monitor-dashboard', params: { dashboardId: id } }">
>   Dashboard anzeigen
> </router-link>
> ```
> `<router-link>` rendert als `<a>` mit echtem `href` — alle Browser-Standard-Interaktionen
> funktionieren. Styling als Button ist via CSS moeglich (`class="btn"`).

#### `router.push()` vs. `router.replace()` — Wann was?

| Aktion | Methode | Begruendung |
|--------|---------|-------------|
| Dashboard-Link klicken (L1/L2 → L3) | `router.push()` via `<router-link>` | Neuer "Ort" → History-Eintrag → "Zurueck" funktioniert |
| SlideOver schliessen | `router.replace()` | SlideOver ist Overlay, kein neuer "Ort" → kein History-Eintrag |
| "Im Editor bearbeiten" | `router.push()` via `<router-link>` | Tab-Wechsel = neuer "Ort" |
| "Zurueck" Button | `router.back()` | Nutzt existierenden History-Stack |
| TimeRange-Aenderung | `router.replace()` mit Query-Update | Filter-Aenderung, kein neuer "Ort" |

> **ThingsBoard-Analogie:** Drill-Down (neuer State auf Stack) = `router.push()`.
> State-Change (aktuellen State ersetzen) = `router.replace()`.
> AutomationOne: DashboardViewer = Drill-Down, SlideOver = State-Change.

**Bestehende Routes in router/index.ts (Zeile 58–75):**

### Block 5: Editor → Monitor Rueckweg

**Problem:** Wenn User ein Dashboard im Editor bearbeitet hat, wie kommt er zurueck zum Monitor?

#### Kontext-Erhaltung via URL-Query-Parameter (Grafana-Pattern)

> **Grafana propagiert Kontext zwischen Views ueber URL-Query-Parameter:**
> `?from=now-6h&to=now&var-zone=gewaechshaus` — Zeitbereich + Template-Variablen
> werden automatisch in Links eingebettet. So geht kein Kontext verloren beim View-Wechsel.

**Loesung — "Im Monitor anzeigen" als `<router-link>` mit Kontext:**

```html
<!-- In CustomDashboardView.vue — nur sichtbar wenn Dashboard scope vorhanden -->
<router-link
  v-if="currentLayout?.scope"
  :to="monitorRouteForLayout"
  class="btn btn-secondary"
>
  Im Monitor anzeigen
</router-link>
```

```typescript
// Computed: Korrekte Monitor-Route basierend auf scope
const monitorRouteForLayout = computed(() => {
  const layout = currentLayout.value
  if (!layout) return null

  if (layout.scope === 'zone' && layout.zoneId) {
    return {
      name: 'monitor-zone-dashboard',
      params: { zoneId: layout.zoneId, dashboardId: layout.id },
      // Grafana-Pattern: Zeitbereich als Query-Param mitgeben
      query: { from: timeRange.value.from, to: timeRange.value.to }
    }
  }
  if (layout.scope === 'cross-zone') {
    return {
      name: 'monitor-dashboard',
      params: { dashboardId: layout.id },
      query: { from: timeRange.value.from, to: timeRange.value.to }
    }
  }
  if (layout.scope === 'sensor-detail' && layout.sensorId && layout.zoneId) {
    return {
      name: 'monitor-sensor',
      params: { zoneId: layout.zoneId, sensorId: layout.sensorId }
    }
  }
  return null
})
```

**Kontext-Erhaltung bei Monitor → Editor → Monitor:**
1. User ist auf `/monitor/zone-1/dashboard/abc?from=now-6h&to=now`
2. Klickt "Im Editor bearbeiten" → `/editor/abc?from=now-6h&to=now` (Zeitbereich mitgenommen)
3. Im Editor: "Im Monitor anzeigen" Link → `/monitor/zone-1/dashboard/abc?from=now-6h&to=now`
4. Alternativ: "Zurueck" Button via `router.back()` → nutzt Browser-History-Stack

**Betroffene Datei:** `CustomDashboardView.vue` — neuer "Im Monitor anzeigen" `<router-link>` wenn Dashboard `scope !== undefined`

### Block 6: Auto-Generierung im Monitor-Kontext

> **Referenz-Pattern: Home Assistant "Areas Dashboard" (2025 Experimental)**
> Home Assistant generiert automatisch Dashboards fuer jeden Raum ("Area") basierend auf den
> dort registrierten Entities. Entities werden nach Domain gruppiert (Lichter oben, Sensoren
> mitte, Aktoren unten). Jede Area bekommt eine eigene Seite mit Tile-Cards.
> Das ist EXAKT das `generateZoneDashboard()` Konzept: Zone = Area, Sensoren/Aktoren = Entities.

**Was bereits existiert:**
- `generateZoneDashboard()` in dashboard.store.ts erstellt Auto-Layouts
- `claimAutoLayout()` macht Auto-Layout zum User-Layout

**Polishing — Sensor-Type-Aware Auto-Generierung:**
- **Erster Besuch einer Zone:** Wenn KEIN Dashboard existiert (weder auto noch user), wird `generateZoneDashboard()` aufgerufen
  - Auto-Layout basierend auf vorhandenen Sensoren — **gruppiert nach Kategorie** (HA Areas-Pattern):

    | Sensor-Typ | Widget-Typ | Anordnung | Smart Defaults aus SENSOR_TYPE_CONFIG |
    |-----------|-----------|-----------|--------------------------------------|
    | Temperatur | `line-chart` (24h) | Obere Zeile, volle Breite (w:12, h:3) | Y-Achse: 0-50°C, Schwellwerte 15-35°C |
    | Feuchtigkeit | `gauge` | Zweite Zeile, halbe Breite (w:6, h:2) | Bereich: 0-100%, Schwellwerte 30-80% |
    | pH | `gauge` mit Bereichs-Indikator | Zweite Zeile, halbe Breite (w:6, h:2) | Bereich: 0-14, Schwellwerte 5.5-6.5 |
    | EC/Leitfaehigkeit | `gauge` | Dritte Zeile | Bereich: 0-5 mS/cm |
    | Licht (PAR/LUX) | `line-chart` (24h) | Volle Breite | Y-Achse: 0-2000 µmol/m²/s |
    | Aktoren | `actuator-card` | Untere Zeile, Drittel-Breite (w:4, h:2) | Status + Laufzeit |

  - **Layout-Algorithmus:** Sensoren von oben nach unten anordnen nach Wichtigkeit:
    Zeitreihen (line-chart) oben (brauchen Platz), Gauges mittig, Aktoren unten
  - Widget-Konfiguration nutzt `SENSOR_TYPE_CONFIG` Smart Defaults (Y-Achsen, Einheit, Schwellwerte)

- **Auto-Update:** Wenn ein neuer Sensor zur Zone hinzugefuegt wird UND Dashboard `autoGenerated: true` → neues Widget ergaenzen
  - NUR wenn User das Auto-Dashboard NICHT uebernommen hat (claimed)
  - Pruefen via `autoGenerated === true`
  - Neues Widget wird an naechste freie Position gesetzt (GridStack `engine.findEmptyPosition()`)

- **"Uebernehmen" Flow auf L2/L3:**
  1. User sieht Auto-Dashboard im DashboardViewer
  2. Kleiner Banner: "Dieses Dashboard wurde automatisch erstellt. Im Editor anpassen?"
     - **Grafana-Pattern:** Informations-Banner mit klarem CTA, nicht blockierend
     - Banner verschwindet nach Klick auf "Spaeter" (Session-basiert, nicht persistent)
  3. Klick "Anpassen" → Editor oeffnet sich mit dem Dashboard (als `<router-link>` zum Editor)
  4. Klick "Uebernehmen" → `claimAutoLayout()` → Badge aendert sich von "Auto" zu "Benutzerdefiniert"
     - Nach Uebernahme: Auto-Update deaktiviert, User hat volle Kontrolle
     - Dashboard-Name wird editierbar (vorher war er "Auto: [Zone-Name]")

### Block 7: Inline-Panels & Side-Panels (Target-System)

**Ziel:** User kann Editor-erstellte Widgets nicht nur als eigene Dashboard-Seite anzeigen,
sondern DIREKT in Monitor L1/L2 oder Hardware-View einbetten.

#### Rendering-Strategie: CSS-Grid-Only fuer Inline-Panels (KEIN GridStack!)

> **Grafana Scenes Erkenntnis:** Die `EmbeddedScene` Klasse rendert Panels OHNE den
> vollen Dashboard-Container. Widgets werden programmatisch in beliebige Container gemountet.
> Fuer AutomationOne bedeutet das: Inline-Panels brauchen KEIN GridStack.
>
> **Warum?** Inline-Panels sind statisch (kein DnD noetig), kompakt und werden
> potentiell MEHRFACH pro View instanziiert. Jede GridStack-Instanz kostet ~77KB Memory
> plus Event-Listener. Bei 3 Inline-Panels auf L1 waeren das 231KB unnoetig.

**CSS-Grid-Only Rendering fuer Inline-Panels:**
```css
/* InlineDashboardPanel.vue — KEIN GridStack noetig */
.inline-panel-grid {
  display: grid;
  grid-template-columns: repeat(12, 1fr);  /* 12-Spalten wie GridStack */
  grid-auto-rows: 80px;                     /* cellHeight wie GridStack */
  gap: 8px;                                  /* margin wie GridStack */
}

/* Widget-Positionen direkt aus DashboardLayout.widgets[].x/y/w/h */
.inline-widget {
  grid-column: var(--gs-x) / span var(--gs-w);
  grid-row: var(--gs-y) / span var(--gs-h);
}
```

```typescript
// Widget-Positionen als CSS Custom Properties setzen:
widgets.forEach(widget => {
  const el = document.createElement('div')
  el.className = 'inline-widget'
  el.style.setProperty('--gs-x', String(widget.x + 1)) // CSS Grid ist 1-basiert!
  el.style.setProperty('--gs-w', String(widget.w))
  el.style.setProperty('--gs-y', String(widget.y + 1)) // CSS Grid ist 1-basiert!
  el.style.setProperty('--gs-h', String(widget.h))
  container.appendChild(el)
  mountWidgetToElement(widget.id, widget.type, widget.config, el, appContext)
})
```

> **Achtung: GridStack x/y ist 0-basiert, CSS Grid ist 1-basiert!**
> `widget.x = 0` → `grid-column: 1 / span w`. Daher `+ 1` beim Mapping.

**DashboardViewer (Block 2) vs. InlineDashboardPanel (Block 7):**

| Aspekt | DashboardViewer | InlineDashboardPanel |
|--------|----------------|---------------------|
| **Rendering** | GridStack `staticGrid: true` | CSS-Grid-Only (kein GridStack) |
| **Grund** | Float/Animate noetig, volle Seite | Kompakt, kein Float, mehrfach instanziiert |
| **Bundle-Impact** | GridStack bereits geladen | 0KB extra |
| **DnD** | Nie (statisch) | Optional (editable Prop) |
| **Container** | Eigene Route/Seite | Zwischen bestehenden Sektionen eingefuegt |

**Neue Komponente:** `InlineDashboardPanel.vue`

```
InlineDashboardPanel.vue
  Props:
    - layoutId: string (Dashboard-Layout-ID)
    - mode: 'inline' | 'side-panel'
    - panelPosition?: 'left' | 'right' (nur bei side-panel)
    - panelWidth?: number (px, nur bei side-panel — Default: 300)
    - editable: boolean (Drag-Rearrange an Endposition erlauben?)

  Rendering:
    - CSS-Grid mit 12-Spalten-System (KEIN GridStack!)
    - Widget-Positionen direkt aus DashboardLayout als CSS Custom Properties
    - Bei editable=true: Drag innerhalb des Panels erlaubt (nativer HTML Drag API)
    - Bei editable=false: Komplett statisch
    - "Bearbeiten" Icon → <router-link> zum Editor mit diesem Layout

  Layout:
    - inline: Volle Breite, zwischen bestehenden Sektionen eingefuegt
    - side-panel: Feste Breite rechts, scrollbar, neben Hauptinhalt
```

#### CSS-Layout fuer Side-Panels: Grid Template Areas (Empfohlen)

> **3 CSS-Ansaetze verglichen:**
>
> | Ansatz | Technik | Vorteil | Nachteil |
> |--------|---------|---------|----------|
> | **A: Grid Template Areas** | `grid-template-areas: 'main side'` | Explizit, lesbar, responsive via Media-Query | Erfordert Media-Query fuer Mobile |
> | **B: Flexbox Sidebar** | `flex-grow: 999` + `min-inline-size: 50%` | Auto-Wrap ohne Media-Query | Weniger Kontrolle ueber Side-Panel-Breite |
> | **C: CSS Variable Toggle** | `--side-panel-width: 0px/300px` | Animiertes Ein-/Ausblenden | Safari Grid-Transition Bugs (Stand 2025) |
>
> **Empfehlung: Ansatz A + C kombiniert** — Grid Template Areas als Basis-Layout,
> CSS Variable fuer den Collapsible-Toggle.

**CSS fuer MonitorView mit Side-Panel:**
```css
.monitor-layout {
  display: grid;
  grid-template-areas: 'main side';
  grid-template-columns: 1fr var(--side-panel-width, 0px);
  gap: 16px;
  transition: grid-template-columns 0.3s ease;
}

.monitor-layout__main {
  grid-area: main;
  overflow-y: auto;
}

.monitor-layout__side {
  grid-area: side;
  overflow-y: auto;      /* Side-Panel scrollt unabhaengig */
  max-height: 100vh;
  position: sticky;       /* Bleibt sichtbar beim Main-Scrollen */
  top: 0;
}

/* Side-Panel offen */
.monitor-layout.has-side-panel {
  --side-panel-width: 300px;  /* 240-360px optimal */
}

/* Responsive: Mobile stacking */
@media (max-width: 768px) {
  .monitor-layout {
    grid-template-areas: 'main' 'side';
    grid-template-columns: 1fr;
  }
  .monitor-layout__side {
    position: static;
    max-height: none;
  }
}
```

**Side-Panel Best Practices (aus Plattform-Vergleich):**
- **Position: RECHTS** — fuer sekundaere Information (Monitoring-Kontext). Links ist fuer Navigation.
- **Breite: 240-360px fest** (nicht prozentual) — bei 12-Spalten-System = 3-4 Grid-Spalten
- **Collapsible:** Toggle-Button (Chevron) am oberen Rand. Zustand in localStorage persistieren
- **Scrolling:** Side-Panel scrollt unabhaengig vom Main-Content (`overflow-y: auto`)
- **Sticky:** Side-Panel bleibt beim Scrollen des Main-Contents sichtbar (`position: sticky`)

### Block 8: Widget-Extraktion als Shared Module

> **Architektur-Referenz: Grafana Scenes `VizPanel`**
> Grafana Scenes trennt die Widget-Rendering-Logik (`VizPanel`) vollstaendig vom
> Container-Layout (`SceneGridLayout` / `SceneFlexLayout`). Ein `VizPanel` kann in
> JEDEM Container leben — Dashboard, EmbeddedScene, Explore, Alerting.
> Die Daten kommen hierarchisch via Scene-Graph (Parent → Child Vererbung).
>
> **AutomationOne-Analogie:** `useDashboardWidgets.ts` = VizPanel-Registry.
> Widgets sind container-agnostisch — sie funktionieren in GridStack (DashboardViewer),
> CSS-Grid (InlineDashboardPanel) und im Editor (CustomDashboardView).
> Die Daten kommen via Pinia Store → Widget Props (statt Scene-Graph).

**Problem:** DashboardViewer (Block 2), InlineDashboardPanel (Block 7) und CustomDashboardView
muessen ALLE dasselbe Widget-Rendering verwenden. Aktuell ist die gesamte Widget-Logik
in CustomDashboardView.vue (1141 Zeilen) eingebettet.

**Loesung:** Shared Widget-Rendering Modul extrahieren:

```
El Frontend/src/composables/useDashboardWidgets.ts   (NEU)
  Exports:
    - widgetComponentMap: Record<WidgetType, Component>
      // Mapping: 'line-chart' → LineChartWidget, 'gauge' → GaugeWidget, etc.
      // Alle 9 Widget-Typen zentral registriert

    - widgetTypesMeta: { type, label, icon, w, h, minW, minH, category }[]
      // Metadaten fuer Widget-Katalog und Auto-Generierung
      // category: 'sensors' | 'actuators' | 'system'

    - mountWidgetToElement(widgetId, widgetType, config, mountEl, appContext): void
      // Vue h()/render() in beliebiges DOM-Element
      // Container-agnostisch: funktioniert in GridStack-Cell UND CSS-Grid-Cell
      // appContext wird fuer Pinia Store Injection benoetigt

    - unmountWidgetFromElement(widgetId, mountEl): void
      // render(null, mountEl) — sauberes Unmount
      // WICHTIG: Vor GridStack.destroy() aufrufen!

    - cleanupAllWidgets(mountedWidgets: Map): void
      // Iteriert ueber alle gemounteten Widgets und unmounted sie
      // Fuer onUnmounted() Hook in DashboardViewer/InlineDashboardPanel
```

**Container-agnostisches Mounting (Schluessel-Prinzip):**
```typescript
// mountWidgetToElement ist UNABHAENGIG vom Container-System:
// - In GridStack: mountEl = grid.getGridItems()[i].querySelector('.grid-stack-item-content')
// - In CSS-Grid: mountEl = document.querySelector('.inline-widget[data-id="..."]')
// - In Editor: mountEl = GridStack-Cell Content-Bereich
// Das Widget weiss NICHT ob es in GridStack oder CSS-Grid lebt.
```

## UX-Prinzipien (eingebettet — nicht extern referenziert)

> Diese Prinzipien sind NICHT theoretisch — jedes einzelne wurde durch den Plattform-Vergleich
> (ThingsBoard, Grafana, Home Assistant, Azure IoT Central, Node-RED Dashboard 2.0)
> bestaetigt und mit konkreter Implementierungs-Relevanz fuer AutomationOne verknuepft.

| Prinzip | Anwendung auf Ebene 3 | Plattform-Referenz |
|---------|----------------------|-------------------|
| **Shneiderman-Mantra** | "Details-on-demand" — L3 ist die Detail-Ebene nach Overview (L1) und Zoom (L2). SlideOver = Detail, DashboardViewer = Custom-Detail | ThingsBoard: Root State → Sub-State (Drill-Down) |
| **Focus+Context** | DashboardViewer hat Breadcrumb + "Zurueck" — User weiss immer wo er ist. SlideOver laesst L2 im Hintergrund sichtbar | Grafana: Panel-Editor Sidebar neben Live-Dashboard |
| **Progressive Disclosure** | L3 ist die LETZTE Ebene (max 3) — hier gibt es die vollen Details. SlideOver-Sektion 4 (Kontext) ist Accordion (eingeklappt) | Grafana Design System: "Show advanced options" Expander |
| **SPOG-Formel** | Sensor-Detail zeigt ALLES zu einem Sensor an einem Ort: Zeitreihe, Stats, Rules, Metadata — 5 Sektionen vertikal gestapelt | Home Assistant: Entity-Dialog (alles vertikal, keine Tabs) |
| **Component-Sprawl #6** | DashboardViewer + InlineDashboardPanel nutzen `useDashboardWidgets` — KEIN Code-Duplikat. Container-agnostisches Widget-Mounting | Grafana Scenes: VizPanel unabhaengig vom Container |
| **Kontext statt Zahlen** | Sensor-Detail: Statistik-Zeile gibt Kontext (Min/Max/Avg MIT Zeitpunkt), Threshold-Linien im Chart, Zone+Subzone-Info | Azure IoT Central: Conditional Formatting (Wert → Farbe/Icon/Text) |
| **Doppelte Kodierung** | Chart-Farben + Threshold-Labels + Statistik-Text — nie Farbe allein (WCAG-konform, Barrierefreiheit) | Grafana: Color + Icon + Text fuer Validierungsfehler |
| **7-Sekunden-Regel** | DashboardViewer-Dashboards muessen in <7s ihre Kernaussage liefern. `staticGrid: true` spart DnD-Init-Zeit | ThingsBoard: View-Mode ohne Edit-Overhead |
| **User Ownership** | Jedes platzierte Element bleibt IMMER editierbar — "Im Editor bearbeiten" Link auf jedem DashboardViewer und InlineDashboardPanel | ThingsBoard: Edit-Mode jederzeit aktivierbar |
| **Browser-Konventionen** | ALLE navigierenden Elemente als `<router-link>` (rendert `<a>` mit `href`). Middle-Click, Rechtsklick, Tab-Oeffnen funktionieren | SPA-Anti-Pattern-Vermeidung: kein `@click + router.push()` fuer Navigation |
| **Kontext-Propagation** | Zeitbereich + Filter via URL-Query-Parameter zwischen Views teilen. Editor → Monitor → Editor ohne Datenverlust | Grafana: `?from=now-6h&to=now&var-zone=...` |
| **Container-Hierarchie** | Inline (1-3 Felder) → Dialog (4-8) → SlideOver (9-15) → Full Page (15+). Container-Wahl nach Informationsmenge, nicht nach Praeferenz | ThingsBoard + Grafana + Home Assistant: Konvergierendes Pattern |
