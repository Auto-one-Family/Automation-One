## Kontext: Was ist Ebene 3?

Der Monitor hat drei Navigationsebenen:

```
Ebene 1: Zonen-Uebersicht (alle Zonen, Cross-Zone-Dashboard-Links)
    ↓ Klick auf Zone
Ebene 2: Zonenansicht (Subzone-Accordions, Sensor/Aktor-Cards, Zone-Dashboard-Links)
    ↓ Klick auf Sensor ODER Klick auf Dashboard-Link
Ebene 3: Detail-Ebene (Sensor-Detail ODER User-konfiguriertes Dashboard)
```

Ebene 3 hat ZWEI Einstiegspunkte:
1. **Sensor-Detail-SlideOver:** Klick auf SensorCard in L2 → Zeitreihe + Detail-Infos (bereits als SlideOver durch Konsolidierung implementiert)
2. **User-Dashboard-Ansicht:** Klick auf Dashboard-Link (L1 oder L2) → Dashboard im View-Mode anzeigen

Dieser Auftrag verbindet die im **Editor-Tab** erstellten Dashboards mit der **Monitor-Ansicht** und schafft den Kreislauf:
- Monitor zeigt Links → User klickt → Dashboard wird angezeigt → User will anpassen → Editor oeffnet sich → User speichert → Monitor zeigt aktualisiertes Dashboard

## Kernfrage: Wo werden User-Dashboards angezeigt?

### Option A: Im Editor-Tab (aktueller Zustand — Navigation weg vom Monitor)

```
Monitor L2 → Klick auf Dashboard-Link → /editor/:dashboardId (Editor-Tab oeffnet sich)
```

**Pro:** Einfach, kein neues Rendering noetig
**Contra:** User verliert Monitor-Kontext, muss zuruecknavigieren

### Option B: Inline im Monitor (Dashboard rendered innerhalb MonitorView)

```
Monitor L2 → Klick auf Dashboard-Link → MonitorView rendert Dashboard inline (SlideOver oder Fullscreen-Overlay)
```

**Pro:** User bleibt im Monitor-Kontext
**Contra:** Komplexer, GridStack muss in SlideOver/Overlay funktionieren

### Option C: Hybrid (EMPFOHLEN)

```
Monitor L2 → Klick auf Dashboard-Link → Dashboard oeffnet sich im VIEW-MODE
  - Kleiner "Im Editor oeffnen" Button oben rechts
  - Zurueck-Button → zurueck zu L2
  - Dashboard wird INNERHALB des Monitor-Tabs als eigene Sub-View gerendert
```

**Pro:** Bester Kompromiss — User bleibt im Monitor, kann aber zum Editor wechseln
**Contra:** Erfordert Dashboard-Rendering-Komponente die OHNE GridStack-Edit-Features funktioniert

**Empfohlene Architektur fuer Option C:**
```
/monitor/:zoneId/dashboard/:dashboardId → MonitorView rendert DashboardViewer
/monitor/dashboard/:dashboardId          → Cross-Zone-Dashboard im Monitor
```

**DashboardViewer** = Neue Komponente (oder Modus von CustomDashboardView) die:
- GridStack im Static-Mode rendert (kein Drag, kein Resize)
- Alle Widget-Typen anzeigen kann
- "Im Editor bearbeiten" Button hat
- Breadcrumb: Monitor > Zone > Dashboard-Name

### Block 1: Sensor-Detail SlideOver Polishing

**Was bereits existiert:** SlideOver mit Zeitreihe (aus Konsolidierung)

**Anforderung — autoops:debug + Playwright Analyse:**
1. SlideOver-Oeffnung nach SensorCard-Klick testen (funktioniert der Trigger?)

3. TimeRangeSelector: Funktionieren alle Presets?
4. CSV-Export: Generiert korrekte Datei?
5. Breadcrumb: Zeigt "Monitor > Zone > Sensor-Name"?

**Anforderung — Polishing:**

- **Zeitreihe:**

  - Threshold-Linien automatisch aus SENSOR_TYPE_CONFIG (wenn vorhanden)

- **Zusatzinformationen unter dem Chart:**
  - Statistik: Min / Max / Durchschnitt / Aktuell (fuer gewaehlten Zeitraum)
  - Letztes Update: Timestamp + Stale-Indikator

  - Metadata (DeviceMetadataSection kompakt, wenn Daten vorhanden)
- **Quick-Actions:**
  - "Sensor konfigurieren" → `/sensors` mit Sensor vorselektiert
  - "Zeitreihe exportieren (CSV)" → CSV-Download (bereits vorhanden)
  - "Im Dashboard einbetten" → `/editor` mit neuem Widget fuer diesen Sensor
- **Schliessen-Verhalten:** ESC-Taste, Klick ausserhalb, X-Button → SlideOver schliesst, URL zurueck zu L2

### Block 2: Dashboard-Viewer Komponente (Option C)

**Neue Komponente:** `DashboardViewer.vue`

**Aufgabe:** Rendert ein Dashboard-Layout im REINEN VIEW-MODE (kein Edit, kein DnD).

**Architektur:**
```
DashboardViewer.vue
  Props:
    - layoutId: string (Dashboard-Layout-ID)
    - showHeader: boolean (Dashboard-Name + Actions)

  Rendering:
    - GridStack im Static-Mode: grid.setStatic(true)
    - Alle Widget-Typen aus widgetComponentMap rendern
    - Widgets empfangen Live-Daten via WebSocket (gleich wie im Editor)
    - KEIN Drag, KEIN Resize, KEIN Zahnrad-Icon, KEIN Loeschen

  Actions:
    - "Im Editor bearbeiten" → router.push('/editor/' + layoutId)
    - "Zurueck" → router.back()
```

**Wiederverwendung aus CustomDashboardView.vue:**
- `widgetComponentMap` (alle 9 Widget-Typen)
- Widget-Mounting-Logik (Vue-Komponente in GridStack-Cell mounten)
- Widget-Konfiguration laden + an Komponente uebergeben
- `onUnmounted()` Cleanup-Logik

**Unterschiede zu CustomDashboardView.vue:**
- KEIN GridStack-Edit-Mode (static: true)
- KEIN Widget-Katalog-Sidebar
- KEIN WidgetConfigPanel
- KEIN Layout-Selector/Switcher
- KEIN Export/Import
- KEIN Create/Delete Layout
- NUR Rendering + Live-Daten + "Im Editor bearbeiten" Button

**Alternative:** Statt neue Komponente → `CustomDashboardView.vue` mit `viewOnly: boolean` Prop erweitern
- **Pro:** Kein Code-Duplikat
- **Contra:** CustomDashboardView ist bereits 897 Zeilen — noch ein Mode macht sie komplexer
- **Empfehlung:** Entscheide basierend auf Code-Analyse ob Extraktion oder Prop sinnvoller ist

### Block 3: Monitor-Routen fuer Dashboard-Ansicht

**Neue Routes in router/index.ts:**

```typescript
// Zone-spezifisches Dashboard im Monitor
{
  path: '/monitor/:zoneId/dashboard/:dashboardId',
  name: 'monitor-zone-dashboard',
  component: MonitorView,  // oder DashboardViewerWrapper
  meta: { title: 'Zone-Dashboard' }
}

// Cross-Zone-Dashboard im Monitor
{
  path: '/monitor/dashboard/:dashboardId',
  name: 'monitor-dashboard',
  component: MonitorView,  // oder DashboardViewerWrapper
  meta: { title: 'Dashboard' }
}
```

**MonitorView muss erkennen:**
- Keine Route-Parameter → L1 (Zone-Uebersicht)
- `:zoneId` → L2 (Zonenansicht)
- `:zoneId/sensor/:sensorId` → L2 + L3 SlideOver (Sensor-Detail)
- `:zoneId/dashboard/:dashboardId` → L2 + L3 DashboardViewer (Zone-Dashboard)
- `/dashboard/:dashboardId` → L3 DashboardViewer (Cross-Zone-Dashboard)

**Template-Logik:**
```html
<!-- L1: Keine zoneId -->
<template v-if="!route.params.zoneId">
  <!-- Zone-Cards + Cross-Zone-Links -->
</template>

<!-- L2: zoneId vorhanden -->
<template v-else-if="!route.params.dashboardId && !route.params.sensorId">
  <!-- Subzone-Accordions + Zone-Dashboard-Links -->
</template>

<!-- L3 Variante A: Sensor-Detail SlideOver -->
<SlideOver v-if="route.params.sensorId" ... />

<!-- L3 Variante B: Dashboard-Viewer -->
<DashboardViewer v-if="route.params.dashboardId" :layoutId="route.params.dashboardId" />
```

### Block 5: Editor → Monitor Rueckweg

**Problem:** Wenn User ein Dashboard im Editor bearbeitet hat, wie kommt er zurueck zum Monitor?

**Loesung:**
- **Im Editor-Tab:** "Dashboard im Monitor anzeigen" Button/Link
  - Fuer Zone-Dashboards: → `/monitor/:zoneId/dashboard/:dashboardId`
  - Fuer Cross-Zone-Dashboards: → `/monitor/dashboard/:dashboardId`
  - Fuer Sensor-Detail-Dashboards: → `/monitor/:zoneId/sensor/:sensorId`
- **Kontext-Erhaltung:** Wenn User vom Monitor zum Editor navigiert (via "Im Editor bearbeiten"):
  - Editor zeigt einen "Zurueck zum Monitor" Button
  - Router-History ermoeglicht `router.back()` zurueck zum Monitor

**Betroffene Datei:** `CustomDashboardView.vue` — neuer "Im Monitor anzeigen" Button wenn Dashboard scope != undefined

### Block 6: Auto-Generierung im Monitor-Kontext

**Was bereits existiert:**
- `generateZoneDashboard()` in dashboard.store.ts erstellt Auto-Layouts
- `claimAutoLayout()` macht Auto-Layout zum User-Layout

**Polishing:**
- **Erster Besuch einer Zone:** Wenn KEIN Dashboard existiert (weder auto noch user), wird `generateZoneDashboard()` aufgerufen
  - Auto-Layout basierend auf vorhandenen Sensoren:
    - Temperatur-Sensoren → `line-chart` Widget (24h)
    - Feuchtigkeit-Sensoren → `gauge` Widget
    - pH-Sensoren → `gauge` Widget mit Bereichs-Indikator
    - Aktoren → `actuator-card` Widgets
  - Widget-Konfiguration nutzt `SENSOR_TYPE_CONFIG` Smart Defaults (Y-Achsen, Einheit, Schwellwerte)
- **Auto-Update:** Wenn ein neuer Sensor zur Zone hinzugefuegt wird UND Dashboard `autoGenerated: true` → neues Widget ergaenzen
  - NUR wenn User das Auto-Dashboard NICHT uebernommen hat (claimed)
  - Pruefen via `autoGenerated === true`
- **"Uebernehmen" Flow auf L2/L3:**
  1. User sieht Auto-Dashboard im DashboardViewer
  2. Kleiner Banner: "Dieses Dashboard wurde automatisch erstellt. Im Editor anpassen?"
  3. Klick "Anpassen" → Editor oeffnet sich mit dem Dashboard
  4. Klick "Uebernehmen" → `claimAutoLayout()` → Badge aendert sich von "Auto" zu "Benutzerdefiniert"

## UX-Prinzipien (eingebettet — nicht extern referenziert)

| Prinzip | Anwendung auf Ebene 3 |
|---------|----------------------|
| **Shneiderman-Mantra** | "Details-on-demand" — L3 ist die Detail-Ebene nach Overview (L1) und Zoom (L2) |
| **Focus+Context** | DashboardViewer hat Breadcrumb + "Zurueck" — User weiss immer wo er ist |
| **Progressive Disclosure** | L3 ist die LETZTE Ebene (max 3) — hier gibt es die vollen Details |
| **SPOG-Formel** | Sensor-Detail zeigt ALLES zu einem Sensor an einem Ort: Zeitreihe, Stats, Rules, Metadata |
| **Component-Sprawl #6** | DashboardViewer wiederverwendet Widget-System aus Editor, keine Duplikate |
| **Kontext statt Zahlen** | Sensor-Detail: Statistik-Zeile gibt Kontext (Min/Max/Avg), Threshold-Linien im Chart |
| **Doppelte Kodierung** | Chart-Farben + Threshold-Labels + Statistik-Text — nie Farbe allein |
| **7-Sekunden-Regel** | DashboardViewer-Dashboards muessen in <7s ihre Kernaussage liefern |
