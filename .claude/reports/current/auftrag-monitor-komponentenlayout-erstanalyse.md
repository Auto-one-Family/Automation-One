## Kontext: Was passiert hier?

Der **Komponenten-Tab** (SensorsView, Route `/sensors`) zeigt aktuell Sensoren und Aktoren organisiert nach Zonen und Subzonen — mit Sparklines, Filtern und CRUD-Funktionalitaet. Diese Monitoring-Struktur ist gut, aber am FALSCHEN Ort. Sie gehoert in den **Monitor-Tab** (MonitorView, Route `/monitor`).

Der MonitorView hat aktuell eine flache 2-Level-Struktur OHNE Subzone-Gruppierung. Die SensorsView hat eine ueberlegene Struktur (Zone-Accordion → Subzone-Accordion → Sensor/Aktor-Cards). Diese Struktur muss nach MonitorView portiert werden.

**Ziel:** MonitorView wird zur zentralen Monitoring-Anlaufstelle mit der guten Struktur aus SensorsView — OHNE CRUD, nur Read-Only Live-Daten.

## IST-Zustand der betroffenen Komponenten

### MonitorView.vue (~985 Zeilen)

- **Pfad:** `El Frontend/src/views/MonitorView.vue`
- **Route:** `/monitor` (name: 'monitor'), `/monitor/:zoneId` (name: 'monitor-zone')
- **Datenquelle:** `useEspStore()` aus `@/stores/esp`

## Was dieser Auftrag implementiert

### Block 1: MonitorView Level 2 — Subzone-Gruppierung portieren

**Kernaufgabe:** Die `sensorsByZone` computed-Logik aus SensorsView (ZoneGroup → SubzoneGroup Nesting) nach MonitorView Level 2 portieren.

**Schritte:**
1. `sensorsByZone` Computed-Property aus SensorsView extrahieren → idealerweise als Composable (`useZoneGrouping.ts`) damit BEIDE Views es nutzen koennen
2. MonitorView Level 2 Template umbauen: Zone-Accordion → Subzone-Accordion → Sensor/Aktor-Cards
3. Accordion-State auf localStorage umstellen (nicht nur in-memory Set)
4. CRUD-Buttons und Config-Panels ENTFERNEN (Monitor = Read-Only)
5. Daten-Anzeige optimieren fuer Monitoring: Aktueller Wert, Sparkline, Status-Dot+Text, ESP-Badge

**UX-Prinzipien (direkt eingebettet — nicht im Life-Repo):**

| Prinzip | Quelle | Anwendung |
|---------|--------|-----------|
| **Shneiderman-Mantra** | Baudisch et al. 2002, 275 cit. | "Overview first, zoom and filter, then details-on-demand" — Zone-Tiles (L1) → Subzone-Gruppierung (L2) → Sensor-Detail (L3) |
| **5-Sekunden-Regel** | Dashboard UX Best Practices | Haupt-Dashboard beantwortet in 5s: "Ist alles OK?" — Zone-Cards mit Status-Ampel |
| **7-Sekunden-Regel** | Sarikaya et al. 2019, 320 cit. | Operational Dashboard braucht <7s bis zur Kernaussage "Muss ich handeln?" |
| **Focus+Context** | Baudisch et al. 2002 | Bei Drill-Down zum Detail muss Overview SICHTBAR BLEIBEN (Breadcrumb mit Status-Indikatoren) |
| **Doppelte Kodierung** | Bancilhon et al. 2023, CHI | Nie Farbe allein — immer Status als Text+Farbe+Form: "Offline (3h)" + roter Kreis |
| **Kontext statt Zahlen** | Dashboard Chart-Editor UX | Statt "23.5" → "23.5°C (optimal: 20-25°C)" mit Fortschrittsbalken |
| **Progressive Disclosure** | Nielsen Norman Group | Max 3 Ebenen — mehr erzeugt "Lost in Navigation" |
| **Component-Sprawl vermeiden** | KI-Antipattern #6 | Eine konfigurierbare Card-Komponente statt 15 leicht unterschiedliche |
| **SPOG-Formel** | Konsolidierungs-Grundregel | "Right Info, Right Time, Right Person, Minimal Friction" — kontextbezogene Progressive Disclosure |

### Block 2: MonitorView Level 1 — Cross-Zone-Dashboard-Links

**Kernaufgabe:** Auf der Ebene 1 (Zone-Tiles) einen Bereich fuer Cross-Zone-Dashboards als LinkCards hinzufuegen.

**Schritte:**
1. DashboardLayout Interface in `dashboard.store.ts` erweitern:
   ```typescript
   interface DashboardLayout {
     // ... bestehende Felder
     scope: 'zone' | 'cross-zone' | 'sensor-detail'
     zoneId?: string  // nur bei scope === 'zone'
     autoGenerated?: boolean
     sensorId?: string  // nur bei scope === 'sensor-detail'
   }
   ```
2. Neue Computed-Property in dashboard.store: `crossZoneDashboards` — filtert Layouts mit `scope === 'cross-zone'`
3. MonitorView L1 Template erweitern: Nach Zone-Tiles eine Sektion "Cross-Zone Dashboards" mit LinkCards
4. LinkCards zeigen: Dashboard-Name, Anzahl enthaltener Sensoren/Zonen, Link zum Dashboard im Editor
5. Diese Sektion erscheint NUR wenn der User Cross-Zone-Dashboards im Editor erstellt hat (keine Auto-Generierung)

### Block 3: Zone-spezifische Dashboard-Links in Level 2

**Kernaufgabe:** In der Zonenansicht (L2) Zone-interne Dashboards als Links anzeigen.

**Schritte:**
1. Neue Computed-Property: `zoneDashboards(zoneId)` — filtert Layouts mit `scope === 'zone' && zoneId === current`
2. Am Ende der L2-Ansicht eine Sektion "Zone-Dashboards" mit Auto-generierten + User-erstellten Dashboards
3. Auto-generierte Zone-Dashboards: Klick → Dashboard oeffnet sich im Editor/View-Mode mit Zone-gefilterten Widgets
4. Link zum Dashboard-Editor fuer Anpassung

### Block 4: Sensor-Detail als Ebene 3

**Kernaufgabe:** Klick auf einen Sensor in L2 → Detailansicht mit Zeitreihe.

**Schritte:**
1. Entscheidung: SensorHistoryView als eigene L3-Ansicht einbetten ODER als SlideOver/Inline-Expansion
2. Route erweitern: `/monitor/:zoneId/sensor/:sensorId` (optional, kann auch als SlideOver bleiben)
3. Detail-Ansicht zeigt: Volle Zeitreihe (24h Default), TimeRangeSelector, Schwellwert-Linien, CSV-Export
4. Breadcrumb muss funktionieren: Monitor → [Zone] → [Sensor]

### Block 5: Auto-Generierungs-Logik fuer Zone-Dashboards

**Kernaufgabe:** Automatisch sinnvolle Standard-Dashboards pro Zone generieren.

**Schritte:**
1. Beim Oeffnen einer Zone: Pruefen ob Auto-Layout existiert (`autoGenerated: true && zoneId === current`)
2. Wenn nicht → generieren basierend auf vorhandenen Sensoren:
   - Pro Sensor-Typ ein passendes Widget (Temperatur → Line-Chart, pH → Gauge)
   - Nutze `SENSOR_TYPE_CONFIG` Smart Defaults fuer Y-Achsen, Farben, Schwellwerte
3. Auto-Layouts bekommen `autoGenerated: true` Flag
4. User kann Auto-Layout "uebernehmen" (wird `autoGenerated: false`) → danach keine Auto-Updates mehr
5. Auto-Layouts aktualisieren sich NUR wenn User sie nicht modifiziert hat

**Vorhandene Bausteine:**
- `ZoneKPI` computed in MonitorView — aggregiert Temp/Humidity/Alarm pro Zone
- `sensorsByZone` computed in SensorsView — gruppiert Sensoren nach Zone/Subzone
- `groupDevicesByZone()` aus useZoneDragDrop — universelle Zone-Gruppierung
- `widgetTypes` Array in CustomDashboardView — alle 9 Widget-Definitionen mit Default-Groessen
- `SENSOR_TYPE_CONFIG` in `utils/sensorDefaults.ts` (importiert von LiveLineChart.vue) — Smart Defaults fuer ~20 Sensor-Typ-Eintraege (inkl. Aliase wie DS18B20/ds18b20, SHT31/sht31/sht31_temp/sht31_humidity) [Korrektur: Quelle ist sensorDefaults.ts, nicht LiveLineChart.vue; 20 Eintraege statt 10]
