## V-SS-04: ESP_00000001 Anomalie

**Ergebnis:** PARTIAL

**Details:**

Die Anomalie ist real aber erklaerbar. Die Datenlage:

| Zeitstempel (UTC)            | sensor_type | raw_value | processed_value | data_source | gpio |
|------------------------------|-------------|-----------|-----------------|-------------|------|
| 2026-03-09 14:30:14          | ds18b20     | 360       | 22.5            | production  | 4    |
| 2026-03-09 14:30:45          | ds18b20     | 360       | 22.5            | production  | 4    |
| 2026-03-09 15:33:43          | ds18b20     | 360       | 22.5            | production  | 4    |
| ...                          | ...         | ...       | ...             | ...         | ...  |
| 2026-03-10 07:18:10          | ds18b20     | 360       | 22.5            | production  | 4    |
| 2026-03-10 09:06:59          | ds18b20     | 360       | 22.5            | production  | 4    |

Der ESP_00000001-Status:
- `status = offline`
- `last_seen = 2026-03-09 14:32:20 UTC`
- `hardware_type = ESP32_WROOM` (echter Hardware-ESP, kein Mock)
- `mock_status` in `device_metadata` ist leer

Letzter Heartbeat-Log: `2026-03-09 14:30:04 UTC` mit `data_source=production`, `health_status=critical`.

**Herkunft der Daten — Befund:**

Die Daten nach `last_seen` kommen NICHT von einem Mock-Generator. Der `data_source`-Wert ist `production` (nicht `mock`). Der ESP_00000001 hat offensichtlich nach dem letzten registrierten Heartbeat noch weiter Sensor-Daten gesendet, ohne dass der Server diese als Heartbeat-Update verarbeitet hat. Die Erklaerung:

1. Der ESP sendet Sensor-Daten (`sensor_data`-Topic) und Heartbeats (`heartbeat`-Topic) auf unterschiedlichen Topics.
2. Der `last_seen`-Timestamp in `esp_devices` wird nur beim Heartbeat-Handler aktualisiert, nicht beim Sensor-Handler.
3. Nach 14:32:20 hat der ESP Sensor-Daten weitergesendet (15:33, 17:01, 18:28, 19:18, 07:18, 09:06), aber keine Heartbeats mehr — oder die Heartbeats wurden nicht vom Server empfangen.

Der konstante Wert `raw_value=360 / processed_value=22.5 / gpio=4` ueber alle Eintraege inklusive heute (09:06 UTC) deutet auf einen steckgebliebenen Wert hin — moeglicherweise ein ESP der physisch noch sendet oder ein retained MQTT-Message-Problem.

**Kein Mock-Generator gefunden:** Der Code-Scan ergibt keinen Background-Task der periodisch Sensor-Daten fuer echte ESP-Geraete (hardware_type=ESP32_WROOM) generiert. Der einzige Simulations-Code (`SimulationScheduler`) arbeitet ausschliesslich mit `MOCK_ESP32`-Geraeten. Der `MOCK_24557EC6` ist das einzige laufende Mock-Geraet.

**Wahrscheinlichste Erklaerung:** Der ESP_00000001 sendet Sensor-Daten via MQTT mit immer demselben Wert (retained message oder persistente Sendeschleife), aber der Heartbeat-Mechanismus ist ausgefallen (schwacher RSSI von -95 dBm in den letzten Heartbeats). Der Server-Offline-Timeout setzt `status=offline` nach ausbleibenden Heartbeats, aber eingehende Sensor-Daten werden weiter akzeptiert.

## V-AL-03: MockActuator Emergency-Stop

**Ergebnis:** PASS (Problem identifiziert, Fix-Empfehlung formuliert)

**Details:**

**CRITICAL-Log-Stelle:** `El Servador/god_kaiser_server/src/services/simulation/actuator_handler.py`, Zeile 257:

```python
logger.critical("[MockActuator] Broadcast emergency stop received!")
```

Diese Zeile steht in `handle_broadcast_emergency()` und wird aufgerufen wenn auf dem Topic `kaiser/broadcast/emergency` eine Nachricht eingeht — oder programmatisch von der Emergency-Stop-API.

**Weitere CRITICAL-Stellen im Zusammenhang:**
- `src/api/v1/actuators.py:998` — `logger.critical("EMERGENCY STOP executed by {user}...")` — wird beim API-Call `POST /api/v1/actuators/emergency-stop` ausgefuehrt
- `src/services/safety_service.py:222` — `logger.critical("EMERGENCY STOP ACTIVATED: All ESPs stopped")` — beim globalen E-Stop
- `src/services/safety_service.py:234` — `logger.critical("EMERGENCY STOP ACTIVATED: ESP {esp_id} stopped")` — beim ESP-spezifischen E-Stop

**Ausloeser:** Das CRITICAL-Log in Zeile 257 tritt auf wenn:
1. Jemand ueber die API einen Emergency-Stop ausloest (dann triggert `actuators.py:966` ein MQTT-Publish auf `kaiser/broadcast/emergency`)
2. Der MQTT-Broker ein retained Emergency-Message liefert beim Neuverbinden
3. Der Server-Container neu startet und `recover_mocks()` Mocks wiederherstellt — wenn dabei eine retained MQTT-Emergency-Message auf dem Broker vorliegt

**Startup-Kontext:** Bei Server-Start laeuft `recover_mocks()` (main.py Step 3.5). Falls ein retained `kaiser/broadcast/emergency`-Payload auf dem Broker liegt (weil ein vorheriger Emergency-Stop mit `retain=True` veroeffentlicht wurde — siehe `actuators.py:970`), wird dieser beim MQTT-Subscribe sofort zugestellt. Das triggert `handle_broadcast_emergency()` → `logger.critical(...)`.

```python
# actuators.py:966-970 — Emergency-Stop publiziert mit retain=True!
publisher.client.publish(
    topic="kaiser/broadcast/emergency",
    payload=broadcast_payload,
    qos=1,
    retain=True,  # <--- PROBLEM: retained message bleibt auf Broker
)
```

**Docker-Container-Start:** `docker ps` zeigt keinen laufenden `el-servador`-Container — Server ist derzeit nicht aktiv. Container-Startzeit konnte nicht ermittelt werden.

**Fix-Empfehlung:** Zwei moegliche Loesungen:

Option A (Log-Level): In `actuator_handler.py:257` CRITICAL → WARNING aendern:
```python
# Vorher:
logger.critical("[MockActuator] Broadcast emergency stop received!")
# Nachher:
logger.warning("[MockActuator] Broadcast emergency stop received!")
```

Option B (Root Cause): Das retain=True beim Emergency-MQTT-Publish entfernen oder ein Cleanup beim Server-Start einfuegen. Beim ersten Start nach einem Emergency-Stop sollte der retained Payload mit einem leeren Payload (clear_emergency) ueberschrieben werden. Das ist die korrektere Loesung, weil sie das tatsaechliche Problem adressiert.

Empfehlung: Option B ist besser, Option A als schnellen Workaround akzeptabel wenn das CRITICAL-Log im Alert-System zu False-Positives fuehrt.

## V-AL-04: Grafana Webhook → Notification Pipeline

**Ergebnis:** PARTIAL (V-AL-02-Befund teilweise widerlegt — Fingerprint wird in Webhook-Handler korrekt gesetzt, aber `_broadcast_to_all()` propagiert ihn NICHT)

**Details: Webhook-Handler-Trace**

**GrafanaWebhookPayload-Struktur** (`webhooks.py:55-71`):
```python
class GrafanaWebhookPayload:
    version, groupKey, truncatedAlerts, status, receiver
    groupLabels, commonLabels, commonAnnotations, externalURL
    alerts: List[GrafanaAlert]
    orgId, title, state, message
```

**GrafanaAlert-Struktur** (`webhooks.py:39-52`):
```python
class GrafanaAlert:
    status: str          # "firing" | "resolved"
    labels: Dict[str, str]
    annotations: Dict[str, str]
    startsAt, endsAt, generatorURL
    fingerprint: Optional[str]  # Grafana-seitig optional!
    silenceURL, dashboardURL, panelURL, values
```

**map_grafana_severity()** (`webhooks.py:136-161`): Prueft `alert.status == "resolved"` → `"info"`. Sonst Severity-Label → critical/warning. Folder-Fallback. Default: `"warning"`.

**correlation_id-Erzeugung** (`webhooks.py:203`):
```python
correlation_id = f"grafana_{alert.fingerprint}" if alert.fingerprint else None
```

**Notification-Pipeline-Trace:**

```
POST /v1/webhooks/grafana-alerts
    └── for alert in payload.alerts:
        └── NotificationCreate(
                fingerprint=alert.fingerprint,      # Zeile 267: KORREKT gesetzt
                correlation_id=correlation_id,      # Zeile 269: gesetzt wenn fingerprint vorhanden
                user_id=None,                       # Zeile 259: Broadcast!
            )
        └── router_service.route(notification)
            └── user_id is None → _broadcast_to_all(notification)
                └── for user in users:
                    └── NotificationCreate(           # Zeile 191-202
                            user_id=user.id,
                            channel=...,
                            severity=...,
                            category=...,
                            title=...,
                            body=...,
                            metadata=...,
                            source=...,
                            parent_notification_id=...,
                            correlation_id=...,       # WIRD propagiert
                            # fingerprint=???         # FEHLT! Nicht propagiert!
                        )
                    └── self.route(user_notification)
                        └── user_id is not None → fingerprint check
                            └── check_fingerprint_duplicate(fingerprint)
                                # fingerprint ist None → check wird uebersprungen!
                                # Fallback: title-based dedup mit 60s Fenster
```

**Root Cause (exakt):**

`El Servador/god_kaiser_server/src/services/notification_router.py`, Zeile 191-202, Methode `_broadcast_to_all()`:

```python
user_notification = NotificationCreate(
    user_id=user.id,
    channel=notification.channel,
    severity=notification.severity,
    category=notification.category,
    title=notification.title,
    body=notification.body,
    metadata=notification.metadata,
    source=notification.source,
    parent_notification_id=notification.parent_notification_id,
    correlation_id=notification.correlation_id,
    # fingerprint fehlt hier komplett!
)
```

Das `fingerprint`-Feld aus dem originalen `NotificationCreate` wird nicht an die per-User-Kopien weitergegeben. Wenn `route()` dann fuer jeden User mit `user_id != None` aufgerufen wird, ist `notification.fingerprint` None → `check_fingerprint_duplicate()` wird uebersprungen → title-based Dedup greift mit 60s Fenster → bei 90 Grafana-Benachrichtigungen in kurzer Zeit schlaegt das Dedup fehl.

**Auto-Resolve-Trace** (`webhooks.py:206-219`):

Wenn Grafana `status: "resolved"` sendet und `fingerprint` vorhanden:
1. `correlation_id = f"grafana_{fingerprint}"` → nicht None
2. `auto_resolve_by_correlation(correlation_id)` → UPDATE notifications SET status='resolved' WHERE correlation_id = 'grafana_{fp}' AND status IN ('active','acknowledged')
3. Resolved alerts werden `continue`d — kein neues Info-Notification wird erstellt

**Sekundaer-Problem: Resolved → Refire-Cycle:**
Wenn Grafana `resolved` sendet aber `alert.fingerprint` ist None (weil Grafana kein fingerprint liefert):
- `correlation_id` ist None → `auto_resolve_by_correlation()` wird nicht aufgerufen
- Stattdessen wird eine neue `status=info` Notification erstellt
- `check_fingerprint_duplicate()` wird nicht aufgerufen (kein fingerprint)
- `check_correlation_duplicate()` greift nur bei broadcast-Dedup (wenn correlation_id vorhanden)
- Result: Info-Notification landet in DB ohne Dedup

**`check_fingerprint_duplicate()` Analyse** (notification_repo.py:228-241):
```python
# Prueft: EXISTS notification WHERE fingerprint = ? AND status IN ('active','acknowledged')
# Exakter Match. Kein Zeitfenster. Status-Filter.
# Greift NICHT bei NULL-Fingerprints — SQL: NULL != NULL in Datenbanken!
# SELECT COUNT(*) WHERE fingerprint = NULL → immer 0 Treffer → is_duplicate = False
```

**Fix-Empfehlung (1-Zeilen-Fix):**

In `notification_router.py`, `_broadcast_to_all()`, Zeile 191-202, `fingerprint` ergaenzen:

```python
user_notification = NotificationCreate(
    user_id=user.id,
    channel=notification.channel,
    severity=notification.severity,
    category=notification.category,
    title=notification.title,
    body=notification.body,
    metadata=notification.metadata,
    source=notification.source,
    parent_notification_id=notification.parent_notification_id,
    correlation_id=notification.correlation_id,
    fingerprint=notification.fingerprint,   # <- 1-Zeilen-Fix
)
```
