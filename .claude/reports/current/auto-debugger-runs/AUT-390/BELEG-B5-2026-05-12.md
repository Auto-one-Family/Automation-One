# BELEG — B5: zone_id / subzone_id Spalten leer bei aggregierten Export-Daten

**Finding-ID:** B5
**Datum:** 2026-05-12
**Linear-Issue:** wird angelegt als Child von AUT-384 (relatedTo AUT-390)
**Kategorie:** error
**Schicht:** El Servador

## Symptom-Zusammenfassung

Beim Export mit `resolution=1m` (oder 5m/1h/1d) und `columns=zone_id` sind alle Werte in der `zone_id`-Spalte leer — obwohl die zugrundeliegenden `sensor_data`-Zeilen zone_id-Werte enthalten. Dasselbe gilt für `subzone_id`. Das Problem tritt auf, weil die Aggregations-Query-Results keine `zone_id`/`subzone_id`-Attribute haben (sie werden nicht in die GROUP-BY-Abfrage eingeschlossen) und `_export_column_value` für aggregierte Rows kein entsprechendes Mapping definiert.

## Code-Beleg

`El Servador/god_kaiser_server/src/api/v1/sensors.py:1557–1576`

```python
def _export_column_value(row: object, col: str, is_aggregated: bool) -> str:
    """Extract a column value from a SensorData row (raw or aggregated) as a string."""
    if is_aggregated:
        mapping = {
            "timestamp": "bucket",
            "raw_value": "avg_raw",
            "processed_value": "avg_processed",
        }
        attr = mapping.get(col, col)   # zone_id → attr = "zone_id"
        if col == "quality":
            return "aggregated"
        val = getattr(row, attr, None) # row hat kein "zone_id" Attribut → None → ""
    else:
        val = getattr(row, col, None)
    ...
```

Das aggregierte Row-Objekt aus `sensor_repo.query_data(resolution="1m")` enthält nur die Aggregate-Felder (`bucket`, `avg_raw`, `avg_processed`, ggf. `sensor_type`, `esp_id`, `gpio`, `unit`). `zone_id` und `subzone_id` werden in der GROUP-BY-Aggregation nicht mitgeführt.

## Kausalerklärung

Zwei mögliche Ursachen (muss per DB-Query verifiziert werden, welche zutrifft):

1. **Aggregations-Query** gruppiert nicht nach `zone_id`/`subzone_id`, daher fehlen diese Felder im Result-Objekt.
2. **`_export_column_value`** hat kein Fallback für `zone_id`/`subzone_id` bei aggregierten Rows — `getattr(row, "zone_id", None)` gibt `None` zurück weil das Attribut nicht existiert.

## Fix-Richtung

Entscheidung erforderlich: Soll `zone_id` bei aggregierten Daten:
- **A) Leer bleiben** (dokumentiert, WARN wird zu INFO) — dann QA-Test anpassen
- **B) Den ersten/häufigsten zone_id-Wert** aus der Gruppe zeigen — dann Aggregations-Query erweitern
- **C) N/A** anzeigen — dann `_export_column_value` Mapping erweitern mit `"zone_id": None` → `"aggregated"`

Einfachste saubere Lösung: Option A (Verhalten dokumentieren), da `zone_id` bei aggregierten Daten semantisch unklar ist (mehrere Zonen könnten aggregiert sein).
