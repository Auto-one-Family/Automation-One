# BELEG — B3: Dateiname enthält 'all' bei subzone_id-only Export

**Finding-ID:** B3
**Datum:** 2026-05-12
**Linear-Issue:** wird angelegt als Child von AUT-384 (relatedTo AUT-390)
**Kategorie:** error
**Schicht:** El Servador

## Symptom-Zusammenfassung

Wenn nur `subzone_id` als Filter übergeben wird (ohne `esp_id` und ohne `zone_id`), enthält der `Content-Disposition`-Header einen Dateinamen mit `all` als Bezeichner: `sensor-export_all_20260511T220606_20260512T220606.csv`. Der Name suggeriert einen vollständigen Export, obwohl tatsächlich nur Daten einer bestimmten Subzone enthalten sind.

**Ergebnis:** WARN

```
Detail: B3: Content-Disposition='attachment; filename="sensor-export_all_20260511T220606_20260512T220606.csv"' — contains 'all': True
```

## Code-Beleg

`El Servador/god_kaiser_server/src/api/v1/sensors.py:1731`

```python
# Build filename: sensor-export_{esp_id}_{start}_{end}.csv
start_str = start_time.strftime("%Y%m%dT%H%M%S")
end_str = end_time.strftime("%Y%m%dT%H%M%S")
filename = f"sensor-export_{esp_id or zone_id or 'all'}_{start_str}_{end_str}.csv"
#                                                ↑ subzone_id fehlt in der Kette
```

## Kausalerklärung

Die Fallback-Kette `esp_id or zone_id or 'all'` berücksichtigt `subzone_id` nicht. Bei einem Request mit `subzone_id=main` (ohne `esp_id` und `zone_id`) sind `esp_id` und `zone_id` `None`, also wird `'all'` als Dateiname verwendet, was irreführend ist.

## Fix-Vorschlag (Minimal)

```python
# sensors.py:1731 — ersetze
filename = f"sensor-export_{esp_id or zone_id or 'all'}_{start_str}_{end_str}.csv"
# durch
filename = f"sensor-export_{esp_id or zone_id or subzone_id or 'all'}_{start_str}_{end_str}.csv"
```
