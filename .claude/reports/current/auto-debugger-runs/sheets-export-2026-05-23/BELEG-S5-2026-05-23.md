## D1 — Quelle

`actuator_history` ist Event-Log. Kein Zugriff auf `actuator_states` fuer Laufzeit.

## D2 — ON/OFF-Pairing-Algorithmus (Variante A, verbindlich)

Algorithmus (deterministisch, retry-safe):

```
1. Lade alle history-Rows seit Cursor (timestamp, id) sortiert.
2. Lade aus DB alle bereits offen-stehenden ONs (vorgaenger-Recovery):
   SELECT MAX(id) per (esp_id, gpio) WHERE _is_on_event(row) AND NOT geschlossen
   -> spaeter durch Restart-Recovery via system_config-Hilfsschluessel
      `sheets_export_history_open_runs` (Map (esp_id,gpio) -> {start_id, start_ts})
3. Pro Row in Chronologie:
   - ON-Event (value > 0 OR command_type in {"set","activate"} mit value>0)
       -> wenn open_runs[(esp,gpio)] existiert: emit alten Run als open_run-close
          (notes=overlapping_on), neuer Run startet
       -> sonst: open_runs[(esp,gpio)] = {start}
   - OFF-Event (value == 0 OR command_type in {"stop","off","emergency_stop"})
       -> wenn open: emit Run-Zeile, loesche open_runs Eintrag
       -> sonst: emit "off_without_on"-Zeile
   - Spezialfall command_type == "emergency_stop":
       -> falls open, schliesst Run mit ausloeser=emergency_stop
       -> falls nicht open, emit isolated row mit ausloeser=emergency_stop
4. Am Batch-Ende: open_runs persistieren in system_config
5. Cursor vorruecken auf hoechste verarbeitete (timestamp, id)
```

## D3 — Ausloeser-Vokabular (freeze)

Mapping aus `issued_by` + Zeitfenster-Korrelation:

| Pattern in `issued_by` | -> `ausloeser` | Quelle |
|---|---|---|
| `logic:<rule_id>` | `rule:<rule_id>:<rule_name>` | Lookup `cross_esp_logic.rule_name` |
| `user:<username>` | `manual:user:<username>` | direkt |
| `api:<key>` | `manual:api:<key>` | direkt |
| `emergency:<x>` | `emergency_stop` | direkt (Ueberschreibt command_type) |
| `system:lwt_disconnect` | `lwt_offline` | direkt |
| `system:actuator_delete` | `manual:system:actuator_delete` | direkt |
| `system:offline_*` / `offline_rule_*` | `offline:<gpio>` | direkt |
| (sonst) | `manual:<issued_by>` (Fallback) | nie leer |

`ausloeser` ist NIE leer (fallback-safe).

## D8 — Korrelation Option A (verbindlich)

- Primaer: `issued_by` (Klassifikation siehe oben).
- Sekundaer (optional): wenn `command_metadata.correlation_id` vorhanden -> Lookup in `logic_execution_history.execution_metadata.correlation_id` im Zeitfenster `[run_start - window, run_end + window]`.
- Tertiaer (Rule-Name-Anreicherung fuer `logic:*`): Zeitfenster-Match `logic_execution_history` WHERE `logic_rule_id == rule_id AND timestamp BETWEEN [run_start - window, run_start + window]`. Anchor: erstes Match nach Zeit. `correlation_window_seconds` default 120 (anpassbar via Methoden-Param).
- Kein harter FK. Bei fehlendem Match -> `ausloeser` bleibt `rule:<id>:<rule_id_short>` (ohne Name).

## Edge Cases (Pflicht)

| Fall | Verhalten | `notes` |
|---|---|---|
| ON ohne OFF | offene Zeile, `run_end_utc=null`, `duration_seconds=null` | `open_run` |
| OFF ohne ON | isolierte Zeile, `run_start_utc=null` | `off_without_on` |
| Emergency-Stop | Run wird geschlossen, `ausloeser=emergency_stop` | `emergency_stop` |
| LWT-Auto-OFF | Run wird geschlossen, `ausloeser=lwt_offline` | `lwt_offline` |
| Server-Restart | open_runs aus `system_config.sheets_export_history_open_runs` rehydriert | `recovered_open` (nach erstem OFF) |
| Doppel-ON (ON ohne OFF dazwischen) | alten Run als `overlapping_on` close, neuer Run startet | `overlapping_on` |
