## IST-Stand vor Änderung

### Frontend (SensorCard.vue, SensorValueCard.vue)
Kein Operator-Hinweis dass die Sonde Zeit zum Stabilisieren braucht. Kein Toast, kein statischer Hinweis vor dem Messen-Klick.

### Firmware (sensor_manager.cpp:triggerManualMeasurement, Zeile 1646)
Single-sample path für alle Sensoren. Kein Multi-Sample für analoge Proben:
```cpp
SensorReading reading;
if (performMeasurement(gpio, reading)) {
    // direkt publishen
}
```
DFRobot DFR0300 V2: min. 800ms Settling-Zeit nach Eintauchen. Erste Messung nach neuem Eintauchen liefert oft unrealistisch hohen Wert.

### (b) Firmware Multi-Sample — sensor_manager.cpp

**Datei:** `El Trabajante/src/services/sensor/sensor_manager.cpp`  
**Funktion:** `triggerManualMeasurement()`, Zeile 1646 (single-value path)

Neuer Analog-Sensor-Check + 3-Sample-Median-Loop:
```cpp
// AUT-314: 3-sample median for analog probes (pH, EC, moisture)
String lower_type_aut314 = String(config->sensor_type);
lower_type_aut314.toLowerCase();
bool is_analog_probe = (lower_type_aut314.indexOf("ph") >= 0 || ...);

SensorReading reading;
if (is_analog_probe) {
    SensorReading samples[3];
    uint8_t valid_count = 0;
    for (uint8_t i = 0; i < 3; i++) {
        if (i > 0) delay(200);   // 200ms zwischen Samples
        if (performMeasurement(gpio, samples[valid_count])) valid_count++;
    }
    // Bubble-sort + pick median
    reading = samples[valid_count / 2];
} else {
    performMeasurement(gpio, reading);  // Bisheriges Verhalten
}
// Gemeinsamer Publish-Pfad (E-P3 Timeout-Guard bleibt)
```

Zeitaufwand: 2 × 200ms = +400ms im Mutex — innerhalb des 5s-Timeouts.  
Scope: Nur `triggerManualMeasurement()` — periodische Messungen unverändert.
