## IST-Stand vor Änderung

### (a) SensorCard.vue:330 — Timeout zu kurz

```
}, 10_000)   // 10 Sekunden
```

ESP-Mutex-Wait (`kManualSensorMutexWaitMs`, `sensor_manager.cpp:1597`) = bis 10s.
Race: Mutex-Wait ≈ Timeout → Wert kommt nach dem Frontend-Timeout an.

### (b) SensorValueCard.vue:73-112 — kein Finality-Watch

```typescript
const isMeasuring = ref(false)
// ...
async function handleTriggerMeasurement() {
  isMeasuring.value = true
  try {
    await sensorsApi.triggerMeasurement(...)
    toast.success('Messung gestartet für GPIO ...')
  } catch (...) {
    toast.error(...)
  } finally {
    setTimeout(() => { isMeasuring.value = false }, 2000)  // Blindes 2s-Reset
  }
}
```

Verstößt gegen C5-Hub Prinzip 2 (Finalitätsmodell Pflicht). Kein `pending → success|failed|timeout` Zustandsmodell. Kein Watch auf `sensor.last_read` vs. `triggerTime`.

### (c) EditSensorModal.vue:212 — falscher API-Endpunkt

```typescript
await espStore.setSensorValue(props.espId, editingSensor.value.gpio, 0, 'good', true)
```

`setSensorValue` ist für Aktor-Steuerung, nicht für Sensor-Trigger. Falscher Endpunkt/Store-Action.

Watch (identisch mit SensorCard.vue:285-303):
```typescript
watch(
  () => [props.sensor.last_read, props.sensor.raw_value] as const,
  ([newLastRead, newRawValue]) => {
    if (!measureTriggerTime) return
    if (!isMeasuring.value && measureState.value !== 'error') return
    const timestampFresh = newLastRead != null && new Date(newLastRead).getTime() > measureTriggerTime
    const lastReadChanged = newLastRead !== preMeasureLastRead
    const valueMutated = newRawValue !== preMeasureTriggerValue
    if (timestampFresh || lastReadChanged || valueMutated) {
      resolveMeasureSuccess()
    }
  }
)
```

Timeout: `20_000` (konsistent mit SensorCard.vue nach Fix (a)).

Template-Button: Error-State-Feedback ergänzt (`AlertTriangle` + Text "Fehler" bei `measureState === 'error'`).
