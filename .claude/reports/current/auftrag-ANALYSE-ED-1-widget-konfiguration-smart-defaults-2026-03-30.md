## Kontext

### Lücken

- **Keine generischen Keys** `temperature`, `humidity`, `pressure` — falls die Firmware solche generic sensor_types senden würde, wäre kein Config-Eintrag vorhanden. Fallback: `category = 'other'`, kein min/max → Gauge zeigt 0-100.
- Die Duplikate (Uppercase/Lowercase) sind **bewusst** für Backward-Compat wenn API/DB verschiedene Case-Schreibweisen sendet.
- `SHT31`/`sht31` Umbrella-Keys haben `unit: '°C'` als Fallback, obwohl der Sensor beide Werte (Temperatur + Feuchte) liefert — das ist korrekt, da der Umbrella-Key selbst nicht direkt für Display genutzt wird (die Sub-Keys `sht31_temp`/`sht31_humidity` werden verwendet).
- **VPD** hat `category: 'air'` → wird als Gauge generiert (CATEGORY_WIDGET_MAP: `air → gauge`). Korrekt für VPD.

## A2: Widget-Config-Flow bei Sensor-Wechsel

### Sequenzdiagramm

```
User wählt neuen Sensor im WidgetConfigPanel.vue
        ↓
@change → handleSensorChange(sensorId)
        ↓
findSensorType(sensorId)
  → iteriert groupedSensorOptions (aus useSensorOptions)
  → gibt sensorType zurück (z.B. "sht31_humidity")
        ↓
SENSOR_TYPE_CONFIG[sensorType] → cfg vorhanden?
        ↓ JA
if (localConfig.value.yMin == null && localConfig.value.yMax == null)  ← BUG: F1
     ↓ NEIN (yMin=0, yMax=3 von VPD-Generierung)
Y-Range NICHT aktualisiert
        ↓
Titel NICHT aktualisiert  ← BUG: F2
        ↓
emit('update:config', localConfig)  → Dashboard Store speichert alte Range + Titel
```

### Root Cause F1 — WidgetConfigPanel.vue:147

```typescript
// WAS (BUG):
if (localConfig.value.yMin == null && localConfig.value.yMax == null) {
  updates.yMin = cfg.min
  updates.yMax = cfg.max
}
```

**Problem:** Auto-generierte Dashboards setzen immer `yMin` und `yMax` aus SENSOR_TYPE_CONFIG:

```typescript
// dashboard.store.ts:767
...(config ? { yMin: config.min, yMax: config.max } : {}),
```

Das bedeutet: yMin und yMax sind NIEMALS null bei auto-generierten Widgets. Die `null`-Prüfung in `handleSensorChange` schlägt daher immer fehl.

Für VPD-Gauge: `yMin = 0`, `yMax = 3`. Sensor-Wechsel auf Luftfeuchte:
- `0 != null` → true → Bedingung schlägt fehl → Range bleibt 0-3 statt 0-100

### Root Cause F2 — WidgetConfigPanel.vue:138

```typescript
function handleSensorChange(sensorId: string) {
  const sType = findSensorType(sensorId)
  const updates: Record<string, any> = { sensorId }
  // ... Nur Y-Range wird (bedingt) aktualisiert
  // Kein Update für 'title'
}
```

Kein einziger Code-Pfad der `title` bei Sensor-Wechsel anfasst. Der Titel "VPD (berechnet)" bleibt.

### Was richtig funktioniert

1. **Unit-Anzeige** in Widgets kommt aus `currentSensor.unit` (Live-Store-Daten), NICHT aus der Widget-Config. Wechsel auf Luftfeuchte → %RH wird sofort korrekt angezeigt.
2. **findSensorType** funktioniert korrekt — 3-Part sensorId `espId:gpio:sensorType` wird über `useSensorOptions` korrekt aufgelöst.
3. **Legacy 2-Part IDs** `espId:gpio`: `sensorType = null`, Fallback auf `currentSensor.sensor_type` in `GaugeWidget.vue:62-64`. Funktioniert, aber kein SENSOR_TYPE_CONFIG-Default möglich bis Sensor aus Store gefunden.

## A3: Gauge-Widget Range-Logik

**Datei:** [GaugeWidget.vue:73-74](El%20Frontend/src/components/dashboard-widgets/GaugeWidget.vue#L73)

```typescript
const effectiveMin = computed(() => props.yMin ?? sensorTypeDefaults.value?.min ?? 0)
const effectiveMax = computed(() => props.yMax ?? sensorTypeDefaults.value?.max ?? 100)
```

### Prioritätskette

```
props.yMin (gespeicherte Config)
    ?? sensorTypeDefaults.min (SENSOR_TYPE_CONFIG[currentSensorType])
        ?? 0
```

### Das `??`-Problem

Der `??` Operator ist `null/undefined`-safe, NICHT falsy-safe. Das heißt:

```typescript
0 ?? 100   // = 0   (0 ist nicht null/undefined)
null ?? 100 // = 100
```

VPD-Gauge hat `yMin = 0` (aus Auto-Generierung). Nach Sensor-Wechsel auf Luftfeuchte:
- `props.yMin = 0` → `0 ?? ...` → **0** (SENSOR_TYPE_CONFIG Fallback wird geblockt)
- Erwartetes Ergebnis für Luftfeuchte: `min = 0` → zufällig "korrekt"!
- `props.yMax = 3` → `3 ?? ...` → **3** statt 100 → **FALSCH**

Das Gauge zeigt einen Bogen von 0 bis 3, während die Luftfeuchte typischerweise 40-80 % beträgt. Der Zeiger würde weit über den Bogen hinauszeigen oder auf Maximum geclippt werden.

### GaugeChart.vue: Hardcoded arc limits

Anders als LineChartWidget (suggestedMin/Max = weich) verwendet GaugeChart **harte Limits** für den Arc. Ein Wert außerhalb [min, max] würde visuell brechen. Das macht den Range-Bug bei Gauge schwerer als bei Charts.

## A4: Line-Chart Y-Achse Einheit-Zuordnung

### LiveLineChart.vue (LineChartWidget)

**Datei:** [LiveLineChart.vue:227-236](El%20Frontend/src/components/charts/LiveLineChart.vue#L227)

```typescript
// Y-axis range: explicit props > SENSOR_TYPE_CONFIG lookup > auto
...(props.yMin != null ? { suggestedMin: props.yMin }
  : props.sensorType && SENSOR_TYPE_CONFIG[props.sensorType]
    ? { suggestedMin: SENSOR_TYPE_CONFIG[props.sensorType].min }
    : {}),
```

`suggestedMin`/`suggestedMax` sind **Chart.js "Empfehlungen"**, keine Limits. Wenn Daten außerhalb des Bereichs liegen, expandiert Chart.js die Y-Achse automatisch. Das bedeutet:

- VPD-Config `yMin=0, yMax=3`, Luftfeuchte-Daten bei 65 %RH → Chart.js expandiert auf 0-70 → Daten korrekt sichtbar
- Aber: Y-Achsen-Beschriftung zeigt `%RH` (korrekt aus `currentSensor.unit`), Range ist suboptimal

**Root Cause für das beobachtete Problem** (Titel "Temperatur" zeigt %RH):
1. Auto-generiertes Widget: Temperatursensor, `sensorType=sht31_temp`, `title="Temperatur"`, `yMin=-40, yMax=125`
2. User wechselt auf `sht31_humidity` → unit ändert sich auf %RH
3. Y-Range bleibt -40/125 (Bug F1) → aber durch `suggestedMin/Max` wird die Achse korrekt expandiert auf Humidity-Bereich
4. Titel bleibt "Temperatur" (Bug F2) → KONFUSION: "Temperatur"-Widget zeigt %RH

### HistoricalChart.vue (HistoricalChartWidget)

**Datei:** [HistoricalChart.vue:571-580](El%20Frontend/src/components/charts/HistoricalChart.vue#L571)

```typescript
y: {
  display: true,
  // KEIN suggestedMin/suggestedMax
  ticks: {
    callback: (val: any) => `${val}${props.unit ? ' ' + props.unit : ''}`,
```

**HistoricalChart hat KEINE yMin/yMax Props** — vollständig auto-scaled aus den API-Daten. Kein Y-Range-Problem. Die Unit kommt aus `props.unit` (= `parsedSensor.sensor.unit` aus HistoricalChartWidget:104).

### Unit-Herkunft — Zusammenfassung

| Widget | Unit-Quelle | Korrekt nach Sensor-Wechsel? |
|--------|------------|------------------------------|
| LineChartWidget → LiveLineChart | `currentSensor.unit` (Live-Store) | ✅ JA |
| GaugeWidget → GaugeChart | `currentSensor.unit` (Live-Store) | ✅ JA |
| HistoricalChartWidget → HistoricalChart | `parsedSensor.sensor.unit` (Store) | ✅ JA |

### sensorId-Format

```typescript

```
3-Part Format korrekt. ✅

### Titel-Generierung

```typescript
title: s.name || config?.label || s.sensorType,
```
- `sensor.name` (Benutzer-gesetzter Name, z.B. "VPD Gewächshaus Süd")
- `SENSOR_TYPE_CONFIG[type].label` (z.B. "VPD")
- raw `sensor_type` als letzter Fallback

### Update-Verhalten bei neuen Sensoren

- `generateZoneDashboard` prüft: `existing = layouts.find(l => l.autoGenerated && l.scope === 'zone' && l.zoneId === zoneId)`
- Falls existing: **KOMPLETTES Layout wird ersetzt** — alle Widgets neu generiert ✅
- Problem: Manuelle Config-Anpassungen auf auto-generierten Widgets gehen dabei verloren

### claimAutoLayout — Duplikate-Risiko

```typescript
function claimAutoLayout(layoutId: string) {
  layouts.value[idx] = { ...layouts.value[idx], autoGenerated: false, ... }
}
```

Nach `claimAutoLayout`: `autoGenerated = false`. Beim nächsten `generateZoneDashboard`-Aufruf wird das geclaimte Layout NICHT gefunden (da `l.autoGenerated` nun false). Es wird ein **neues** Layout mit `autoGenerated: true` erstellt. → Beide Layouts existieren parallel, das neue überschreibt nicht das geclaimte.

Dies ist wahrscheinlich beabsichtigt (User hat Layout "übernommen" und will es nicht mehr auto-überschreiben). Aber es kann zu UI-Verwirrung führen wenn Zone-Dashboards doppelt erscheinen.

### Neue Widgets aus Katalog

```typescript
function addWidget(type: string) {
```

### Verbesserungspotenzial

## A7: Widget-Typ-Duplizierung: Historical vs. Monitor-L2

### Monitor-L2 Chart (MonitorView.vue)

**Datei:** [MonitorView.vue:2193-2195](El%20Frontend/src/views/MonitorView.vue#L2193)

### HistoricalChart.vue (Dashboard-Widget)

Standalone-Komponente mit:
- Time Range Selector (1h/6h/24h/7d/30d)
- Zoom/Pan (chartjs-plugin-zoom)

- `sensorsApi.queryData({esp_id, gpio, sensor_type, start_time, end_time, ...})`

### Vergleich

| Feature | Monitor-L2 | HistoricalChartWidget |
|---------|-----------|----------------------|
| Time Range wählbar | Eigener Selector in SlideOver | Eigener Selector im Widget |
| Multi-Sensor Overlay | ✅ (bis 3 Sensoren) | ❌ (nicht vorhanden) |

### Urteil

**BEABSICHTIGTE Duplizierung**. Monitor-L2 ist kontextuell (immer der aktuell angezeigte Sensor, Multi-Overlay für Vergleich), während HistoricalChartWidget frei konfigurierbar ist. Die Code-Duplizierung ist real aber durch unterschiedliche UX-Anforderungen gerechtfertigt.

Monitor-L2 fehlt: Gap Detection, VPD Zones — diese könnten nachgerüstet werden wenn HistoricalChart.vue extrahierbar ist.
