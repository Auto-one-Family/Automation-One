```python
@staticmethod
def build_actuator_emergency_topic(esp_id: str, kaiser_id: str = "god") -> str:
    """
    Build actuator emergency stop topic.

    Used by ESPs to publish emergency stop events.

    Args:
        esp_id: ESP device ID
        kaiser_id: Kaiser ID (default: "god")

    Returns:
        kaiser/{kaiser_id}/esp/{esp_id}/actuator/emergency
    """
    return f"kaiser/{kaiser_id}/esp/{esp_id}/actuator/emergency"
```

**Der GHOST-Kommentar:** Zeile 994 — `"Used by ESPs to publish emergency stop events."`

### Gegenprüfe `main.cpp:629`

**Datei:** `El Trabajante/src/main.cpp`
**Zeile:** 629
**Commit:** `705a060acb24c8d97eac19c4404d2835c3ea2094` (Tue Apr 21 10:28:56 2026 +0200)

```cpp
mqttClient.queueSubscribe(TopicBuilder::buildActuatorEmergencyTopic(), 1, true);
```

### Erklärung

Der Docstring behauptet, ESPs *publizieren* auf `actuator/emergency`. Tatsächlich **subscribed** der ESP auf dieses Topic (`queueSubscribe`, QoS 1). Das Topic ist ein Server→ESP-Kanal: Der Server sendet den Emergency-Stop-Befehl, der ESP empfängt und verarbeitet ihn. Die Docstring-Richtungsangabe ist um 180° falsch.

Korrekte Aussage: *"Server publishes on this topic to trigger an emergency stop. ESP subscribes to receive the command."*

**Fix (kein Code-Eingriff dieses Auftrags):** Docstring-Zeile 994 ersetzen durch:
```
Server publishes on this topic to send an emergency stop command to a specific ESP.
ESP subscribes and executes the stop sequence on receipt.
```

### Status

### Code-Evidenz

**Zeile:** 335 (nicht 172 wie im Auftrag angegeben — Zeile 172 enthält `std::atomic<bool> g_mqtt_connected{false}`)
**Commit:** `5c34c3f60e3882c39a3da4b1a5fbc3a1b2431f38` (Fri Apr 24 02:05:50 2026 +0200)

```cpp
esp_mqtt_client_config_t mqtt_cfg = {};
mqtt_cfg.uri = broker_uri;
mqtt_cfg.keepalive = config.keepalive;
mqtt_cfg.disable_clean_session = 0;   // ← Zeile 335: clean_session=true (aktiv)

mqtt_cfg.lwt_topic = lw_topic_str.c_str();

```

`disable_clean_session = 0` bedeutet: Deaktivierung des Clean-Session-Flags ist aus → **`clean_session = true`** ist aktiv. Bei jedem Disconnect löscht der Broker die ESP-Session samt ausstehender QoS-2-Nachrichten.

### Dokumentation vorhanden?

Grep auf `docs/`, `.claude/`: **kein ADR gefunden.** Die Einstellung ist ein undokumentierter Legacy-Default aus der ESP-IDF-Konfiguration.

### ADR-Datei

Angelegt unter: `docs/analysen/adr-clean-session-strategie-2026-04-26.md`

### TM-Empfehlung

`clean_session=true` belassen. Begründung im ADR — die 5 Kompensationsmechanismen decken das QoS-2-Verlustrisiko hinreichend ab, und `clean_session=false` bringt Session-Bloat-Risiken bei vielen ESPs mit sich.
